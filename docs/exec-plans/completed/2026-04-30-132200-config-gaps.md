# Plan: Address config gaps vs Hermes

## Context
Research doc `docs/reference/RESEARCH-config-co-vs-hermes-gaps.md` identified three gaps in `co-cli` configuration compared to Hermes:
1. No separate secrets file (Hermes uses `.env`).
2. No `${VAR}` template expansion in file-backed config values.
3. No commentable config format (Hermes uses `config.yaml`).
We need to address these gaps to improve configuration ergonomics, security, and parity.

## Problem & Outcome
**Problem:** The current `settings.json` cannot have comments, forcing users to store secrets in the main config file or export them manually, and preventing variable reuse across settings.
**Failure cost:** Secrets are more likely to be committed or shared inadvertently, configurations are harder to read and document, and users experience friction porting workflows from standard `.env`-based setups.
**Outcome:** Users can manage secrets securely via `~/.co-cli/.env` (not committed, `0o600`-enforced); env vars from `.env` are available to config resolution without overwriting shell-set vars.

## Scope
- Add `python-dotenv` to dependencies. (**PyYAML and YAML config format: out of scope** — YAML ergonomics are marginal for a personal CLI with a small config surface; dual-format fallback adds permanent complexity.)
- Update `co_cli/config/core.py` to support loading `~/.co-cli/.env` into the environment before config resolution.
- Update `co_cli/bootstrap/security.py` to check permissions on `.env` in addition to `settings.json`.
- **`${VAR}` template expansion: out of scope** — if `.env` is loaded into `os.environ`, config code can read vars directly; the expansion layer adds regex-traversal complexity for a rarely-used pattern.

## Behavioral Constraints
- **File Precedence Warning:** If both `config.yaml` and `settings.json` exist, the system must warn the user (to avoid confusion) and prefer `config.yaml`.
- **Backward Compatibility:** If `config.yaml` does not exist, the system must continue to load `settings.json` seamlessly without throwing an error.
- **Env Precedence:** Environment variables loaded from `.env` must not overwrite explicit system environment variables already present in the shell.
- **Expansion Safety:** Expansion must not crash if a `${VAR}` is missing; it should either leave it as is or expand to empty string. (We will choose: leave as is if missing, matching bash-like but safer fallback, or substitute with empty string? We'll leave as is if missing).

## High-Level Design
1. Dependency injection: `uv add python-dotenv`.
2. `core.py`: Load `.env` using `dotenv.load_dotenv(USER_DIR / ".env", override=False)` before `settings.json` is parsed, so env vars from `.env` are available but do not overwrite shell-set vars.
3. Security check: Modify `check_security()` in `security.py` to verify `0o600` permissions on `.env` if it is present.
