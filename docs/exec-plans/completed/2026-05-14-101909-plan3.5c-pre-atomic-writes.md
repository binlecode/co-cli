# Plan — System-Wide Atomic File-Write Hygiene

Task type: refactoring

**Scope framing.** This is a **system-wide hygiene plan**, not a 3.5c-family prerequisite. The scope is every full-overwrite mutation path in `co_cli/` — across `memory/`, `skills/`, `agents/`, and `tools/system/`. The plan does two coherent things at once: (a) fixes write-atomicity gaps in load-bearing paths (`installer.py`, `index.py`, `dream.py`) so any future caller can rely on the contract; and (b) consolidates three independent atomic-write implementations (`_atomic_write_skill`, `write_curator_state`, `write_records`) into one canonical helper that all packages import.

The numbering `3.5c-pre` is retained only because the audit surfaced during 3.5c planning and the same fix happens to unblock 3.5c's drop-the-drain simplification. The work itself is not gated by, scoped to, or dependent on the 3.5c family — once this lands, *any* mutation path in `co_cli/` (current or future) uses the same canonical primitive.

## Scope — System-Wide, Cross-Package

This plan touches every full-overwrite mutation path in `co_cli/`. The scope is **horizontal across packages**, not vertical within one feature family:

### Why three helpers exist

Three independent atomic-write implementations exist:
1. `co_cli/memory/_mutator.py:atomic_write` — leading-underscore module (package-private per `CLAUDE.md` rule); cannot be imported outside `co_cli/memory/`.
2. `co_cli/tools/system/skills.py:_atomic_write_skill` — private to the tool; adds pid+uuid temp suffix.
3. `co_cli/skills/curator.py:write_curator_state` and `co_cli/skills/usage.py:write_records` — both use `tempfile.NamedTemporaryFile`-as-context-manager directly.

The fragmentation is the result of "I need atomic write here" being solved locally each time. Consolidation gives one well-tested helper and removes the duplication.

## Problem & Outcome

**Problem.** Full-overwrite mutation in `co_cli/` is solved locally each time: three independent atomic-write helpers (`_atomic_write_skill`, `write_curator_state`, `write_records`), one helper that's missing exception cleanup (`_mutator.atomic_write`), and three callsites with no atomicity at all (`installer.py:84`, `index.py:41-50`, `dream.py:106`). The result is a system-wide reliability gap with three failure shapes: (1) mid-write cancellation leaves partial `.md` files on disk; (2) the two-step FTS5 upsert can leave a ghost row indexed without chunks; (3) future authors writing mutation paths have to choose between four patterns. Fixing it once at the system level — one canonical helper, all packages — closes the gap permanently and removes the per-call-site decision.

**Failure cost.** A `kill -9` or OOM mid-`skill_manage` install (or mid-review-write under 3.5c) produces:
- Partial skill `.md` files on disk → frontmatter parse fail next session → skill silently dropped.
- FTS5 rows without chunks → `skill_search` returns hits that resolve to empty content → confusing UX.

**Outcome.** One canonical `atomic_write_text(path, content)` helper at `co_cli/memory/mutator.py`, imported by every full-overwrite mutation path in `co_cli/` (memory, skills, agents, tools/system). The two-step FTS5 upsert wraps both writes in a single SQLite transaction via a new public `MemoryStore.transaction()` context manager. Cancellation at any point leaves files and indexes in a consistent state — and any *new* mutation path added later inherits this contract by importing the helper. The downstream effect of unblocking 3.5c's drain-machinery removal is incidental, not the design goal.

## Behavioural Constraints

1. **Single canonical helper.** All full-overwrite mutation in `co_cli/` uses `co_cli.memory.mutator.atomic_write_text` (directly or via thin wrappers that call it). No new local `tempfile.NamedTemporaryFile` + `os.replace` blocks in mutation paths.

2. **Temp-file lifecycle.** `atomic_write_text` cleans up the tempfile on success (via `os.replace` consuming it) AND on exception (via the existing `NamedTemporaryFile(delete=False)` + a `try/except` that calls `Path(tmp.name).unlink(missing_ok=True)`). The current implementation in `_mutator.py:8-13` does NOT clean up on exception — this is a bug fix piggybacked on the promotion. Add the cleanup.

3. **Temp-file location.** Tempfile is created in `path.parent` (same directory as the target). Required for `os.replace` atomicity (cross-filesystem rename is not atomic on POSIX). The current `_mutator.py:9-11` already does this — preserve.

4. **Transaction semantics for FTS5 upsert.** `SkillIndex.upsert` runs both `_store.index` and `_store.index_chunks` inside a single SQLite transaction. On exception during `index_chunks`, the `index` insert rolls back. Achieved via `with self._store._conn:` (or a new `transaction()` method on MemoryStore — implementation choice during TASK-2).

5. **Success-path callers see no behavior change; failure-mode shape changes by design.** On the success path, atomic write is transparent — `Path.exists()`, `Path.read_text()`, and external observers see the file either fully old or fully new. File permissions are unchanged (`os.replace` preserves mode bits when target exists, or uses umask for new files — same as `write_text` today). On the failure path, behavior intentionally changes: a mid-write cancellation now leaves the *old* file intact instead of a partial *new* file. Similarly, a failed FTS5 upsert (BC4) now leaves no row instead of a split-state ghost row.

6. **MemoryStore exposes `transaction()` as a public context manager** (resolves Open Question 2). `SkillIndex` lives in `co_cli/skills/` — outside `co_cli/memory/` — so reaching into `self._store._conn` would violate the `_prefix.py` package-private rule (`CLAUDE.md` §"Engineering Rules"). The transaction primitive is the right public surface.

## Failure Modes

1. **Tempfile leak on exception.**
   - Mitigation: `try/finally` in `atomic_write_text` calls `unlink(missing_ok=True)` on the temp path if `os.replace` did not consume it. Test asserts no `.tmp` files remain in target dir after a write that raises.

2. **Cross-filesystem rename.** If `path.parent` is on filesystem A and the system tempdir is on filesystem B, `os.replace` is not atomic.
   - Mitigation: `tempfile.NamedTemporaryFile(dir=path.parent, ...)` keeps tmp on the same FS as target. Already the case in `_mutator.py:10`; preserve.

3. **FTS5 transaction edge: `_store.index` succeeds, `_store.index_chunks` fails.** Before this plan: row is indexed, chunks missing — split state. After this plan: both roll back; row is not present in FTS5 at all.
   - Implication: search returns no hit for the skill until the next `upsert` retry. Better than a ghost hit.

4. **Existing call sites that rely on `_atomic_write_skill`'s pid+uuid temp suffix.**
   - Mitigation: confirm no external observer (test, script, doctor command) scans for `*.tmp.{pid}.*` artifacts. If found, keep the suffix; otherwise drop it.

5. **MemoryStore does not expose `_conn` cleanly.** If `_conn` is leading-underscore (package-private), the SkillIndex fix needs a public `transaction()` method on MemoryStore instead of `with self._store._conn:`. The choice here is delegated to TASK-2 implementation.

## High-Level Design

### Helper consolidation

```python
# co_cli/memory/mutator.py
import os
import tempfile
from pathlib import Path


def atomic_write_text(path: Path, content: str, *, encoding: str = "utf-8") -> None:
    """Write content to path atomically via tempfile + os.replace.

    Tempfile lives in path.parent so os.replace stays on the same filesystem
    (required for atomic rename on POSIX). On exception at any point — entering
    the context, writing content, or os.replace — the tempfile is cleaned up.
    """
    tmp_path: Path | None = None
    try:
        with tempfile.NamedTemporaryFile(
            "w", dir=path.parent, suffix=".tmp", delete=False, encoding=encoding
        ) as tmp:
            tmp_path = Path(tmp.name)
            tmp.write(content)
        os.replace(tmp_path, path)
        tmp_path = None  # consumed by replace
    finally:
        if tmp_path is not None:
            tmp_path.unlink(missing_ok=True)
```

Note: `tmp_path` is assigned **immediately after entering the context** (before `tmp.write`), so a raise inside `tmp.write` still leaves `tmp_path` set and the `finally` cleans up the orphan. Assigning after `tmp.write` would leave a real `.tmp` file on disk if the write raised mid-flight (disk-full, encoding error). The test suite asserts this exact path.

All other helpers (`_atomic_write_skill`, `write_curator_state`, `write_records`) become one-liners delegating to this function, optionally with `mkdir(parents=True, exist_ok=True)` upstream of the call.

### FTS5 upsert transaction

```python
# co_cli/skills/index.py
def upsert(self, name: str, description: str, path: str) -> None:
    """Index or replace a skill row under source='skill'. Atomic across the two writes."""
    from co_cli.memory.text_chunker import Chunk

    with self._store.transaction():  # or `with self._store._conn:` if exposed
        self._store.index(source="skill", path=path, title=name, description=description)
        self._store.index_chunks(
            "skill",
            path,
            [Chunk(index=0, content=f"{name}: {description}", start_line=0, end_line=0)],
        )
```

If `MemoryStore.transaction()` doesn't exist, add it:

```python
# co_cli/memory/memory_store.py
@contextmanager
def transaction(self) -> Iterator[None]:
    """Group multiple index/remove calls into one SQLite transaction."""
    with self._conn:
        yield
```

## Open Questions

1. **Q:** Should `tools/files/write.py` (user-facing Write tool) be atomic-by-default?
   **Tentative answer:** No, out of scope for 3.5c-pre. The user-callable Write tool is direct-write by design. Atomic-rename would silently change symlink-following semantics (hermes hit this with config-file symlinks). Worth a separate grill if/when raised.

2. **Q:** Should `MemoryStore.transaction()` be added as a public API, or should `SkillIndex` reach into `self._store._conn` directly?
   **Decision (BC6):** Public `transaction()` method. Per `CLAUDE.md` §"Engineering Rules" `_prefix.py` rule, leading-underscore attrs are package-private; `SkillIndex` lives in `co_cli/skills/` (outside `co_cli/memory/`), so it must use a public surface.

3. **Q:** Should observability writers (`session_review.py`, `skill_curator.py` run.json/run.md) be migrated, given they're cosmetic if corrupted?
   **Tentative answer:** Yes. The migration is one-line per call site and parity simplifies future audits. The "cosmetic" caveat doesn't mean "should be inconsistent."
