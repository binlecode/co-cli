# spec-org-audit

## Problem

`docs/specs/` has grown to 17 files / 4,575 lines covering co-cli's subsystems. The shape evolved opportunistically: most specs are flat-named in one directory; some sit at very different sizes (`compaction.md` at 752 lines vs `sessions.md` at 111); a few patterns aren't applied consistently (the memory tier uses a `memory.md` foundation + two channel sub-specs, but skill and dream don't). The user wants a structured assessment of current partitioning + organization with concrete suggestions for which specs to split, merge, rename, or leave alone.

## Status

Open decisions resolved — ready for /orchestrate-plan spec-org-audit.

## Open Decisions Resolved — 2026-05-14

### D1. core-loop.md naming
- Question: rename `core-loop.md` to `loop-context.md`?
- Recommended: keep `core-loop.md` — "loop-context" narrows the focus while the doc also owns retries, interrupts, finalization.
- Chosen: keep `core-loop.md`.
- Why: title accurately describes the turn-orchestration territory the doc owns; "loop-context" would mislead readers.

### D2. Audit scope
- Question: full sweep, problem-focused, or naming-only?
- Recommended: full sweep — produces a coherent reorg plan.
- Chosen: full sweep, all 13 open decisions.
- Why: user wants a structured assessment, not a quick patch.

### D3. Partitioning principle (flat vs grouped)
- Question: flat directory, grouped subdirectories, or two-tier hybrid?
- Recommended: stay flat — 17 files is comfortably navigable; no link rewrites; sub-spec growth happens as new flat files.
- Chosen: stay flat.
- Why: directory regrouping is high-disruption with marginal navigational gain.
- Constraint: new sub-specs (e.g. `skill-lifecycle.md`, D8) stay at the top level using a hyphenated prefix convention.

### D4. compaction.md (752L) — split?
- Question: monolithic, layer-split (spill/proactive/recovery), or extract-recovery-only?
- Recommended: layer-split.
- Chosen: keep monolithic.
- Why: cascade L0→L1→L2→L3→Recovery is one mental model; cross-references between layers are dense; reader cost of one long file is lower than three.

### D5. Memory tier shape
- Question: keep 3-way split, trim foundation, collapse knowledge in, or one file?
- Recommended: keep 3-way split.
- Chosen: keep 3-way split (`memory.md` + `knowledge.md` + `sessions.md`).
- Why: foundation/channels pattern is the model the rest of the partitioning should imitate (per D8); channels are genuinely different (artifact lifecycle vs append-only transcripts).

### D6. 00-mission.md naming
- Question: drop the `00-` prefix?
- Recommended: rename to `mission.md`.
- Chosen: keep `00-mission.md`.
- Why: the numeric prefix is a deliberate "read this first" signal; user wants to extend the convention (see D9).
- Constraint: this decision is load-bearing for D9.

### D7. prompt-assembly.md placement
- Question: keep standalone, fold into core-loop.md, or fold into personality.md?
- Recommended: keep standalone.
- Chosen: keep standalone.
- Why: it sits at the seam between core-loop, personality, and compaction; doesn't belong wholly inside any of them.

### D8. Skill self-evolution coverage (plan 3.5b)
- Question: new `skill-lifecycle.md` sub-spec, section in `skill.md`, or two skill sub-specs?
- Recommended: new `skill-lifecycle.md`.
- Chosen: new `skill-lifecycle.md`.
- Why: mirrors memory.md+channels pattern; keeps `skill.md` from drifting past 700L as 3.5c + plan 4 ship.
- Constraint: `skill.md` becomes the foundation (storage, dispatch, model surface); `skill-lifecycle.md` owns usage sidecar, reviewer, curator state machine, archive/restore, `/skills curator` CLI.

### D9. docs/specs index
- Question: add `README.md`, keep `system.md` as the index, or both?
- Recommended (initial offering): add `README.md`.
- Chosen: rename `system.md` → `01-system.md` so the numeric-prefix carries the reading-order signal alongside `00-mission.md`.
- Why: extends D6's convention; no need for a separate README file; directory listings naturally order entry-point specs first.
- Constraint: this is the second numbered spec (after `00-mission.md`); other specs stay flat unless the convention is deliberately extended.

### D10. Approval system — extract own spec?
- Question: keep distributed, extract `approvals.md`, or extract approvals + shell classifier?
- Recommended: keep distributed.
- Chosen: keep distributed.
- Why: approval mechanics aren't large enough to merit own real estate; each touching spec owns its own approval surface.

### D11. Backward-compat sections — sunset policy
- Question: keep as-is, prune older than two versions, or centralize in HISTORY.md?
- Chosen: **remove all backward-compat sections; adopt project-wide zero-backward-compat rule.**
- Why: Co is brand new with no installed base. Compat shims (aliases in code, "Removed/Renamed" tables in specs, sunset timelines) are pure system overhead with no upside. `git log` is the authoritative history.
- Constraint: project-wide rule, not just spec cleanup. Saved as feedback memory `feedback_zero_backward_compat`. Applies to code (renames are immediate, no aliases), specs (no compat tables added going forward), and exec-plans (do not list backward-compat as a constraint).

### D12. Spec partitioning rubric
- Question: add a short rubric to `agent_docs/spec-conventions.md`, leave informal, or rubric + lint?
- Recommended: add a short rubric.
- Chosen: add a short rubric.
- Why: captures these decisions so the next person doesn't re-litigate; lint enforcement is premature given the agreed-on `compaction.md` exception.
- Constraint: rubric body — split when >500L AND 3+ separable sub-domains, OR sibling sub-spec already exists with parallel structure. Merge when <80L AND no growth runway. Never split for pure topic-count. Mass renames are immediate (no "Compat Notes" sections; see [[feedback_zero_backward_compat]]).

### D13. dream.md vs knowledge.md
- Question: keep separate, rename to `knowledge-lifecycle.md`, or merge?
- Recommended: keep separate.
- Chosen: keep separate as-is.
- Why: dream is a batch lifecycle pass with its own orchestration, prompts, timeouts, and state file; knowledge owns surface + storage. Cross-link is correct.

### D14. self-planning.md placement
- Question: keep top-level, rename to `planning.md`, or fold into `tools.md`?
- Recommended: keep top-level.
- Chosen: keep top-level as `self-planning.md`.
- Why: real subsystem with own tools, runtime state, and compaction integration; the `self-` prefix disambiguates from human-authored exec-plans.
