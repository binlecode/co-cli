# Plan: Split errors log

## Context
Currently `co-cli.jsonl` contains both operational info logs, tool call arguments/results, model interactions, and system errors. Autonomous agents can be noisy. `co-cli` does not have an easy-to-read, isolated file purely for warnings and errors (warnings/errors are currently dumped into `co-cli.jsonl`). 

Based on peer reviews (`hermes-agent`, `ElizaOS`), splitting errors into a dedicated log file (`errors.jsonl` or `errors.log`) is a critical operational standard for debugging autonomous failure loops without wading through gigabytes of tool/model JSON tracing.

The current setup utilizes a `RotatingFileHandler` attached to the root logger via the `_attach_handler` function in `co_cli/observability/_file_logging.py`. This function is idempotent and extremely easy to reuse.

## Problem & Outcome
**Problem:** When the agent hits a doom-loop or API connection issue, identifying the root cause requires tailing/grepping `co-cli.jsonl` and visually parsing past massive `ToolReturn` JSON structures.
**Failure cost:** High debugging friction for developers and maintainers when diagnosing silent failures or autonomous loop failures.
**Outcome:** A dedicated `errors.jsonl` file that contains *only* `WARNING` and `ERROR` logs. It uses the same rotating log behavior (max 2MB, 2 backups).

## Scope
- Update `co_cli/observability/_file_logging.py` to add a second rotating handler for `errors.jsonl`.
- Update `docs/specs/observability.md` to document the new file.

## Behavioral Constraints
- No changes to existing `co-cli.jsonl` behavior. It remains the master file for INFO+ and OTel span records.
- `errors.jsonl` must be rotated at a fixed 2MB / 2 backups (hardcoded, mimicking hermes, does not need to use observability settings).
- `errors.jsonl` must format output as JSON using the existing `_JsonRedactingFormatter`.
- Log level for `errors.jsonl` must be `logging.WARNING`.

## High-Level Design
In `co_cli/observability/_file_logging.py` inside `setup_file_logging`, call `_attach_handler` a second time:

```python
    # New handler for errors/warnings only (WARNING+)
    _attach_handler(
        root,
        log_dir / "errors.jsonl",
        level=logging.WARNING,
        max_bytes=2 * 1024 * 1024,
        backup_count=2,
        formatter=formatter,
    )
```

Because `_attach_handler` ensures idempotency by checking `existing.baseFilename == target`, calling it for a second distinct file is entirely safe and conforms to the existing initialization architecture.
