# `/status` — consolidated current-state report (openclaw-style snapshot, no new history)

Task type: additive REPL feature — one read-only slash command reusing existing status sources; no new state, no store, no schema change.

## Context

co-cli surfaces status in three disjoint places, none of which is a single on-demand report:

- **Startup banner** (`co_cli/bootstrap/banner.py:58-120`, `display_welcome_banner`) — one-time at launch: version, model, memory line (`build_memory_line`, `:41`), dream line (`build_dream_line`, `:24`), and `Tools/Skills/MCP/Commands` counts. Counts computed inline (`:77-80`); Dir + git-branch at `:114-115`.
- **Footer toolbar** (`co_cli/display/core.py:409` `render_footer_toolbar`, fed by `StatusSnapshot` at `:239-248`, built by `_build_status_snapshot` in `co_cli/main.py:492-509`) — continuous but compact: session label, mode, context %, background-task count, approval count, queue depth/preview.
- **Per-domain slash commands** — `/dream` (`commands/dream.py:23` `handle_dream_slash`), `/usage` (`commands/usage.py:28` `_cmd_usage`), `/memory`, `/tools`, `/skills`, `/tasks`, `/background`, `/approvals`, `/sessions`, `/history` — each owns one slice.

The consolidated "where am I right now" view exists only at startup (banner), then scrolls away; the footer is too small for detail; reconstructing the full picture mid-session means running 4–6 commands.

**Peer survey** (`docs/reference/RESEARCH-status-command-peer-survey.md`, 2026-06-14): hermes, openclaw, and opencode all ship `/status`, **all current-state snapshots, none carrying history** — history is universally delegated to `/sessions` `/history` `/usage`. openclaw is the rich template: one command fanning out over ~10 subsystems into emoji/section plain text, with no blocking I/O at render except one time-boxed probe. co already owns the *history* half; the gap is the **consolidated snapshot**.

### Code-state verification (claims checked against HEAD)

- `BUILTIN_COMMANDS` is **defined** in `co_cli/commands/registry.py:26` and **populated** in `co_cli/commands/core.py:45-96` via `BUILTIN_COMMANDS["name"] = SlashCommand(name, help, handler)`. The new handler module must `from co_cli.commands.registry import BUILTIN_COMMANDS` (as `banner.py` does), not from `core`. Handler signature: `async def handler(ctx: CommandContext, args: str) -> None`; output via `from co_cli.display.core import console` (confirmed in `handle_dream_slash`, `_cmd_usage`).
- **Every needed datum already has a cheap, synchronous, local source:**
  - session id / dir / branch — `deps.session.session_path.stem`; `deps.workspace_dir` + `deps.config.workspace_path`; git-branch via the same lookup the banner uses (`banner.py:114-115`).
  - model + personality — `deps.config` (llm provider/model); `deps.config.personality`.
  - context % — `deps.runtime.current_request_tokens_estimate / deps.model_max_ctx` (exact calc lifted from `_build_status_snapshot`, `main.py:498-500`).
  - work-in-flight — `deps.session.background_tasks`, `deps.session.session_approval_rules`, REPL input queue (the same three the footer reads).
  - dream — `status_daemon(USER_DIR)` (`daemons/dream/process.py`, filesystem read) + `HousekeepingState` at `$CO_HOME/daemons/dream/_dream_state.json` (`last_housekeeping_at`, cumulative `stats`).
  - cumulative tokens + cost — `aggregate(deps.usage_log_path, ...)` → `UsageTotals` (`commands/usage.py:38,52`).
  - counts — `len(deps.tool_catalog)`, `len(get_skill_catalog(deps.skill_catalog))`, `len(deps.config.mcp_servers or {})`. The command count is **not** `len(BUILTIN_COMMANDS)` alone: the banner (`banner.py:80-82`) computes `cmd_count = len(BUILTIN_COMMANDS) + sum(1 for s in deps.skill_catalog.values() if s.user_invocable)` (builtins + user-invocable skills). The TASK-1 helper must reproduce this exact formula to keep banner output byte-identical.
  - memory backend + counts — `build_memory_line` formats this today from the memory store.
- No datum requires network or model invocation. `status_daemon` and `aggregate` are local file reads — the openclaw "no blocking I/O at render" discipline is achievable for free.

## Problem & Outcome

**Problem.** The consolidated current-state view exists only at startup and can't be re-summoned; mid-session, status is fragmented across the footer and ~6 commands.

**Outcome.** A read-only `/status` slash command prints one sectioned report of current state, assembled on demand from the sources above — a superset of the banner, made queryable any time, copy-pasteable.

**Failure cost:** Without it, a user mid-session who wants the full picture (is the dream daemon alive? how close to the context limit? how much has this session cost? how many background tasks/approvals pending?) must run and mentally stitch 4–6 separate commands, and the startup banner's consolidated view is unrecoverable once scrolled away. Silent failure mode: a crashed dream daemon or a near-limit context is easy to miss because no single surface shows it on demand.

## Scope

**In scope.** One read-only `/status` slash command; extraction of the banner's count/branch/dream/memory data into reusable helpers so banner and `/status` share one source of truth; functional tests.

**Out of scope.**
- Per-turn history, timelines, trends in `/status` — no peer does this; `/usage`/`/history`/`/sessions` own it.
- A non-interactive `co status` CLI command — possible follow-up; `co dream status` already covers the daemon slice. (Open Question 2.)
- Footer/banner changes — separate concern.
- A reactive central status store (opencode's model) — heavier than co needs; the ad-hoc assembly path (`_build_status_snapshot`) is the chosen pattern per the survey.
- `docs/specs/tui.md` update — handled post-delivery by `/sync-doc`, not a task here.

## Behavioral Constraints

- **Read-only.** Invoking `/status` must never write a memory item, skill, file, or daemon state, and must never enqueue a model request. (Testable: no new files under `CO_HOME` after invocation.)
- **No blocking I/O at render.** Only local filesystem reads (`status_daemon`, `aggregate`) and in-memory `deps` reads are permitted; no network call and no model invocation, even when the dream daemon is enabled/running.
- **Never raise to the REPL.** A failure to read any one section's source (e.g. missing `_dream_state.json`, empty usage log) must degrade that section to a placeholder (`—` / "unavailable"), not abort the whole report. The command must always print something.
- **Snapshot, not history.** `/status` shows only current values plus already-cumulative counters that other surfaces already expose (session token total, housekeeping stats); it must not introduce any new persisted time-series.
- **No banner output change.** The TASK-1 extraction must leave `display_welcome_banner` output byte-identical.

## High-Level Design

`/status` is a thin read-only handler that gathers from existing sources and prints a sectioned report (mirroring openclaw's grouping, rendered with co's existing `console` / `make_table` styling):

- **Session** — id (last 8), dir + git branch, personality/soul, mode (idle/active).
- **Model & context** — provider/model, context % (reuse `_build_status_snapshot` calc), cumulative session tokens + cost (from `aggregate(..., origin=ORIGIN_SESSION)`).
- **Dream** — enabled / running / disabled + queue depth + `last_housekeeping_at` (reuse `handle_dream_slash`'s three branches over `status_daemon`).
- **Work in flight** — background tasks, pending approvals, input-queue depth.
- **Capabilities** — memory backend + item/session counts; Tools/Skills/MCP/Commands counts.
- **Degraded flags** — any memory-degradation flag the banner already surfaces.

To avoid a second divergent copy of the banner's logic, the banner's count/branch/dream/memory data is first lifted into values-returning helpers (TASK-1) that both the banner and `/status` call. Each section reads from its source under a guard so one missing source degrades to a placeholder, never aborts (per Behavioral Constraints).

### ✓ DONE TASK-2 — `_cmd_status` handler + registration

- **files:** `co_cli/commands/status.py` (new), `co_cli/commands/core.py`

If profiling during implementation shows it is slow on a large ledger, fall back to showing session tokens from the live runtime estimate and cross-link `/usage` for cumulative cost rather than recomputing.
