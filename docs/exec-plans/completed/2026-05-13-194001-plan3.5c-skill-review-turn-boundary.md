# Plan 3.5c — Skill Review Turn-Boundary Firing

Task type: code-feature

## Context

Convert the existing session-end review trigger into hermes's turn-boundary model. The session reviewer agent and its prompts carry forward unchanged; only the trigger changes.

- Counter `iterations_since_review` on `CoSessionState`, bumped per turn by `turn_result.tool_iterations` — the count of `ModelResponse`s in the turn's **new** messages that contain at least one `ToolCallPart` (one bump per API round-trip with tools, approximating hermes's `_iters_since_skill` at `run_agent.py:10476` but gated on tool-producing iterations only).
- After each `_finalize_turn`, if counter ≥ `review_nudge_interval` (default 5; see §"Default threshold rationale" below), spawn `asyncio.create_task(_maybe_run_session_review(deps, history_copy))`. Reset counter.
- Review runs concurrently with the next user turn; never awaited by the foreground turn loop.
- Single-in-flight: skip if a previous review task is still running; counter NOT reset on skip. Concurrent skill-library writes by overlapping reviewer and foreground `skill_manage` calls are safe via the atomic-write helper + FTS5 transaction.
- On REPL exit, **cancel** any pending review task (no await). No inline review fires at exit (hermes parity).
- Sessions shorter than `review_nudge_interval` produce no review — accepted loss. Default chosen at 5 (not 10) based on local session-length measurement; see §"Default threshold rationale".
- Library-shape maintenance (umbrella discipline for narrow skills) is handled **preventively** by the reviewer's 3-tier umbrella-preference prompt; no reactive consolidation agent.

## Problem & Outcome

**Problem.** Session-end review fires exactly once per session, inline at REPL exit. Long sessions get a single extraction pass over a long transcript that may exceed the reviewer's iteration budget. Mid-session learnings the user encounters in a follow-up session may be lost.

**Outcome.** Turn-boundary firing distributes review work across the session lifetime. Each fire covers the most recent ~5 tool-call iterations as its high-signal slice. More, smaller passes; fresher per-pass context; the foreground turn loop is never blocked.

### Out of scope

- **Tail-fallback at REPL exit.** Hermes parity — no inline review fires at session end. Short sessions (< `review_nudge_interval`) produce no review; their learnings are accepted as a marginal loss. Avoids the up-to-120s shutdown stall the tail would have introduced.
- **Reactive consolidation.** Library-shape maintenance (umbrella rollup of narrow skills) is handled preventively by session_review's 3-tier umbrella-preference prompt, not by a separate consolidation agent.
- **Garbage collection / archival.** Skills accumulate; users manage manually. No state machine.
- **Pre-review history truncation.** The reviewer sees the same transcript shape as session-end review; turn-boundary just fires more often.
- **Cross-fork coordination beyond single-in-flight.** No work-stealing, no review queueing, no priority.
- **Mid-session prompt rebuild for the foreground agent.** The static `<available_skills>` manifest is rendered once at startup (`co_cli/main.py:412-414`) and baked into the agent's system instructions. Skills created mid-session by a turn-boundary reviewer are **not** visible to the foreground model until the next `co chat` boot, even after `refresh_skills` updates `deps.skill_commands`. This is pre-existing behaviour; turn-boundary firing makes it more user-visible but doesn't change it. Addressing requires rebuilding the agent on skill-catalog change — out of scope for 3.5c.

## Behavioural Constraints

(Reviewer agent, prompts, and status callback are unchanged from the prior session-end implementation. Only the BCs added by this plan are listed.)

1. **Turn-boundary review concurrency model.** Review fires as `asyncio.create_task(...)` from the post-turn hook. The next user turn begins immediately; review runs concurrently. If a new trigger fires while a previous review is still running, skip (single in-flight on the review task). Concurrent skill-library writes by the reviewer and the foreground agent's `skill_manage` calls are made safe by `atomic_write_text` (`co_cli/persistence/atomic.py`) + `MemoryStore.transaction()`. On REPL exit, the pending review task is **cancelled, then bounded-drained for up to 2s** (lets cancellation finalizers run cleanly; correctness is guaranteed by atomic-write atomicity regardless of whether the drain completes).

2. **Counter does not reset on skip.** When a trigger fires but is skipped (in-flight check fails), the counter is NOT reset. The next eligible trigger after the in-flight task completes fires immediately.

3. **No inline review at REPL exit.** `_drain_and_cleanup` cancels the pending review task, drains for ≤2s, and proceeds to dream cycle + cleanup. The pre-3.5c session-end review call at `_drain_and_cleanup:146` is **deleted entirely** — turn-boundary firing during the session is the only review path. Sessions shorter than `review_nudge_interval` produce no review (hermes parity). This avoids the up-to-120s shutdown stall a tail-fallback would impose; the cost is accepting that very short sessions contribute nothing to the skill library.

4. **Background task yields before sync setup.** `_maybe_run_session_review` adds `await asyncio.sleep(0)` at entry — before the `review_enabled` / `deps.model` guards and before delegating to `run_session_review`. This yields control to the event loop so the REPL can accept user input before `fork_deps_for_reviewer` + `build_agent` + `serialize_messages` execute synchronously. The yield is the structural guarantee; no numeric budget is asserted (the hook itself is a counter check + `asyncio.create_task` call, both trivially fast).

5. **Counter advances on error and interrupted turns — but only for cleanly-completed segments.** `_run_foreground_turn` reaches `_finalize_turn` (and therefore `_post_turn_hook`) whenever `run_turn` returns a `TurnResult` — including `outcome == "error"` and `interrupted == True`. The counter accumulates `turn_result.tool_iterations` from those turns too. The accumulator increment lives at orchestrate.py ~394, *after* `turn_state.latest_result = result` — so it counts each segment that returned a result, even if a subsequent segment then errors. Mid-stream failures inside `_execute_stream_segment` itself (e.g. `httpx.ReadError`, network teardown before `AgentRunResultEvent`) raise before line 394 and contribute zero to the count, even though partial tool calls may have streamed. This is acceptable: a torn segment has no `result.new_messages()` to inspect anyway, so there is nothing reliable to count. Tests verify error- and interrupted-turn counter bumps for the completed-segment case. The hook itself is only skipped when `run_turn` raises without returning a `TurnResult` (no counter mutation happens).

## Default threshold rationale

The default `review_nudge_interval` is **5**, lower than hermes's analogue of 10. Justification is measured from the current user's local session history (`~/.co-cli/sessions/*.jsonl`, 7 sessions sampled 2026-04 to 2026-05):

| Metric | Value |
|---|---|
| Median tool iterations per session | 6 |
| Mean | 5 |
| Sessions ≥ 10 iters | 1/7 (14%) |
| Sessions ≥ 5 iters | 4/7 (57%) |
| Sessions ≥ 3 iters | 4/7 (57%) |

With threshold 10, 6/7 sessions in the sample would produce no review — the no-tail-fallback choice would consistently lose review on the modal case. Threshold 5 catches the median session exactly once and is the smallest interval that still leaves room for the reviewer's ~120s timeout budget to overlap a typical session's remaining turns. Sessions with < 5 iterations (3/7 in the sample, mostly trivial one-shot queries) still produce no review; this is accepted as a marginal loss.

Sample is small (n=7) and skewed toward one developer's workflow. The env var `CO_SKILLS_REVIEW_NUDGE_INTERVAL` lets any user re-tune without code changes. The default is a reasoned starting point, not a permanent commitment; re-evaluate after broader telemetry. Hermes parity is preserved on shape (no tail fallback, single-in-flight, atomic-write safety) — only the threshold magnitude differs, and intentionally so.

## Failure Modes

1. **Turn-boundary review delays the next user turn.**
   - Mitigation: review spawned via `asyncio.create_task(...)`, never awaited by foreground turn loop. The hook body is a counter check + `asyncio.create_task` call; both are trivially fast and not on the critical path. No numeric budget asserted.

2. **Two reviews fire concurrently on fast-moving session.**
   - Mitigation: single in-flight on the review task. New trigger while previous running is skipped; counter NOT reset so a future eligible trigger gets a fresh fire after the in-flight task completes.

3. **Pending review still running at REPL exit.**
   - Mitigation: `_drain_and_cleanup` **cancels** the pending review task, then waits ≤2s for cancellation finalizers (bounded drain). Safe due to atomic-write atomicity — any in-flight skill or knowledge write is either fully old or fully new. The pending review's findings beyond the last atomic boundary are discarded; the next session's review picks up from where this one stopped. If the task ignores cancellation for 2s, shutdown proceeds regardless.

4. **Reviewer patches a skill the foreground agent is about to use.**
   - Mitigation: review writes are atomic via `atomic_write_text`; foreground reads from `deps.skill_commands` (refreshed after each write via `refresh_skills`). Race is brief and benign — at worst, foreground reads a slightly-newer version. Skill file format is stable across edits.

5. **Reviewer hangs in background.**
   - Mitigation: outer `asyncio.wait_for(REVIEW_TIMEOUT_SECONDS)`. On timeout, the task is cancelled and the error logged. The agent's `REVIEW_MAX_ITERATIONS` budget is the inner limit.

6. **Status callback surfaces while user is typing.**
   - Mitigation: existing `TerminalFrontend._status_live` buffering already handles this (see `co_cli/display/core.py:282–285`). Status messages buffer during active input and flush when input ends.

7. **Background task does sync setup before first `await`, briefly blocking the REPL.**
   - Mitigation: `_maybe_run_session_review` adds `await asyncio.sleep(0)` at entry (per BC4 above). Structural guarantee — no numeric budget asserted.

8. **Short session (< `review_nudge_interval` iterations) produces no review.** With no tail-fallback, sessions that never trip the threshold and have no completed turn-boundary fires contribute nothing to the skill library.
   - Mitigation: accepted by design (hermes parity). Learnings from very short sessions are marginal. Users who want every session reviewed can lower `review_nudge_interval` via env var; the cost is more frequent reviews during longer sessions.

9. **Narrow-skill proliferation across multiple review passes.** With turn-boundary firing, multiple review passes happen per session. Pass-B could create a sibling of pass-A's creation (e.g. `test-fixture-pattern.md` + `fixture-conventions.md`).
   - Mitigation: session_review's prompt enforces the 3-tier umbrella-preference order (UPDATE LOADED → UPDATE EXISTING UMBRELLA → CREATE NEW UMBRELLA), and `render_skill_manifest` at `co_cli/agents/session_review.py:122-124` prepends the full skill manifest (bundled + user-installed) to the reviewer's instructions, so prior passes' creations are visible to subsequent ones within the same session.
   - **Caveat — registry refresh + render reorder required.** `fork_deps_for_reviewer` shares `parent.skill_commands` by reference at fork time (`co_cli/deps.py:349`), but `set_skill_commands` *rebinds* the field on whichever fork calls it (`co_cli/skills/registry.py:14`). When pass-A's `skill_manage` runs inside its child deps, only the child's field is updated — the parent's dict still points at the old contents, so pass-B forks from the parent and sees a stale manifest. **Compounding issue:** `run_session_review` currently renders the manifest *before* the fork (lines 122-124, against `deps.skill_commands`), so a post-fork `refresh_skills(child_deps)` alone would not change the already-built instructions string. **Fix:** reorder `run_session_review` so fork → `refresh_skills(child_deps)` → `render_skill_manifest(child_deps.skill_commands, child_deps.skills_dir, child_deps.user_skills_dir)` → build instructions → `build_agent(...)`. Disk is shared and atomic, so pass-B sees pass-A's writes once it reloads. This is in scope for 3.5c (small reorder in TASK-1).

### Post-turn hook

Integration point: `co_cli/main.py:_run_foreground_turn` at line 135. Current shape:

3.5c wraps so the hook fires between finalize and return:
```python

next_history = await _finalize_turn(turn_result, message_history, deps, frontend)
_post_turn_hook(deps, next_history, turn_result.tool_iterations)
```

`_post_turn_hook` is intentionally sync — its body is constant-time (counter bump + threshold check + task spawn). No `await`s inside.

**Where the count comes from.** `turn_result.tool_iterations` is an authoritative accumulator populated inside `co_cli/context/orchestrate.py`: a per-segment increment in `_execute_stream_segment` (right after `turn_state.latest_result = result` at line ~394), summing `ModelResponse`s with at least one `ToolCallPart` from `result.new_messages()`. The accumulator is carried on `_TurnState`, then copied into `TurnResult.tool_iterations` at each of the three build sites (success / error / interrupted).

This is the right layer because a single user turn can span multiple `_execute_stream_segment` calls (initial segment + approval-resume segments via `_run_approval_loop` + HTTP 400 reformulation retries + overflow-recovery retries). `latest_result.new_messages()` only covers the latest segment, and slicing `turn_result.messages[prior_len:]` would fail on overflow-recovery compaction (line 714 replaces `current_history` with the compacted version). Per-segment accumulation is immune to both.

The hook itself:
```python
# co_cli/main.py — new helper
def _post_turn_hook(
    deps: CoDeps,
    message_history: list[ModelMessage],
    turn_iteration_count: int,
) -> None:
    """Bump iterations counter; spawn background review when threshold tripped.

    Sync function — body is constant-time; the actual review runs via
    asyncio.create_task. Safe to call from any await point.
    """
    if deps is None:
        return
    settings = deps.config.skills
    if not settings.review_enabled:
        return
    # Symmetry with _maybe_run_session_review:177-178 — skip silently when no
    # model is wired (test fixtures, bootstrap paths). Avoids spawning a task
    # that would immediately return.
    if deps.model is None:
        return

    deps.session.iterations_since_review += turn_iteration_count
    if deps.session.iterations_since_review < settings.review_nudge_interval:
        return

    # Single in-flight: skip if previous review is still running.
    # Concurrent skill-library writes by reviewer + foreground skill_manage
    # are made safe by atomic-write atomicity, so no further coupling needed.
    task = deps.session.background_review_task
    if task is not None and not task.done():
        return  # counter NOT reset; subsequent triggers will retry once task completes

    deps.session.iterations_since_review = 0
    # Reuse _maybe_run_session_review directly — it already handles wait_for + timeout
    # + status callback + error swallow. The only delta for turn-boundary firing is
    # that we wrap it in create_task instead of awaiting it inline.
    deps.session.background_review_task = asyncio.create_task(
        _maybe_run_session_review(deps, list(message_history))  # defensive copy
    )
```

Note: no separate `_run_review_bg` helper. The modified `_maybe_run_session_review` at `co_cli/main.py:173-195` (with `await asyncio.sleep(0)` at entry + `except asyncio.CancelledError` block added by this plan) is used directly as the task body, wrapped in `asyncio.create_task(...)`.

Why cancel-with-bounded-drain for review: `atomic_write_text` guarantees that any in-flight skill/knowledge write is at an atomic boundary — either fully old or fully new. The pending review's remaining work (anything past the last completed atomic write) is discarded, but no corruption survives. The 2-second drain just lets the cancelled task's `except asyncio.CancelledError` branch log and unwind cleanly; if the task ignores cancellation for 2s, we proceed regardless.

**Note on dream-cycle ordering.** The cancelled review's pending outputs (if any) are not visible to the dream cycle. This is acceptable because the dream cycle's primary input is knowledge artifacts already written to disk by prior completed turn-boundary reviews — not a session-end "final consolidation pass".

**Note on message_history parameter.** Deleted as part of 3.5c (TASK-2). After removing the inline review call, the parameter is unused and must be dropped per zero-backward-compat. Caller at `co_cli/main.py:486` is updated in the same edit.
