## Problem & Outcome

**Problem.** co assembles one context budget and one static prefix regardless of model. A frontier model is about to be selectable alongside a weak local one whose limits the prompt is tuned to counter; the 65k budget wastes gemini's 1M window, and there is no seam for any prompt to differ per model.

**Failure cost.** (1) gemini runs on qwen's 65k budget — early-compacting, headroom wasted. (2) gemini-pro adds an inline budget branch and a later overlay adds an inline prompt branch → two drifting model-branches (erosion). (3) Over-build a speculative prompt fork now → violates measure-first.

**Outcome.** One `ModelProfile` resolved from `config.llm`, consumed by both `max_context_tokens` and the static instruction builders. Default qwen path is **byte-identical prompt + identical budget** (pure capability). gemini-pro's budget comes from the profile (folds gemini TASK-2). The seam exposes a profile-overlay hook that returns `None` today — B/C fill it later, evidence-gated.

**Shippable contract:** the profile seam + gemini budget consuming it + the default-path byte-identity/budget tests. No prompt content changes.

## Behavioral Constraints
- **Default backend unchanged.** Ollama/qwen byte-identical prompt + identical `max_context_tokens`; acceptance asserts both.
- **WEAK_LOCAL 64k is a hard clamp.** `WEAK_LOCAL -> 65_536` is the fixed baseline the weak-model profiling (plans 2/3) is calibrated against — never relaxed or made model-tunable. Only the FRONTIER budget is deferred/relaxed (OQ-3).
- **One resolver, no scattered checks.** `ModelProfile` resolved once from `config.llm` near `_LLM_SETTINGS`; both budget and instruction builders read it. No inline `if provider==` in assembly or compaction.
- No hardcoded `~/.co-cli`; centralized knobs in `_LLM_SETTINGS` / profile, never inline (`feedback_evals_centralized_settings`, `feedback_tests_use_config_model_settings`).
- Zero-backward-compat (`feedback_zero_backward_compat`): no shim for the old single-budget path.
- This plan edits **no rule `.md`** → no floor-guard delta. (B/C carry the floor-guard discipline.)

1. **Profile granularity.** Binary `frontier | weak_local` now, or a capability tuple later? Default: **binary** — the only distinction with evidence; revisit at a third model.
2. **Overlay mechanism shape.** A `None`-returning `static_instruction_builder` closure (cleanest — matches `_personality_critique_provider`) vs a `profile` param threaded into `build_base_instructions`. Default: the closure (additive, no base-builder signature churn). Resolve at TASK-1.

Decision: the profile supplies the **default** for that field — the field stays authoritative, a user override still wins, the num_ctx contract is preserved. WEAK_LOCAL default = hard 65_536; FRONTIER default = half the provider's 1M max window (524,288). co's existing `compaction_ratio = 0.50` (hermes-parity single-shot 50%) is the clamp off the budget — the peer-converged pattern (clamp = ratio × window, never a magic absolute). The pricing-cliff cost-clamp is deliberately NOT gauged in plans 1/2/3 — gauging it now is useless and costly; it is deferred to a post-1/2/3 calibration-driven setting once real frontier usage exists. Peer anchor for that future value: opencode's 200K cost-tier boundary (`session.ts:432`) — a *cost* clamp, applied only when cost (not fit) drives.
