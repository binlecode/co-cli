# Plan: Remove M2c (compact_assistant_responses)

Task type: refactor

---

## Context

`compact_assistant_responses` (M2c) is the third history processor in co-cli's compaction stack. It truncates any `TextPart` or `ThinkingPart` in older `ModelResponse` messages that exceeds 2,500 chars, keeping the first 20% and last 80% of the content with a `[...truncated...]` marker.

Research performed 2026-04-21 established two independent grounds for removal:

1. **Active bug — signed ThinkingPart corruption.** pydantic-ai's `ThinkingPart` carries a `signature` field (Anthropic, Bedrock, Google, OpenAI all sign thinking blocks against the full content). M2c mutates `part.content` via `_truncate_proportional` but leaves `signature` intact. On the next API call, the provider receives a content/signature mismatch and returns HTTP 400. hermes documents exactly this failure mode in `run_agent.py:9763`: "Any upstream mutation (context compression, session truncation, message merging) invalidates the signature → HTTP 400."

2. **No valid design rationale for TextPart.** Neither fork-claude-code nor hermes truncates assistant text content in the live context. TextPart in M3's tail is preserved verbatim — M2c truncates it unnecessarily, losing signal. TextPart in M3's middle is fed to the summarizer — M2c degrades the summarizer's input before M3 fires. There is no peer precedent and no design rationale that survives scrutiny.

`_truncate_proportional` and `OLDER_MSG_MAX_CHARS` have no callers outside M2c, its tests, and its eval section — both are safe to delete.

No replacement is needed. ThinkingPart accumulation is addressed by M3 dropping entire messages from the middle; partial truncation of ThinkingPart is worse than both alternatives (keep-full or drop-entirely).

Workflow artifact hygiene: no stale plans found for this scope.

## Problem & Outcome

**Problem:** `compact_assistant_responses` silently corrupts signed `ThinkingPart` blocks — it truncates `content` while leaving `signature` pointing to the original full content. Any turn where the corrupted `ThinkingPart` is in M3's preserved tail will produce HTTP 400 on the next API call.

**Failure cost:** API calls fail with HTTP 400 on any session that uses extended thinking and where M2c has fired on a ThinkingPart still in the tail. The error is silent to the user until the next turn.

**Outcome:** M2c is completely removed. The compaction stack runs three history processors (M2a truncate_tool_results, M2b enforce_batch_budget, M3 summarize_history_window). ThinkingPart blocks are never mutated outside of M3's boundary planner (which drops entire messages, not parts).

## Behavioral Constraints

- BC1: After removal, `history_processors` in `build_agent()` must contain exactly three processors in order: `truncate_tool_results`, `enforce_batch_budget`, `summarize_history_window`.
- BC2: No `ThinkingPart.content` or `ThinkingPart.signature` field must be mutated anywhere in the processor chain.
- BC3: `_truncate_proportional` and `OLDER_MSG_MAX_CHARS` must not appear anywhere in `co_cli/` after removal.
- BC4: Removal must not affect M2a, M2b, or M3 behavior — all existing tests for those processors must continue to pass.

It should either be explicitly acknowledged in the plan scope or extracted into a separate targeted fix with its own tests.
