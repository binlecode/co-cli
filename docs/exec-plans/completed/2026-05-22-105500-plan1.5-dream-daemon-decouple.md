# dream-daemon-decouple

## Problem

Plan 1 (`online-reviewer-and-daemon-mvp`, shipped 2026-05-22) implemented the dream daemon with **two parallel cross-process mechanisms** where one was sufficient:

1. **File queue** at `~/.co-cli/dream/queue/*.json` — durable producer-consumer bridge with atomic writes, idempotent drain, and startup catch-up via `_initial_drain()`.
2. **Unix socket** at `~/.co-cli/dream/sock` — used both for control messages (STATUS, STOP) and for redundant wake-up signaling (`REVIEW` opcode).

The file queue alone provides durability, atomicity, idempotency, and producer/consumer decoupling on liveness. The socket-as-wake-up channel adds:

- A hidden ordering invariant (file-first, socket-second) in `_send_review_kick` with zero compile-time enforcement — a refactor that flips the order silently breaks correctness
- A silent failure mode: best-effort `try/except: pass` wrap. If the socket is permanently broken (stale socket file, permission, hung daemon), the producer never knows; orphaned queue files sit on disk until the next kick happens to nudge successfully OR the next daemon restart
- Dual code paths for the same outcome: startup `_initial_drain` and socket-triggered `_drain_queue` — two places to forget invariants when modifying drain logic
- Tight timing coupling between REPL and daemon — re-introducing the very coupling the file queue was designed to break

Per [[feedback-queue-sole-bridge]]: when a durable file/disk queue is the cross-process bridge, side-channel signaling is anti-pattern. The fix is to make the filesystem the **sole** bridge — for work, for status, for lifecycle.

This plan is a **correction to the shipped Plan 1 implementation.** It is independent of any future plan and does not block or get blocked by Plan 2 housekeeping work.

## Scope

Refactor the daemon and the REPL producer to use the filesystem as the sole cross-process bridge:

- **Producer** (REPL `_send_review_kick`): write the queue file atomically. Nothing else.
- **Consumer** (daemon main loop): poll the queue directory at a configurable low frequency; drain whenever non-empty.
- **Status** (CLI): read PID file + scan queue directory + read state file directly. No daemon round-trip.
- **Stop** (CLI): send SIGTERM. Standard POSIX process lifecycle.

No Unix socket. No realtime event handling. No best-effort signal nudges anywhere in the system.

## Goals

1. **Delete the Unix socket IPC entirely.** Remove `co_cli/daemons/dream/_ipc.py`, all socket-binding code in `_process.py`, all socket client helpers in `co_cli/commands/dream.py`.

2. **Convert daemon main loop to pure polling.** Drain queue → sleep → repeat. Signal handlers (SIGTERM, SIGINT) for clean shutdown.

3. **Producer-side (REPL) simplification.** `co_cli/main.py:_send_review_kick` collapses to the atomic file write only. Remove the socket-nudge block and the unused `socket` import (verify no other usage).

4. **CLI `status`** replaces socket query with local file inspection: PID-file liveness check + queue-directory scan + `_dream_state.json` read.

5. **CLI `stop`** replaces socket STOP message with SIGTERM. Standard POSIX.

6. **PID file lifecycle** — daemon writes its PID at startup, removes it on clean exit. Signal handler ensures cleanup on SIGTERM.

7. **Configurable poll interval** — `dream.poll_interval_seconds` (default 5).

## Non-goals

- **No new functionality.** Steady-state behavior is unchanged.
- **No new commands or surfaces.** Just substituting filesystem/signal equivalents for socket IPC.
- **No backward compatibility.** Socket binary and code paths disappear in the same change. CLI is updated in lockstep. Per [[feedback-zero-backward-compat]].
- **No coupling with Plan 2.** Plan 2 housekeeping is unaffected by this correction except that its `co dream run` task will be implemented against the post-correction main loop (sentinel-file based) instead of a socket RUN opcode. That's a Plan 2 implementation detail, not a dependency on this plan.

## Design

### Daemon main loop — post-correction

```python
async def main_loop(deps, queue_dir, state, cfg):
    await _initial_drain(deps, queue_dir, cfg, state)
    shutdown = asyncio.Event()
    loop = asyncio.get_running_loop()
    loop.add_signal_handler(signal.SIGTERM, shutdown.set)
    loop.add_signal_handler(signal.SIGINT, shutdown.set)

    while not shutdown.is_set():
        if list_queue_files(queue_dir):
            await _drain_queue(deps, queue_dir, cfg, state)
            continue
        try:
            await asyncio.wait_for(shutdown.wait(), timeout=cfg.poll_interval_seconds)
        except asyncio.TimeoutError:
            pass

    logger.info("Dream daemon shutting down")
```

The loop has two wake conditions:
1. **Timeout** (every `poll_interval_seconds`) — triggers a queue scan when idle
2. **Shutdown event** (set by SIGTERM/SIGINT handlers) — exits the loop

**Skip-sleep-when-busy.** The `continue` after `_drain_queue` returns means the loop **does not sleep between back-to-back drains** — it rescans immediately. The `poll_interval_seconds` sleep only fires when the queue is found empty. This gives cheap idle polling at the 5s rhythm AND zero wasted sleep when the producer is bursting. Under sustained pressure, the daemon stays in catch-up mode without inter-drain idle gaps.

No socket. No IPC. No realtime push from producers.

### Producer side — `_send_review_kick`

`co_cli/main.py:_send_review_kick` collapses from ~26 lines to the atomic write only:

```python
def _send_review_kick(deps: CoDeps, *, domain: str, persisted_message_count: int) -> None:
    """Write a kick file to the dream queue.

    Fire-and-forget against the filesystem. Daemon picks up the file on its
    next polling iteration. Producer never touches the daemon's address space.
    """
    session_id = deps.session.session_path.stem
    created_at = datetime.now(UTC).isoformat()
    ts = datetime.now(UTC).strftime("%Y-%m-%dT%H%M%S.%f")
    kick_path = DREAM_QUEUE_DIR / f"{ts}-{uuid.uuid4()}.json"
    payload = {
        "domain": domain,
        "session_id": session_id,
        "persisted_message_count": persisted_message_count,
        "created_at": created_at,
    }
    atomic_write_text(kick_path, json.dumps(payload))
```

The `socket` import in `co_cli/main.py` is removed (verify no other usage in that file).

### STATUS — file-based

`co dream status` does direct local inspection:

```python
def status() -> str:
    if not DREAM_PID_FILE.exists():
        return "daemon not running"
    pid = int(DREAM_PID_FILE.read_text().strip())
    try:
        os.kill(pid, 0)
    except ProcessLookupError:
        return f"daemon not running (stale PID file: {pid})"
    queue_depth = len(list_queue_files(DREAM_QUEUE_DIR))
    state = read_dream_state()
    return format_status(pid=pid, queue_depth=queue_depth, state=state)
```

No daemon round-trip. Faster than the previous socket-based version.

### STOP — SIGTERM

`co dream stop` reads PID and signals:

```python
def stop() -> str:
    if not DREAM_PID_FILE.exists():
        return "daemon not running"
    pid = int(DREAM_PID_FILE.read_text().strip())
    try:
        os.kill(pid, signal.SIGTERM)
    except ProcessLookupError:
        DREAM_PID_FILE.unlink(missing_ok=True)
        return "daemon already stopped (stale PID file cleaned up)"
    for _ in range(20):
        time.sleep(0.5)
        if not _process_alive(pid):
            return "daemon stopped"
    os.kill(pid, signal.SIGKILL)
    return "daemon force-killed (did not respond to SIGTERM in 10s)"
```

### PID file

Path: `~/.co-cli/dream/dream.pid`

Lifecycle:
- Daemon startup: `atomic_write_text(DREAM_PID_FILE, str(os.getpid()))`
- Daemon clean exit (signal handler): `DREAM_PID_FILE.unlink(missing_ok=True)`
- Daemon crash: file becomes stale; CLI detects via `os.kill(pid, 0)` → `ProcessLookupError`

### Latency comparison

| Path | Plan 1 (socket-signal) | Post-correction (polling) |
|---|---|---|
| REPL kick → daemon drain start | Sub-millisecond | Up to `poll_interval_seconds` (default 5s; avg 2.5s) |
| CLI status → response | Daemon socket round-trip | Direct file/process inspection (instant) |
| CLI stop → daemon exit | Socket STOP message + ACK | SIGTERM + signal handler |

Reviewer LLM work takes 10+ seconds per item. The 2.5s average wake latency is invisible in user perception.

## Risks

- **Latency regression for sensitive callers.** Reviewer work is the only consumer and it's background. No realtime caller exists in the codebase today. Acceptable.

- **Signal handler edge cases.** SIGTERM arriving during `_drain_queue`: the drain checks `shutdown.is_set()` **between items**, never inside one. The in-flight item is allowed to finish (its LLM call and `done/` move complete), then the loop exits before starting the next item. Worst-case clean-shutdown latency is one reviewer call (~10s), comfortably inside `co dream stop`'s 10s SIGTERM → SIGKILL budget. Items not yet started remain in `queue/` and are picked up by the next startup `_initial_drain`. The kick is durable until acknowledged via successful move to `done/`.

- **PID file staleness on crash.** Daemon crashes without cleanup → stale PID file remains. Two layers handle this: (a) the next `co dream start` runs the singleton-on-start check, sees the dead PID via `ProcessLookupError`, and overwrites the stale file; (b) `co dream status` and `co dream stop` likewise probe with `os.kill(pid, 0)` and treat `ProcessLookupError` as "not running." Standard daemon hygiene; documented in spec.

- **Multiple daemons explicitly disallowed.** Singleton is the deliberate stance — one daemon, serial drain. Reviewer throughput in this scope is one item per ~10s. Intra-process concurrency (asyncio semaphore over reviewer calls) is the scaling axis if measured load demands more; multi-daemon is not. Decision documented here so future plans don't re-litigate it.

- **Test ergonomics improve.** Without a socket, integration tests use the filesystem boundary: write queue files directly, send SIGTERM via `os.kill`, observe the queue/done/failed directories. No IPC mocking required. Simpler.

- **Manual housekeeping triggers (future Plan 2).** Plan 2's `co dream run` previously planned a `RUN` socket opcode. Post-correction, it would use a sentinel file in the queue directory (e.g., write `~/.co-cli/dream/queue/run.tag`; daemon polls, sees the tag, runs housekeeping, deletes the tag). This is a Plan 2 implementation detail and does not block this correction.
