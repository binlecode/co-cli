# Phase 3.6 — Delegate child = full agent surface (orchestrator visibility model)

**Parent milestone:** `2026-06-24-234633-loop-decoupling-milestone.md` (Phase 3.6 — the delegation thread 2.5 → 3.5 → 3.6, owned-path-only). **Depends on:** `loop-decoupling-phase3-5` (delivered, pending `/review-impl`) — child→parent approval propagation (the `runtime.frontend` seam + `propagate_approvals`) must hold the trust boundary before the child is given a full surface. **Peer input:** `docs/reference/RESEARCH-loop-decoupling-peer-survey.md` §"Subagent engagement → Child surface, approval, depth, return".

## Context

**Principle (the design driver):** a delegated child is a **full agent**, not a lesser one. It is run as a separate task agent for **context isolation + main-workflow focus**, *not* because it is smaller or dumber. So it should be exposed to the **same tooling surface as the orchestrator**, and decide for itself which tools it needs — including loading deferred tools on demand. The approval gate (which propagates child→parent as of Phase 3.5) is the trust boundary for the child exactly as it is for the main agent.

**Two kinds of task agent (the scope split):**
- **Closed-surface specialists** — the daemon specs (`MEMORY_REVIEW_SPEC`, `SKILL_REVIEW_SPEC` in `co_cli/daemons/dream/_reviewer.py:71,87`; the user-profile spec in `_housekeeping.py:567`). Their `tool_names` is an exact, intentional list (they already name DEFERRED-tier tools directly — `user_profile_write`, `skill_patch` — proving a spec can name any tool; the subagent toolset is flat with no visibility tier). **No change to these.**
- **The general worker** — `DELEGATE_CHILD_SPEC` (`co_cli/agent/delegation.py:73`), handed an *arbitrary* task string at runtime by the orchestrator. This is the one task agent whose surface cannot be pre-curated, and the subject of this plan.

**Current state (verified this session):**
- The subagent path is **flat, no visibility tier**: `_build_subagent_toolset` (`loop.py:373-384`) builds a plain `FunctionToolset` of `spec.tool_names` (registered `requires_approval=False`, `sequential=not is_concurrent_safe`), wrapped in `_CallSeamToolset`. No `_tool_visibility_filter`.
- The orchestrator path **has the tier**: `assemble_routing_toolset` (`core.py:51-65`) = `_CallSeamToolset(CombinedToolset([native, *mcp]).filtered(_tool_visibility_filter))`. The filter (`toolset.py:72-95`) reads `deps.tool_catalog[name].visibility` + `deps.runtime.revealed_tools` and hides `DEFERRED`-not-revealed.
- `tool_view` (`co_cli/tools/system/tool_view.py:55`, ALWAYS-visibility) reveals a deferred tool by adding its canonical name to `deps.runtime.revealed_tools` (`tool_view.py:98`); it reads `deps.tool_catalog` for the deferred set and honors per-tool `check_fn` gates.
- `deferred_tool_awareness_prompt` (`_instructions.py:84`) emits the per-tool name+one-liner stubs from `deps.tool_catalog` + `revealed_tools` — how the orchestrator *knows* deferred tools exist without their schemas.
- `build_tool_defs` (`preflight.py:175`) → `get_visible_tools` (`dispatch.py:94`) reads `deps.toolset` — so giving the child a filtered toolset is exactly what makes deferral work for it.
- `fork_deps` resets `runtime` (`deps.py:456`) → the child's `revealed_tools` is **fresh per fork** (reveals isolated to that child run). `tool_catalog` is shared by reference (`deps.py:453`) → the child's filter/`tool_view` read the same visibility metadata.
- Approval is orthogonal to visibility: a revealed approval-required tool still routes through `collect_inline_approvals` → parent frontend (or auto-deny headless). Phase 3.5's `propagate_approvals=True` path is unchanged by this plan.

**The current general-child surface is a curated 14-tool allowlist** (`_CHILD_TOOL_NAMES`, `delegation.py:47-62`). It cherry-picks ALWAYS reads + two DEFERRED reads (`session_view`, `image_view`) and omits everything else — so an uncommon delegated subtask that needs a deferred specialty tool (e.g. `google_drive_read`) fails, because the child can neither see nor load it.

## Problem & Outcome

**Problem:** the general delegate child has a hand-curated, closed surface, so it fails on any subtask whose tool needs weren't pre-listed — and the alternatives (orchestrator pre-grants tools per delegation; or dump every tool flat) either couple the delegation interface to the child's internals or bloat the small-model prefill.

**Outcome:** the general child inherits the **orchestrator's visibility model** — the ALWAYS-core visible, the DEFERRED tools advertised as awareness stubs, and `tool_view` to self-load any of them on demand — minus a minimal structural blocklist. The orchestrator delegates with a **tool-agnostic** interface (task in, summary out): it does not review or pick the child's tools. The child, which knows its own decomposition, loads what it needs; approval-required loads still prompt the user via the parent frontend (Phase 3.5).

**Failure cost:** without it, delegation is only safe for the pre-listed common case; the moment the orchestrator delegates a less-common subtask (research a Drive folder, inspect a background task, read the user profile), the child silently lacks the tool and returns a degraded summary — pushing the user back to doing multi-step act-and-verify work inline, which is exactly the context pressure delegation exists to relieve. The secondary cost is architectural: the only alternatives re-introduce parent↔child tool coupling (orchestrator picks tools) or small-model prefill bloat (flat full surface).

## Scope

**In:**
- A **surface mode** on `TaskAgentSpec` selecting flat-exact (default, today's behavior) vs the orchestrator visibility model.
- A **general-child toolset builder**: reuse the **same** `assemble_routing_toolset` the orchestrator uses (parametrized with `name_blocklist`, CD-m-2) over `build_native_toolset()` **plus the connected MCP toolsets** — yielding native+MCP, visibility-filtered, blocklisted, `_CallSeamToolset`-wrapped: the orchestrator surface minus the blocklist. Result: ALWAYS-minus-blocklist visible now, DEFERRED-minus-blocklist (native **and** MCP) revealable via `tool_view`.
- **Expose the connected MCP toolsets on `CoDeps` (G1-B):** set at bootstrap from the `connected` entries (`bootstrap/core.py:405-413`), shared by reference through `fork_deps` like `tool_catalog`, so the delegate builder composes the live session-open MCP toolsets rather than re-connecting. This is the only new plumbing, and it removes the prior MCP-out divergence (Decisions G1-B).
- **Deferred-awareness stub injection** into the general child's instructions so it knows what is loadable.
- `DELEGATE_CHILD_SPEC` switched to the visibility mode; `_CHILD_TOOL_NAMES` allowlist **removed**; `delegate` tool docstring + child instructions updated.
- **Blocklist** = `{"delegate"}` (recursion / the hard depth cap; structural). This **reverses Phase 3.5's "Out: broader-trust child tools"** exclusion: `gmail`/`memory` writes/`skill` writes/`task_start`/`user_profile_write` become reachable, gated by approval propagation, identical to the main agent.
- A real-Ollama owned-path flow test: the general child reveals a deferred tool via `tool_view` and uses it; a revealed approval-required tool propagates to the parent frontend; context isolation preserved; daemon/specialist specs unchanged.

**Out:**
- **`clarify` in the child.** Kept out this round (the child instructions still say "no user-question channel"). Now technically propagatable via the `runtime.frontend` seam (`_handle_clarify` already uses `frontend.prompt_question`); raised as a trivial follow-up (Open Questions), not built here.
- **Any change to closed-surface specialist/daemon specs.** They stay flat-exact, byte-for-byte.
- **Changes to the approval-propagation machinery** (Phase 3.5 owns it). This plan only widens *what the child can reveal*; the gate is unchanged.
- **Depth cap.** Stays hard 1 (`delegate` blocklisted + `agent_depth` guard).

## Behavioral Constraints

- **The general child sees the orchestrator's surface, not a curated subset.** ALWAYS tools visible; DEFERRED tools (native **and** MCP) advertised as stubs and loadable via `tool_view`; minus `{"delegate"}`.
- **The delegation interface stays tool-agnostic.** `delegate(task)` — no tool parameter. The orchestrator does not enumerate or grant the child's tools.
- **The trust boundary is the approval gate, equally for child and main agent.** A revealed approval-required child call propagates to the parent frontend (approve→runs, deny→blocked, headless→auto-deny). No tool is withheld *as a trust measure* — `delegate` is excluded only for the recursion/depth invariant.
- **Context isolation preserved.** Only the `delegate` summary enters parent history; the child's intermediate tool calls/results (including `tool_view`) do not.
- **Reveals are child-local.** `fork_deps` gives the child a fresh `revealed_tools`; the child's reveals never leak to the parent or across delegations.
- **The approval subject — not the child's reasoning — is the decision surface for delegated calls (PO-m-2).** A child's reasoning is silent on the parent terminal, so the user judges a delegated approval-required call (e.g. a `gmail_draft`, a `memory_delete`) from the prompt's **subject** (the concrete file path / recipient / value) plus the `[delegated subtask]` provenance marker, not from the child's chain of thought. This is a deliberate, documented standing trade (mirrors Phase 3.5's PO-m-3): the gate stays a thin-but-sufficient decision surface. Dev must confirm the propagated prompt surfaces concrete subject detail for the broader-trust tools now reachable; if a tool's subject is too thin to decide on, that is a prompt-subject fix, not a reason to re-restrict the surface. (G1-A: TASK-3 now asserts this on a real broader-trust write, not in prose alone.)
- **Closed-surface specialists unchanged.** Daemon specs keep flat-exact `tool_names`; the daemon regression set stays green.
- **Writes stay serialized.** The reused native toolset carries `sequential=not is_concurrent_safe` (the Phase-3.5 CD-M-1 contract) for `file_write`/`file_patch`.

## High-Level Design

```
delegate(task)                         # tool-agnostic interface — no tool params
  └─ delegate_to_child(parent_deps, task)
       frontend = parent_deps.runtime.frontend        # Phase 3.5 seam
       child_deps = fork_deps(...)                     # fresh runtime → fresh revealed_tools
       run_standalone_owned(DELEGATE_CHILD_SPEC, child_deps, task,
                            propagate_approvals=True, frontend=frontend)
         deps.toolset = <general-child toolset>        # ← NEW: visibility model, not flat
           = assemble_routing_toolset(native, mcp_toolsets, name_blocklist=BLOCKLIST)
             # native+MCP, _tool_visibility_filter, blocklist, _CallSeamToolset
             # = the orchestrator surface minus blocklist (G1-B)
         instructions = spec.instructions(deps)   # ← _delegate_child_instructions: includes deferred stubs, recomputed per step
         ── child step loop ──
           visible tools = ALWAYS − blocklist  (+ any already in child.revealed_tools)
           child may call tool_view("google_drive_read")   # self-load a deferred tool
             → child_deps.runtime.revealed_tools.add(...)   # child-local
           next step: google_drive_read now visible; child calls it
             → collect_inline_approvals (propagate) → parent frontend if gated
         return summary                                 # only this enters parent history
```

**Why reuse `assemble_routing_toolset` over native + connected-MCP toolsets (G1-B).** It already composes native+MCP, applies `_tool_visibility_filter`, and wraps in `_CallSeamToolset` (`core.py:51-65`); `build_native_toolset()` registers every native tool with the correct `requires_approval` and `sequential=not is_concurrent_safe` (`toolset.py:124-138`). The delegate task wants exactly the orchestrator surface minus the blocklist — so the builder is the same `assemble_routing_toolset(native, mcp_toolsets, name_blocklist=BLOCKLIST)` call the orchestrator makes (CD-m-2's single composition point), with one extra arg. No duplicated registration, no second filter, `sequential` correct for free, and **MCP parity for free** since the connected MCP toolsets are now exposed on `deps`.

**Why a spec mode, not a per-call flag.** The flat-exact vs visibility-model choice is a property of the *agent kind* (closed specialist vs open worker), fixed at spec-definition time — not per invocation. A `surface_mode` field on `TaskAgentSpec` (default flat-exact) keeps daemon specs byte-for-byte and routes only `DELEGATE_CHILD_SPEC` to the new builder. **Watch (G1-E):** `surface_mode` and 3.5's `propagate_approvals` call-arg are correlated by construction today (delegate = visibility+propagate; daemons = flat+no-propagate). They sit on different axes (spec definition vs call-context), so they are not collapsed now — but if a *third* task-agent kind ever appears, collapse them into one `kind` discriminator rather than adding a third flag.

**Why awareness stubs in the child instructions.** Without them the child sees only the ALWAYS core and has no idea deferred tools exist (it would never call `tool_view`). Injecting `deferred_tool_awareness_prompt`-equivalent content (the same name+one-liner stubs, minus blocklist) is what makes the child's self-loading possible — the lazy-loaded form of "the child has a broad surface" (codex/claude-code), fit to co's small-model prefill budget.

**Why the blocklist is just `{"delegate"}`.** Per the principle, nothing is withheld as a trust measure (approval propagation is the trust boundary). The only structural exclusion is recursion: `delegate` stays out so the hard depth cap holds at the surface level too (belt-and-suspenders with the `agent_depth` guard). `clarify` is deferred to Open Questions (out this round). **Why narrower than hermes/openclaw (PO-m-1):** hermes and openclaw *do* blocklist durable-outward writes (`memory`/`send_message`/`sessions_send`/`cronjob`) from their children — but they have **no approval propagation**, so for them withholding from the surface is the *only* place to enforce safety. co has child→parent approval propagation (Phase 3.5) that they lack, so co recovers that durable-write safety **at the gate** (every such call prompts the user) rather than **at the surface** — withholding a tool gives strictly *less* safety than gating it (the user gets the prompt either way) at the cost of silently failing uncommon subtasks. So co's narrower blocklist is a consequence of having a stronger gate, not a weaker stance.

**Why MCP is in the delegate surface (G1-B).** All three delegation peers build the child by **inheriting the parent surface** (codex = full parent surface; hermes/openclaw = inherit − blocklist) and **none carve out integration/MCP tools** — their blocklists are durable-outward writes and infra/session tools, never integrations (`RESEARCH-loop-decoupling-peer-survey.md:96,105`). Excluding MCP would diverge from all three peers on the very axis this plan converges (surface construction), for a reason that was purely co-internal plumbing (the connected MCP toolsets weren't reachable from `deps`). Exposing them on `deps` and routing through `assemble_routing_toolset` makes the delegate surface the true orchestrator surface minus blocklist — MCP tools advertised as deferred stubs, `tool_view`-loadable, approval-gated identically — and dissolves the otherwise-latent incoherence of stubs advertising an MCP tool the native-only toolset could not load.
