# Plan: Canon FTS Merge — Move Canon Recall into Unified FTS Pipeline

Task type: refactor

---

## Context

The memory system has three recall channels: sessions, knowledge artifacts, and canon. Sessions and knowledge share the unified FTS/hybrid pipeline (`MemoryStore` + `co-cli-search.db`, `source='session'` / `source='knowledge'`). Canon is isolated: it uses an in-process file scan with custom token-overlap scoring (`canon_recall.py`, 83 lines) and never touches the DB.

The isolation was motivated by corpus size (small scenes, <1KB average), but it produces a bespoke retrieval stack — custom tokenizer, stopword filter, title 2× weight heuristic, `_MIN_SCORE=2` noise floor — that compensates for what FTS5 handles natively. There are no tests exercising canon in `tests/`; coverage lives only in `evals/eval_canon_recall.py`.

**Current-state validation:** Doc and source are consistent. `canon_recall.py` exists as described. `memory_store.py:sync_dir()` accepts arbitrary `source` labels and already handles frontmatter parsing, hash-skip, and stale removal — no structural change needed. The `eval_canon_recall.py` sub-cases for score-floor (`>= 2`) and rank-quality (`top >= 2× hit-2`) test token-overlap algorithm properties that have no meaning under BM25; these sub-cases must be replaced as part of the eval update.

No stale exec-plans found for this slug.

## Problem & Outcome

**Problem:** Canon uses a separate, bespoke retrieval path (token-overlap scoring in `canon_recall.py`) that bypasses FTS5, requires no DB, and provides no semantic/hybrid recall.

**Failure cost:** Canon files get worse recall quality than knowledge artifacts — token-overlap is weaker than BM25 FTS5, and semantic search (hybrid mode) is completely unavailable for canon scenes. Adding a new canon file to a soul requires no indexing step, which silently produces a two-tier system where the same agent has its memories recalled by different algorithms with no visible signal of which is active.

**Outcome:** Canon indexed into `co-cli-search.db` under `source='canon'` at bootstrap (one chunk per file, no splitting). Canon recall uses `MemoryStore.search(sources=['canon'])` like every other channel. `canon_recall.py` is deleted. The agent's result contract is unchanged: `memory_search` still returns `{channel: 'canon', role, title, body, score}` hits with full body inline.

## Scope

**In:**
- `MemoryStore.sync_dir()`: add `no_chunk` flag — when True, insert full body as a single `Chunk(index=0)` (no `chunk_text()` call)
- `MemoryStore.get_chunk_content()`: new method — `SELECT content FROM chunks WHERE source=? AND doc_path=? AND chunk_index=?`
- Bootstrap `create_deps()`: add `_sync_canon_store()` step after `_sync_memory_store()`; no-ops when `store is None` or `personality` is empty
- `_search_canon_channel()` in `recall.py`: replace `search_canon()` with `store.search(sources=['canon'])` + full-body fetch via `get_chunk_content()`
- Delete `canon_recall.py`; clean up all imports
- Update `eval_canon_recall.py`: remove direct `search_canon` usage; replace score-floor and rank-quality sub-cases with FTS-appropriate assertions

**Out:**
- No change to `memory_search` tool signature, result contract, or display format
- No change to canon file location (`souls/{role}/memories/*.md`) or ownership (system-managed, never written by agent tools)
- No change to `character_recall_limit` config key
- No change to sessions or knowledge channels
- No spec doc updates (handled by post-delivery `sync-doc`)

## Behavioral Constraints

1. `memory_search` result shape for canon hits must remain `{channel: 'canon', role, title, body, score}` with full body in `body` — not a snippet.
2. Canon body must come from `chunks.content` (the stored full text), not from the FTS5 `snippet()` function (which truncates).
3. When `memory_store is None` (grep backend), `_search_canon_channel` must return `[]` — same as sessions channel under the same condition.
4. When `config.personality` is empty/None, canon indexing and recall must both no-op and return `[]`.
5. `sync_dir(..., no_chunk=True)` must store exactly one `chunks` row per file regardless of file length. No size guard is applied — the full post-frontmatter body is stored as-is; canon files are expected to remain small (<1KB per scene).
6. Hash-skip must work for canon: re-running `_sync_canon_store()` on unchanged files must produce zero DB writes.
7. Canon docs deleted from the filesystem must be evicted from the index on the next bootstrap (handled by existing `remove_stale()` in `sync_dir()`).

## High-Level Design

### Indexing (new, bootstrap-time)

```
create_deps()
  → _sync_memory_store(store, config, frontend, knowledge_dir)   # unchanged
  → _sync_canon_store(store, config, frontend)                   # NEW: Step 7b
       if store is None or not config.personality: return
       canon_dir = souls_dir / role / "memories"
       store.sync_dir("canon", canon_dir, glob="*.md", no_chunk=True)
```

`sync_dir("canon", canon_dir, no_chunk=True)` path through `MemoryStore`:
- Glob `canon_dir/*.md` (non-recursive; memories are flat)
- Parse frontmatter for title
- SHA256 hash-skip: skip file if DB hash matches
- Insert `docs` row with `source='canon'`
- Insert single `chunks` row: `Chunk(index=0, content=body_text, start_line=0, end_line=N)`
- `remove_stale("canon", current_paths, directory=canon_dir)` — evict deleted files

### Recall (rewritten)

```
_search_canon_channel(ctx, query):
    role = ctx.deps.config.personality
    store = ctx.deps.memory_store
    if not role or store is None: return []
    limit = ctx.deps.config.knowledge.character_recall_limit
    results = store.search(query, sources=["canon"], limit=limit)
    hits = []
    for r in results:
        body = store.get_chunk_content("canon", r.path, 0) or ""
        hits.append({
            "channel": "canon", "role": role,
            "title": r.title or Path(r.path).stem,
            "body": body, "score": r.score,
        })
    return hits
```

`get_chunk_content("canon", doc_path, 0)` — a direct `SELECT content FROM chunks WHERE source=? AND doc_path=? AND chunk_index=?` lookup. Canon docs are always chunk_index=0, so one query per hit.

### Degradation

| Condition | Behavior |
|---|---|
| `store is None` (grep backend) | canon unavailable; `_search_canon_channel` returns `[]` |
| `personality` empty/None | canon indexing skipped; recall returns `[]` |
| canon dir missing | `sync_dir()` returns 0 (directory.exists() check) |
| FTS5 no match | BM25 returns `[]` — same as "no results" in any channel |
