## Problem & Outcome

**Problem.** The Phase 1 queue is a black box: the user sees a depth count but cannot see *what* is
queued, cannot remove a mis-typed item, and cannot cancel a queue they no longer want. Every queued
item is locked in until it drains as a turn — a fat-fingered or now-irrelevant submission still costs
a full turn.

**Outcome.** A `/queue` command surface (list / clear / pop) lets the user inspect and prune pending
items mid-turn, and the toolbar shows a preview of the head item, not just a count. A mis-typed or
now-irrelevant queued item can be dropped before it ever costs a turn.

**Failure cost.** Without this, the queue stays unmanageable: a wrong mid-turn submission can only be
flushed by letting it run (wasting a turn and polluting history) or killing the session. As turns get
longer (reasoning models, deeper tool chains) the cost of an un-cancellable queued mistake grows.

## Scope

**In (Phase 2):**
- `/queue` command (registered builtin, visible in `/help` + completer):
  - `/queue` (no args) → list pending items with 1-based indices + truncated preview.
  - `/queue clear` → flush all pending items.
  - `/queue pop [n]` → drop item at index `n` (default: last). 1-based.
- **Mid-turn bypass** for `/queue`: it executes immediately against the live queue even while a turn
  is active (a controlled, documented exception to Phase 1 C5 — see C1 below), because it manages the
  buffer itself and never touches `run_turn`/`deps.session`/`message_history`.
- Queue access threaded to commands via a narrow `input_queue: deque[str] | None` field on
  `CommandContext` (stdlib type, passed by reference — no display-package coupling).
- **Queued-item preview** in the toolbar: head item shown truncated (e.g. `1 queued: "fix the …"`),
  replacing the bare count when depth ≥ 1.

**Out (deferred):**
- **Collect mode** (config-gated merge drain) and **edit-before-send** (`/queue edit` → pull a queued
  item back into the input line) → **Phase 3** (`repl-queue-policy`). Both were in the original
  Phase-2 charter but cut at Gate-1 critique: collect mode is a drain-*policy* variant that belongs
  with Phase 3's cap/drop/mode work, and edit-before-send is the only feature forcing a
  `build_repl_app` signature change + a `prompt_toolkit` buffer reference onto `_ReplRuntime` —
  surface disproportionate to its ROI when `/queue pop` + retype already covers the case. See the
  Phase 3 plan (`repl-queue-policy`).
- Queue cap + drop policy; hermes `busy_input_mode` {interrupt/queue/steer}; background-event
  auto-enqueue → **Phase 3**.
- Reordering queued items; per-item metadata beyond preview text → not planned.
- File-backed queue persistence → rejected (Phase 1 C8: in-memory session-lifetime by design).

## Behavioral Constraints

C1. **`/queue` is the one controlled bypass to Phase 1 C5.** All other mid-turn input still enqueues.
`/queue` runs immediately mid-turn because it is a buffer operation, not an agent turn: it never calls
`run_turn`, never mutates `deps.session`/`message_history`/`deps.runtime`, so it carries none of the
data-race risk C5 guards against. Mid-turn, the `accept_handler` recognizes the `/queue` prefix and
runs the queue-control logic inline (via `runtime.schedule_control(...)`, **not** `_arm_turn` — it is
not a turn and must not carry the `_drain_next` callback). Idle, `/queue` flows through the normal
`_handle_one_input` → `dispatch` path like any builtin.

C2. **Depth authority unchanged (Phase 1 C7).** The deque on `_ReplRuntime` stays the single source of
truth. `/queue clear`/`pop` mutate it by reference; every mutation triggers exactly one toolbar
repaint. No depth caching, no second queue copy.

C3. **Preview is derived, never stored.** The toolbar preview reads `runtime.queue[0]` at render time
and truncates for display. No preview string is cached on the snapshot beyond what render needs. Empty
queue → no segment (Phase 1 behavior preserved).

C4. **Index discipline.** `/queue` listing and `pop` use **1-based** indices (user-facing).
Out-of-range or non-integer args produce a usage error, never an exception or silent no-op on the
wrong item.

C5. **`CommandContext` stays REPL-agnostic.** The new `input_queue` field is `deque[str] | None`
(stdlib only), `None` outside REPL context (headless, tests that don't exercise the queue) — mirroring
the existing `completer`/`frontend` optionality. No `_ReplRuntime` import into `co_cli/commands/`.

C6. **The queue-control core renders via the module `console`, not a `Frontend`.** Existing
list-style commands print through `co_cli.display.core.console` (`help.py`, `tasks.py`), and
`_build_accept_handler(runtime, dispatch, on_status)` closes over neither `frontend` nor `deps`. So the
core signature is `(queue: deque[str], args: str) -> None` — it operates on the deque and prints via
`console`. Depth/preview repaint is triggered separately: the mid-turn bypass calls the existing
`on_status` after the core runs; the idle path repaints via the normal turn-completion status push.

## High-Level Design

```
 ┌──────────── _chat_loop (session scope) ───────────────────────────┐
 │  runtime = _ReplRuntime(queue=deque())                            │
 │  ctx_input_queue = runtime.queue   # same object, by reference    │
 │                                                                    │
 │  accept_handler(buffer):                                          │
 │    text = buffer.text                                             │
 │    if _is_queue_command(text):        # NEW — C1 bypass            │
 │        runtime.schedule_control(_run_queue_control(text, runtime)) │
 │        return False                                               │
 │    if runtime.turn_active:                                        │
 │        if text.strip(): queue.append(text); on_status()           │
 │    else:                                                          │
 │        _arm_turn(dispatch(user_input=text))                       │
 │    return False                                                   │
 │                                                                    │
 │  _drain_next(task):                   # unchanged from Phase 1     │
 │    if should_exit or empty: return                                │
 │    nxt = queue.popleft()                                          │
 │    on_status(); _arm_turn(dispatch(user_input=nxt))               │
 └────────────────────────────────────────────────────────────────────┘

 /queue (idle path via dispatch; mid-turn path via bypass) shares
 one queue-control core operating on the same deque:
   list | clear | pop [n]
```

**Bypass predicate (C1, C6).** `_is_queue_command(text)` mirrors the real `dispatch` parse exactly
(`co_cli/commands/core.py:95-96`): strip, take `text[1:].split(maxsplit=1)`, lowercase the command
token, test `== "queue"`. The *same* remaining-args slice feeds the *same* core both paths use, so
idle `/queue  CLEAR` and mid-turn `/queue clear` cannot diverge. The mid-turn bypass returns `False`
from `accept_handler` (non-blocking, Phase 1 invariant) and runs the core on a `schedule_control`
task — never `_arm_turn` (no `_drain_next` callback; it is not a turn).

**Where the queue-control logic lives.** A single private helper `co_cli/commands/_queue_control.py`
implements `list`/`clear`/`pop [n]` against a `deque[str]`, printing via the module-level
`console` (C6 — not a `Frontend`; matches `help.py`/`tasks.py`). Both entry points call it:
- **Idle**: `BUILTIN_COMMANDS["queue"]` → `_cmd_queue(ctx, args)` calls the core on `ctx.input_queue`,
  returns `None` (→ `LocalOnly`; never a history list, or it would arm a turn).
- **Mid-turn**: `accept_handler` detects `/queue`, schedules a control task that calls the same core on
  `runtime.queue`, then `on_status()` to repaint depth/preview.
Factoring into one core avoids the dual-path divergence the Phase 1 review warned about.

**Why thread the deque, not a new InputQueue class.** Phase 1 chose `deque[str]` deliberately; the
list/clear/pop operations are all expressible on a deque (pop-by-index via list-copy is fine at queue
sizes humans produce). A wrapper class would be speculative surface for three operations. If Phase 3
needs per-item metadata it can promote to `deque[QueueItem]` then (cheap, in-memory, no migration —
Phase 1 C8).
