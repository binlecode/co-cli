# agent-loop-cap

## Context

Two cap axes were identified in the `agent-loop-cap-roadmap` briefing (now superseded) as
not yet implemented:

1. **Iteration cap (cost axis):** bound the total number of `ModelResponse`s per user turn.
   Currently unlimited — a runaway tool loop can exhaust Ollama GPU time or Gemini quota
   without a circuit breaker.

2. **Tool-call cap hard-stop (safety axis):** the per-llm_iteration cap (`MAX_TOOL_CALLS_PER_MODEL_TURN = 6`)
   already rejects excess calls and errors back to the LLM (`co_cli/tools/tool_call_limit.py`,
   `CoToolLifecycle.wrap_tool_execute`). What's missing is the consecutive-violation hard-stop:
   if the LLM keeps exceeding the cap despite error feedback, the turn must be killed, not
   looped indefinitely.

### Current state — verified against source

- `co_cli/tools/tool_call_limit.py`: `MAX_TOOL_CALLS_PER_MODEL_TURN = 6`. Named constant —
  confirmed, not a magic number.
- `co_cli/tools/lifecycle.py:CoToolLifecycle.wrap_tool_execute` (line 181): enforces cap
  per `ctx.run_step`; rejects excess calls with `make_exceeded_payload()` JSON error
  (error-back-to-LLM ✓). No consecutive-violation tracking.
- `co_cli/tools/lifecycle.py:CoToolLifecycle.after_node_run` (line 202): emits
  `tool_budget.enforce_tool_call_limit` span event. No side-effect on turn flow.
- `co_cli/deps.py:CoRuntimeState`: has `tool_call_limit_run_step: int = -1` and
  `tool_calls_in_model_turn: int = 0` for per-run-step brake; neither is reset by
  `reset_for_turn()` (implicit reset via `ctx.run_step` transition).
- `co_cli/context/orchestrate.py:_TurnState.llm_iterations` (line 160): counts all
  `ModelResponse`s per turn; no cap enforced on it.
- `co_cli/config/llm.py`: no `max_iterations_per_turn` field.
- `co_cli/config/core.py`: `DEFAULT_DOOM_LOOP_THRESHOLD = 3` — precedent for named default
  constants driving turn-level hard-stops.
- `tests/test_flow_tool_call_limit.py`: covers cap enforcement and span event. Stale
  assertions in this file must be audited when new fields are added.

### Design decisions (closed before planning)

| Decision | Answer |
|---|---|
| Iteration cap default | 90 (global, provider-agnostic; matches hermes-agent) |
| Per-provider defaults | Removed — over-design; no evidence different providers need different caps |
| 0 = disabled | `max_iterations_per_turn=0` disables the cap; no model_validator needed |
| Tool-call cap exceeded behavior | Error back to LLM (current); no partial batch |
| Hard-stop threshold | 3 consecutive violating llm_iterations |
| No magic numbers | All values as named module-level constants |

## Problem & Outcome

**Problem.** Without an iteration cap, a runaway tool loop can exhaust GPU time on Ollama
or quota on Gemini silently. Without a hard-stop, the LLM can ignore tool-call cap errors
indefinitely — the existing `wrap_tool_execute` rejection is advisory, not terminal.

**Outcome.** Every user turn has a hard ceiling on total LLM iterations (provider-specific
named default, user-overridable) and on consecutive tool-cap violations (named constant,
configurable). Both caps terminate the turn cleanly with a user-visible status message.

**Failure cost.** Without the iteration cap: Ollama sessions can lock the GPU for minutes
on loops; Gemini sessions can exhaust daily quota. Without the hard-stop: a model that
persistently issues too many tool calls per turn loops until the context overflows or the
user interrupts, with no actionable error surface.

## Scope

### In scope

1. **Named default constant** for iteration cap:
   `DEFAULT_MAX_ITERATIONS_PER_TURN = 90` in `co_cli/config/llm.py`.

2. **`LlmSettings.max_iterations_per_turn`** field: `int`, global default 90,
   `0` disables the cap; env var `CO_LLM_MAX_ITERATIONS_PER_TURN`; added to
   `LLM_ENV_MAP`. No `model_validator` needed — no provider-specific logic.

3. **Iteration cap enforcement** in `co_cli/context/orchestrate.py:run_turn`:
   after `_run_approval_loop` returns, check `turn_state.llm_iterations` against
   `deps.config.llm.max_iterations_per_turn`; emit status and return error turn result if
   exceeded.

4. **`TOOL_CAP_HARD_STOP_CONSECUTIVE = 3`** named constant in `co_cli/tools/tool_call_limit.py`.

5. **`consecutive_tool_cap_violations: int = 0`** field on `CoRuntimeState`
   (`co_cli/deps.py`); added to `reset_for_turn()`.

6. **Consecutive-violation tracking** in `CoToolLifecycle.after_node_run`
   (`co_cli/tools/lifecycle.py`): after a `CallToolsNode`, if
   `tool_calls_in_model_turn > MAX_TOOL_CALLS_PER_MODEL_TURN` → increment
   `consecutive_tool_cap_violations`; else → reset to 0.

7. **Hard-stop check** inside `_run_approval_loop` (`co_cli/context/orchestrate.py`):
   after each `_execute_stream_segment`, check
   `deps.runtime.consecutive_tool_cap_violations >= TOOL_CAP_HARD_STOP_CONSECUTIVE`;
   if so, set `turn_state.tool_cap_hard_stop = True` and break. Then in `run_turn` after
   `_run_approval_loop` returns, check the flag and return error turn result with status.

8. **`tool_cap_hard_stop: bool = False`** field on `_TurnState`
   (`co_cli/context/orchestrate.py`): set by `_run_approval_loop` when threshold crossed;
   read by `run_turn` to drive exit.

9. **Tests** covering both new cap behaviors (see Testing section).

### Out of scope

- Changing `MAX_TOOL_CALLS_PER_MODEL_TURN = 6` value (already sized for 64K-floor tail).
- Per-provider tool-call cap (same limit for all providers).
- Iteration cap for daemon/task agents (only session-agent turn loop in scope).
- Config UI / `/config` display changes.
- Spec writes (`/sync-doc` runs post-delivery).

## Behavioral Constraints

- **Iteration cap is hard.** When `turn_state.llm_iterations >= max_iterations_per_turn`,
  the turn must terminate immediately via `_build_error_turn_result` — no additional
  segments may run.
- **Hard-stop is the turn terminator, not the per-call cap.** `MAX_TOOL_CALLS_PER_MODEL_TURN`
  errors back to the LLM for remediation. Only after `TOOL_CAP_HARD_STOP_CONSECUTIVE`
  consecutive violating steps is the turn killed.
- **No partial batches on hard-stop.** When the hard-stop fires, the turn returns with
  the messages accumulated so far; no partial tool-call results are injected.
- **Consecutive counter is per-turn.** `consecutive_tool_cap_violations` must be 0 at
  the start of every turn (`reset_for_turn()` clears it). A violation from a previous
  turn must not contribute to the next turn's count.
- **Non-violating llm_iteration resets the consecutive counter.** If the LLM respects the
  cap on a retry, `consecutive_tool_cap_violations` resets to 0 — only unbroken
  consecutive violations trigger the hard-stop.
- **Status messages are user-visible.** Both caps emit a `frontend.on_status(...)` message
  before returning the error turn result so the user knows why the turn ended.
- **All numeric thresholds are named constants or config fields.** No literal integers
  for cap values, defaults, or thresholds in production code paths.
- **`CO_LLM_MAX_ITERATIONS_PER_TURN` must be respected by `LlmSettings` at load time.**
  Setting this env var must override the provider-specific default.

## High-Level Design

### Iteration cap

`LlmSettings` gains a new field:

```python
# co_cli/config/llm.py
DEFAULT_MAX_ITERATIONS_PER_TURN: int = 90

class LlmSettings(BaseModel):
    ...
    # 0 = disabled.
    max_iterations_per_turn: int = Field(default=DEFAULT_MAX_ITERATIONS_PER_TURN, ge=0)
```

`LLM_ENV_MAP` gains `"max_iterations_per_turn": "CO_LLM_MAX_ITERATIONS_PER_TURN"`.
Setting `CO_LLM_MAX_ITERATIONS_PER_TURN=0` disables the cap. No `model_validator` needed.

Enforcement in `run_turn` — placed **before** the `_length_retry_settings` continue so a
`finish_reason="length"` turn cannot bypass the check by looping further:

```python
# Immediately after _run_approval_loop, before boosted_settings evaluation:
cap = deps.config.llm.max_iterations_per_turn
if cap > 0 and turn_state.llm_iterations >= cap:
    frontend.on_status(
        f"Iteration cap reached ({cap} LLM calls this turn) — stopping."
    )
    turn_state.outcome = "error"
    return _build_error_turn_result(turn_state)
```

### Tool-call cap hard-stop

New constant:

```python
# co_cli/tools/tool_call_limit.py
TOOL_CAP_HARD_STOP_CONSECUTIVE: int = 3
```

New runtime field and reset:

```python
# co_cli/deps.py:CoRuntimeState
consecutive_tool_cap_violations: int = 0

def reset_for_turn(self) -> None:
    ...
    self.consecutive_tool_cap_violations = 0
```

Tracking in `CoToolLifecycle.after_node_run`:

```python
if isinstance(node, CallToolsNode):
    issued = ctx.deps.runtime.tool_calls_in_model_turn
    if issued > MAX_TOOL_CALLS_PER_MODEL_TURN:
        ctx.deps.runtime.consecutive_tool_cap_violations += 1
    else:
        ctx.deps.runtime.consecutive_tool_cap_violations = 0
    # ... existing span event emission unchanged
```

New flag on `_TurnState`:

```python
# co_cli/context/orchestrate.py
@dataclass
class _TurnState:
    ...
    tool_cap_hard_stop: bool = False
```

Hard-stop check inside `_run_approval_loop`, after each `_execute_stream_segment`:

```python
if (
    deps.runtime.consecutive_tool_cap_violations
    >= TOOL_CAP_HARD_STOP_CONSECUTIVE
):
    turn_state.tool_cap_hard_stop = True
    break
```

Check in `run_turn` after `_run_approval_loop`:

```python
await _run_approval_loop(turn_state, agent, deps, active_settings, frontend)
if turn_state.tool_cap_hard_stop:
    frontend.on_status(
        f"Tool-call cap exceeded {TOOL_CAP_HARD_STOP_CONSECUTIVE} consecutive "
        "llm_iterations — stopping."
    )
    turn_state.outcome = "error"
    return _build_error_turn_result(turn_state)
```
