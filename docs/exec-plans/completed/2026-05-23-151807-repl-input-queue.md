# repl-input-queue

## Context

This is a **full rewrite** of the original draft. That draft (codex-style producer/consumer
split, `asyncio.TaskGroup`, sync-prompt cancel/restart sandwich, `KeyBindings`-based SIGINT
routing) was written against the **pre-refactor** terminal stack and is obsolete. The
**terminal-single-owner refactor** (`docs/exec-plans/completed/2026-05-26-214739-terminal-single-owner-refactor.md`,
shipped v0.8.258) landed as the explicit **Phase 0** prerequisite for this plan, and its
"Downstream impact" subsection (a‚Äìd) declared this plan blocked-for-rewrite. This document is
that rewrite.

**What Phase 0 changed, and why it shrinks this plan:**
- The REPL is now driven by a single persistent `prompt_toolkit.Application(full_screen=False)`
  under `patch_stdout()` (`co_cli/main.py:559‚Äì560`). Input is **event-driven** via the input
  `TextArea`'s `accept_handler` ‚Äî there is no `prompt_async()` read loop. So there is **no
  producer coroutine** and **no `TaskGroup`** to build; "type while busy" needs only the
  `accept_handler` to enqueue instead of drop.
- `prompt_approval` / `prompt_question` / `prompt_confirm` are now `async def` and run their
  blocking read via `run_in_terminal(...)` (`co_cli/display/core.py:514/536/555`). Stdin
  arbitration during approvals is `run_in_terminal`'s job ‚Äî the original C4 cancel/restart
  sandwich (TASK-3) is **deleted, not rebuilt**.
- The SIGINT handler swap is gone; Ctrl+C is handled through `KeyBindings` in
  `co_cli/display/_app.py:build_key_bindings` (`:79‚Äì100`). The original C6 SIGINT rationale is moot.
- `patch_stdout` is live; there are zero Rich `Live` instances. The original line-42 premise
  ("Rich `Console` captures `sys.stdout` at import, so `patch_stdout` can't proxy") was false and
  is dropped. The always-live-during-turn prompt the old plan deferred to Phase 2 already exists.

## Problem & Outcome

**Problem.** While a turn runs, mid-turn submissions are **dropped** (`main.py:452`, BC6 from
Phase 0). Type-ahead is lost on every long turn ‚Äî the user types into a void and re-types after
the turn lands. Pure UX regression vs. every peer agent CLI.

**Outcome.** Submissions during an active turn **enqueue** (FIFO) instead of dropping; idle
submissions run immediately. The queue drains **one item per turn boundary**. Queue depth shows
in the bottom toolbar. `Esc` interrupts the running turn and advances to the next queued item;
`Ctrl+C` exits (double-press). Matches the Claude Code / opencode interaction model.

**Failure cost.** Without this, type-ahead stays silently dropped on every long-running turn ‚Äî
and the gap widens as reasoning models and deeper tool chains make turns longer.

## Scope

**In (Phase 1):**
- Flip the `accept_handler` drop-seam (`main.py:452`) to **enqueue** mid-turn submissions.
- An in-memory FIFO (`collections.deque[str]`) owned on `_ReplRuntime`, drained one item per turn
  boundary (normal completion **and** `Esc`-cancel).
- `queue_depth: int` on `StatusSnapshot`, rendered as `"{n} queued"` (omitted at 0) between `mode`
  and `ctx`.
- Key remap: `Esc` = interrupt active turn + advance queue; `Ctrl+C` = exit (double-press);
  `Ctrl+D` = EOF (unchanged).
- Slash commands enqueue with everything else (no bypass surface) ‚Äî they already flow through
  `_handle_one_input`.

**Out (deferred ‚Äî see "Deferred Phases"):**
- `/queue` inspect / `/queue clear` / `/queue pop`; queued-item preview; collect (merge) mode;
  edit-before-send ‚Üí **Phase 2**.
- Queue cap + drop policy; hermes-style `busy_input_mode` (`interrupt`/`queue`/`steer`);
  background-event auto-enqueue ‚Üí **Phase 3** (`steer` needs a `run_turn()` mid-turn injection
  primitive co lacks).
- File-backed queue persistence; alt-screen / full-screen TUI ‚Üí **rejected** (not a phase).

## Behavioral Constraints

C1. **`run_turn()` unchanged.** Single-turn, event-emitting, one `user_input: str`. No queue logic
in the agent loop. The queue lives entirely in the REPL layer. (Source:
`co_cli/context/orchestrate.py:run_turn`.)

C2. **No producer; the queue is the back-pressure layer.** Input is event-driven ‚Äî the owned
`Application` is always reading the input `TextArea`, even mid-turn. There is no producer
coroutine and no `TaskGroup`. The `accept_handler` callback only *enqueues or schedules* (never
blocks the app input loop, per co's same-loop concurrency discipline).

C3. **FIFO, one item per turn boundary.** Each drain pops exactly one item via `deque.popleft()`
and runs one `_handle_one_input(...)`. No batching, no merging. Blank/whitespace submissions are
dropped at enqueue (never occupy a slot) ‚Äî matches the existing no-op at `main.py:367`.

C4. **Cancel is a turn boundary.** `Esc` cancels the active `turn_task`; the same drain hook that
fires on normal completion fires on cancel and advances to the next queued item. The interrupted
query is **abandoned** ‚Äî it is not re-run, and only the *remaining* queued items advance.
   - **What survives the cancel:** the `CancelledError` arm in `_dispatch` (`main.py:531‚Äì533`) calls
     `frontend.cleanup()` ‚Üí `_clear_inflight()`, which discards the **in-flight (uncommitted)**
     streaming buffer. Only output already **committed** to scrollback (via `print_formatted_text`)
     remains visible; the live partial at the moment of cancel is cleared, not preserved.

C5. **All input enqueues ‚Äî including slash commands.** No bypass surface in Phase 1. A slash
command typed mid-turn enqueues and runs when drained. Eliminates data races against
`deps.runtime` / `deps.session.*` that the active turn is mutating.
   - **Boundary (not a violation):** while a mid-turn approval / question prompt is up, the
     blocking read runs under `run_in_terminal(...)` (`co_cli/display/core.py:514/536/555`), which
     **suspends the owned Application** ‚Äî so the `accept_handler` does not fire and keystrokes are
     consumed by the approval read, not the queue. This is correct: the prompt needs that input.
     "All input enqueues" holds whenever the app's input loop is live (i.e. not inside a
     `run_in_terminal` sub-prompt).

C6. **Key mapping (via `KeyBindings`):**
   - `Esc`: if a turn is active, cancel `turn_task` (drain hook advances the queue). No exit
     arming. Idle: no-op. Note ‚Äî binding `@kb.add("escape")` *overrides* prompt_toolkit's default
     escape handling; the idle branch does not inherit a free line-clear. If a visible idle
     behavior is wanted, own it explicitly (e.g. clear `buffer.text`); otherwise the idle no-op is
     acceptable. Do not assume the default fires.
   - `Ctrl+C`: route to `_handle_one_input(user_input=None)` ‚Äî press once arms the 2 s window and
     prints the hint; again within 2 s exits. `Ctrl+C` **no longer cancels the turn** (interrupt
     moved to `Esc`); a double-press exit tears the app down, which cancels any in-flight turn.
     - **Boundary ‚Äî a turn completing between the two presses disarms the window.** Both the
       `Ctrl+C` control dispatch and the running turn task write `runtime.state`; the turn's
       completion writes `last_interrupt_time=0.0` (`main.py:541`-region). Because `Ctrl+C` no
       longer cancels the turn, a turn that **finishes inside the <2 s window between presses**
       overwrites the armed `last_interrupt_time`, so the second `Ctrl+C` re-arms instead of
       exiting. The common case (a long turn still running across both presses) is unaffected ‚Äî the
       turn hasn't written yet, so the press-1 arming survives to press-2. Today this can't happen
       (current `Ctrl+C` cancels the turn, so it never completes-and-writes); under the new
       exit-only model it is an accepted edge, not a regression to fix.
   - `Ctrl+D`: EOF ‚Üí `_handle_one_input(eof=True)` ‚Üí exit (unchanged).
   - Implementation note: a bare `Esc` binding shares the ESC prefix with escape sequences (arrow
     keys etc.), so prompt_toolkit applies its escape-disambiguation timeout. Accept the default;
     do not bind `Esc` `eager=True` (it would swallow arrow-key sequences).

C7. **Queue depth is authoritative.** Single source: `len(runtime.queue)`. `accept_handler`
triggers `update_status(...)` on enqueue; the drain triggers one on dequeue. No depth caching.
Toolbar segment `"{n} queued"` for `n > 0`, omitted otherwise; order
`session_label ¬∑ mode ¬∑ {n} queued ¬∑ ctx ¬∑ {bg} bg ¬∑ {n} approval(s)`.
   - **`update_status` must repaint.** Today `TerminalFrontend.update_status`
     (`co_cli/display/core.py:400`) only stores `self._footer_snapshot` ‚Äî it does **not** call
     `self._invalidate()`. Every pre-queue caller happened to sit next to another render trigger
     (streaming deltas, turn-boundary transitions), so the missing invalidate never bit. The drain
     path (`_drain_next` firing inside an `add_done_callback`) is the first caller with **no
     co-located input/render event**, so depth would go stale until the next render. Fix in TASK-1:
     `update_status` calls `self._invalidate()` (a no-op when `_app is None`, so headless/stub tests
     are unaffected). This makes "real time" true for the whole class, not just the queue.

C8. **Queue lifecycle = session lifecycle.** Constructed on `_ReplRuntime` in `_chat_loop` scope;
in-memory; not persisted; lost on exit. The `_drain_and_cleanup` path is unaffected. A drained item
that sets `should_exit` (`exit` / `quit` / an exiting slash command) ends the session via
`app.exit()`; any items still queued behind it are dropped ‚Äî `_drain_next`'s `should_exit` guard
stops the drain. This is intended (exit is exit), not a flush bug.

C9. **Consumer-side exceptions preserve the queue.** A non-cancel exception inside
`_handle_one_input` / `run_turn` is caught by the existing arm at `main.py:534`, surfaced to the
user, and the drain hook still advances to the next queued item. Error paths never flush the queue.

C10. **`Frontend` protocol stays stable.** No new methods. New state (`queue_depth`) flows via the
existing `StatusSnapshot`. `HeadlessFrontend` needs only the new dataclass-field default.

C11. **Failure modes to guard against (each gets a test):**
   - Type-ahead during a turn is lost ‚Üí typed-ahead enqueues and drains FIFO after the turn.
   - Depth drift ‚Üí depth observed via `update_status` matches `len(queue)` after each transition.
   - `Esc` mid-turn fails to advance ‚Üí after `Esc`, the next queued item runs as the next turn.
   - Double `Ctrl+C` still exits cleanly with queued items present (queue dropped, cleanup fires).

**Dispatch signature (the diagram elides it for width).** `_dispatch` is keyword-only with
*both* args required: `_dispatch(*, user_input: str | None, eof: bool)` (`co_cli/main.py:512`).
Every call mirrors the existing accept-handler ‚Äî `dispatch(user_input=text, eof=False)`,
`dispatch(user_input=None, eof=False)`, `dispatch(user_input=None, eof=True)`. The `dispatch(...)`
shorthands above drop `eof=False`; real calls must not.

**Why a `deque`, not `asyncio.Queue`.** Nothing awaits `queue.get()` ‚Äî there is no consumer
coroutine. The drain is triggered by the turn task's `add_done_callback`. A `deque` with explicit
`append`/`popleft` is the correct primitive; `asyncio.Queue` would add await semantics we don't use.

**Element type is `deque[str]` ‚Äî not a frozen contract.** Phase 1 queues raw input strings. A later
phase (Phase 2 stable-id `/queue pop`, or Phase 3 distinguishing user vs background-sourced prompts)
may promote this to `deque[QueueItem]` (a small dataclass). That is an accepted, cheap refactor ‚Äî the
queue is in-memory and session-lifetime (C8), so there is **no persisted state and no migration**;
it touches only the enqueue/drain/preview sites + their tests in one layer. Do not design Phase 1
around a richer element type speculatively, but do not treat `deque[str]` as immovable either.

**Why `_arm_turn` + `add_done_callback`.** Both the idle-submit path and the queued-drain path must
(1) set `runtime.turn_task` and (2) attach the drain callback so the *next* item advances. Routing
both through one `_arm_turn` helper keeps the turn-task lifecycle in one place; the done-callback
makes "cancel is a turn boundary" (C4) fall out for free ‚Äî `Esc`-cancel and normal completion both
fire `_drain_next`.

**Concurrency discipline (inherited from Phase 0).** The `accept_handler` only enqueues/schedules,
never blocks the app input loop. Exactly one turn owns the terminal at a time ‚Äî the queue buffers,
it does not run turns concurrently ‚Äî so Phase 0's BC4 two-output-path ordering caveat does **not**
bite: drain happens strictly at turn boundaries.

## Deferred Phases (scoped, not yet planned)

### Phase 2 ‚Äî Queue UX & control

Goal: the queue is inspectable and editable, and multi-prompt drain is ergonomic.
- `/queue` (list pending), `/queue clear` (flush), `/queue pop` (drop last) ‚Äî the management
  surface deferred from Phase 1.
- Queued-item preview in the toolbar / in-flight region (not just a count).
- Optional **collect mode**: gemini-style `join("\n\n")` merge of pending items into one turn,
  config-gated; FIFO one-per-turn stays the default.
- Edit-before-send for a queued item.
- Re-examine the BC4 two-output-path ordering seam only if any Phase 2 feature drains off a turn
  boundary (none currently planned to).

### Phase 3 ‚Äî Advanced policy & mid-turn steering

Goal: bounded queues and richer busy-input intents ‚Äî gated on a primitive co lacks today.
- Queue cap + drop policy (`old` / `new` / `summarize`) for runaway typing.
- Hermes-style `busy_input_mode` ‚àà {`interrupt`, `queue`, `steer`}. `interrupt`/`queue` are
  reachable on the Phase 1 foundation; **`steer` (inject into a *running* turn without cancelling)
  requires a `run_turn()` mid-turn injection primitive that does not exist** ‚Äî this is the real
  cost and why it is last.
- Background-event auto-enqueue: dream daemon / task completions push synthesized prompts into the
  same FIFO (hermes pattern ‚Äî uniform delivery surface).

### Rejected (all phases)

- File-backed queue persistence ‚Äî the in-memory session-lifetime queue (C8) is deliberate.
- Alt-screen / full-screen TUI (Textual / Option C) ‚Äî rejected in Phase 0.
