## Problem & Outcome

**Problem:** Four runtime bugs survive into the TUI pre-launch window. The most impactful (F4) causes the memory extractor's cursor to drift unboundedly after any inline compaction event, resulting in perpetual fallback extraction and stale delta slices on every subsequent cadence tick. The remaining three are observability and UX gaps that reduce debuggability and user confidence during testing.

**Failure cost (F4 — functional correctness, surfaces in any session with compaction):** After any compaction event, `last_extracted_message_idx` accumulates to `cursor_start + len(delta)` where `cursor_start` is the pre-compaction value and `delta = next_history[-20:]` (fallback). On the next cadence tick, cursor is still `>> len(next_history)`, triggering the fallback again. The cursor never normalizes, causing every subsequent extraction to re-scan the most recent 20 messages — re-extracting already-processed content and skipping genuinely new content. This is a functional correctness issue that appears in any session long enough to trigger inline compaction.

**Failure cost (F5 — hardening-grade, reduces test confidence):** Shell policy `DENY` blocks silently return `tool_error` with no `tool_denied` log event. Security analysts and testers cannot distinguish policy blocks from transient failures in OTel traces.

**Failure cost (F6 — hardening-grade, reduces test confidence):** MCP discovery failures appear as ephemeral status banners at startup (visible for ~2 seconds) but are not recorded in `deps.degradations`. There is no programmatic way to query which MCP servers failed after startup.

**Failure cost (F7 — hardening-grade, reduces user confidence):** A hung segment times out with "LLM segment timed out — model did not respond in time." The user has no actionable next step and may not know whether to retry, shorten their prompt, or check model health.

**Outcome:** All four bugs fixed. The compaction cursor resets correctly. Shell denials emit structured log events. MCP failures are recorded in degradations. Timeout messages are actionable.

## Behavioral Constraints

1. **F4 fix must not extract on compaction turns.** When `deps.runtime.history_compaction_applied` is True, `_finalize_turn()` must reset `last_extracted_message_idx` to `len(next_history)` and skip the cadence-gated extraction for that turn. Compacted content has already been summarized — double extraction is wasteful and semantically wrong.
2. **F4 fix must not touch the extractor's internal cursor-update logic.** The `cursor_start + len(delta)` formula in `_extractor.py:133,140` is correct for non-compaction cases and must not change.
3. **F5 structured log must match the existing format.** Shell DENY must emit `logger.debug("tool_denied", ...)` using the same positional format as the user-denial log at `orchestrate.py:217–222`: `tool_name`, `subject_kind`, `subject_value`.
4. **F6 degradation entry must use the existing `degradations` dict schema.** Key = `"mcp.<prefix>"`, value = one-line error string. Must not alter how other degradations are written.
5. **F7 message must be a single-line `frontend.on_status()` call — no new UI primitives.** Message must include `co config` as the concrete next step. Must NOT reference `_LLM_SEGMENT_HANG_TIMEOUT_SECS` — the constant is hardcoded with no env-var override; mentioning it implies false configurability.
6. **No test skips.** All new tests must pass or fail (no `pytest.mark.skip`). Tests must use real `CoDeps` — no mocks.
7. **All tasks touch ≤5 files.** Each task is completable in a single agent session.

## High-Level Design

### F4: Compaction cursor reset (`_finalize_turn`)

`_finalize_turn` in `main.py` already has access to `deps.runtime.history_compaction_applied` and `next_history`. The fix:

```python
# After: next_history = turn_result.messages
if deps.runtime.history_compaction_applied:
    # Summarizer already processed the compacted content.
    # Reset cursor to end of compacted history — skip extraction this turn.
    deps.session.last_extracted_message_idx = len(next_history)
else:
    # Normal extraction cadence (existing logic unchanged)
    n = deps.config.memory.extract_every_n_turns
    ...
```

This is a two-line guarded reset. The extractor's internal `cursor_start + len(delta)` formula remains correct for all non-compaction turns.

### F5: Shell deny structured log (`shell.py`)

Add `logger.debug(...)` call before `return tool_error(...)` for `DENY` decisions, using the same format as the user-denial log in `orchestrate.py`. Shell-deny subject kind = `"shell"`, value = first token of `cmd` (consistent with `ApprovalKindEnum.SHELL` subject resolution in `tool_approvals.py`).

```python
if policy.decision == ShellDecisionEnum.DENY:
    logger.debug(
        "tool_denied tool_name=%s subject_kind=%s subject_value=%s",
        "run_shell_command",
        "shell",
        cmd.split()[0] if cmd.strip() else "",
    )
    return tool_error(policy.reason, ctx=ctx)
```

### F6: MCP degradations dict (`bootstrap/core.py`)

After the existing `for prefix, err in discovery_errors.items(): frontend.on_status(...)` loop, also record each discovery failure in `degradations`. The dict is assembled just before `create_deps` returns it:

```python
for prefix, err in discovery_errors.items():
    frontend.on_status(f"MCP server {prefix!r} failed to list tools: {err}")
    degradations[f"mcp.{prefix}"] = err[:120]  # cap to one line
```

`degradations` is declared and passed into `create_deps`'s local scope (already exists at that call site).

### F7: Timeout message with recovery guidance (`orchestrate.py`)

Replace the bare timeout message with a two-part message: what happened + what to do. The internal constant `_LLM_SEGMENT_HANG_TIMEOUT_SECS` is hardcoded with no env-var override — do not reference it in the user-facing message.

```python
except TimeoutError:
    frontend.on_status(
        "LLM segment timed out — model did not respond."
        " Try a shorter prompt, or check model health with `co config`."
    )
```

No new config is needed.
