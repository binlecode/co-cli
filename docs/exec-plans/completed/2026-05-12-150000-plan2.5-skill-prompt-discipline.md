## Context

A comparison with hermes (2026-05-12) revealed that co-cli's skill tools ship *how* the model uses them but not *when*. Hermes encodes behavioral triggers directly in its `SKILL_MANAGE_SCHEMA` description — a multi-paragraph string that tells the model when to create, when to patch, what quality a skill should have, and to offer saving after complex tasks. Co-cli's tool docstrings are functional but silent on all of this.

The second gap: co-cli's context rules (`04_tool_protocol.md`, `05_workflow.md`) have zero skill-specific guidance. The model has no protocol for skill discovery, use, drift correction, creation, or collaborative offer-to-save. Plan 3 already planned `06_skill_protocol.md`; this plan extracts and ships it earlier so the docstring enrichment and the protocol rule land together as a coherent LLM instruction layer.

### Why both docstrings and a protocol rule

They serve different instruction paths:

- **Docstrings** are read by the model at schema-load time, before any turn. They answer: *given that I'm calling this tool, what should I know?* Good for per-tool pre-conditions (read before edit, search before create, quality bar for `create` content).
- **Protocol rule** is injected into the system prompt at agent construction. It answers: *what are my standing behavioral commitments for this capability?* Good for proactive reflexes (discover before starting, drift-correct without being asked, offer-to-save after complex work).

Both are needed. Docstrings alone don't drive proactive behavior; the protocol rule alone doesn't give per-tool guidance at call time.

## Problem & Outcome

**Problem.** The model has five skill tools but no behavioral instruction governing when to use them proactively:
1. No creation triggers → model never promotes a successful pattern to a skill.
2. No patch-on-failure rule → model won't self-correct a skill it just used and hit issues with.
3. No discovery trigger → model may skip `skill_search` entirely for user-installed skills.
4. No read-before-write discipline → model may `edit` without inspecting current content.
5. No quality bar at call time → `skill_manage(action='create')` content may violate §6.
6. No offer-to-save → model never surfaces skill creation to the user after complex work.

**Outcome.**

1. **`skill_manage` docstring** — enriched with: creation triggers (when to call `create`), patch-on-failure rule (`patch` immediately after hitting an issue with a loaded skill), quality bar reference (§6 contract: description, phases, invocation line), offer-to-save prompt.
2. **`skill_search` docstring** — enriched with: dedup guard (search before creating to avoid duplicates), trigger guidance (when manifest doesn't cover the need).
3. **`skill_view` docstring** — enriched with: read-before-write discipline (call before `edit` or `patch`).
4. **`co_cli/context/rules/06_skill_protocol.md`** — five-reflex scaffolding: Discovery, Use, Drift, Create, Offer-to-save. Tone and register match existing rules files.
5. **Behavioral tests** — verify protocol rule loads into rendered prompt; verify five section headers; verify docstring key phrases present.

## Behavioural Constraints

1. **Docstrings describe behavior, not mechanics.** The existing docstrings already cover mechanics (args, actions, return shape). The enrichment adds *when* and *why*, not more *how*.
2. **Docstring additions are additive, not rewriting.** Existing arg descriptions stay. New guidance goes in a dedicated paragraph before or after the existing content, clearly delimited.
3. **Protocol rule ≤80 lines.** Matches the register of existing rules files. Dense prose, no tables, no headers below `##`.
4. **Five distinct reflexes, five `##` sections.** One section per reflex: Discovery, Use, Drift, Create, Offer-to-save. No collapsing.
5. **Create trigger is quantitative.** "After a coherent procedure of 3+ steps you'd repeat" — matches Plan 3's design decision. Not "when it feels right."
6. **Offer-to-save requires user confirmation.** Distinct from the autonomous Create reflex. The rule must say "confirm with the user before invoking `skill_manage(action='create')`."
7. **Discovery rule is conditional, not mandatory.** Fires "at the start of any multi-step task," not on every turn. Avoids wasting a tool call on trivial replies.
8. **Quality bar in `skill_manage` docstring references §6 by name.** The model knows to look up the authoring contract; we don't repeat it inline.
9. **Dedup guard in `skill_search` is instructive, not blocking.** "Search before creating" is guidance, not enforced by code. The docstring is the enforcement point.

## High-Level Design

### `skill_manage` docstring enrichment

New paragraph added after the existing action table, before the Args section:

```
Behavioral guidance:

Create when: a multi-step procedure (3+ coherent steps) succeeded and
is likely to recur for the same kind of task — or the user explicitly
asks to save a workflow. Name by task type, not the specific instance.
Content must conform to skill.md §6 (description, H1, **Invocation:**
line, at least one ## Phase N — <name> section).

Patch immediately when: you loaded and executed a skill and hit an
issue not covered by it — wrong command, missing step, stale output.
Fix the skill before finishing the task; don't leave it degraded.

Read first: call skill_view(name) before edit or patch to inspect
current content.

Offer-to-save: after difficult or iterative work, briefly offer the
user a save — "Want me to save this as a /<task-type> skill?" Confirm
before invoking create on their behalf.

Search before creating: call skill_search(query) to confirm no skill
for this task type already exists.
```

### `06_skill_protocol.md` — five-reflex rule file

Verbatim content (from Plan 3's High-Level Design, reproduced here as the canonical source):

```markdown
# Skill protocol

Skills sit at a different operational tier than memory. Memory holds
facts you recall to inform reasoning during a task; skills hold
procedures that define how to structure the task itself. Before any
multi-step task, check the skill surface first; recall facts within
whatever procedure applies.

The bundled skill manifest is visible above (in the <available_skills>
block). User-installed and dynamically-created skills are searchable
via `skill_search`.

## Discovery

At the start of any multi-step task, scan the bundled skill manifest.
If exactly one skill clearly applies, load it with `skill_view(name)`
and follow it.

If the manifest does not cover what you need, call
`skill_search(query)` with keywords from your goal. User-installed
skills and skills you created in this session live there. Err on the
side of searching — it is cheaper to look and find nothing than to
skip a relevant skill.

Skip discovery for trivial single-step replies where the manifest
clearly doesn't apply.

## Use

A skill body loaded via `skill_view` is your procedure for the task,
not reference material to summarize. Follow its steps. The skill
defines how the task is done here — its phases, rules, and terminal
invariants take precedence over your default approach.

If the skill calls for tools or commands you don't recognize, look them
up before executing. Don't substitute.

## Drift

If a skill you loaded has stale steps, wrong commands, missing
pitfalls, or no longer matches the codebase, fix it immediately. Call
`skill_manage(action='patch', name=<skill>, old_string=...,
new_string=...)` for surgical fixes, or
`skill_manage(action='edit', name=<skill>, content=...)` for structural
overhauls. Don't wait to be asked. Unmaintained skills become
liabilities.

## Create

After completing a multi-step task (3+ coherent steps), consider
whether the procedure is reusable. If yes — same steps you'd run for
similar tasks — promote it to a skill with
`skill_manage(action='create', name=<task-type>, content=<§6-body>)`.
Name by task type, not the specific instance. Content must conform to
`skill.md` §6: description, H1, `**Invocation:**` line, at least one
`## Phase N — <name>` section.

Search first: call `skill_search` to confirm no skill for this task
type already exists.

Don't create for one-offs. The bar is "would I run this again for the
same kind of task."

## Offer-to-save

After difficult or iterative work where you executed a coherent
procedure, briefly offer the user a skill creation suggestion:
"This looked like a reusable procedure. Want me to save it as a
/<task-type> skill?" Skip for simple one-offs. Confirm with the user
before invoking `skill_manage(action='create')` on their behalf — the
Create reflex above covers autonomous creation; this rule covers
collaborative creation.
```
