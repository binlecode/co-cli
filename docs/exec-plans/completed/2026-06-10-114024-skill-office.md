# Skill — `office` (docx / pptx / xlsx extraction over `shell_exec`)

Task type: code + skill content

> **Built on the folder-skill model** (every skill is a `<name>/SKILL.md` folder that can carry
> bundled assets), shipped in **v0.8.332**
> (`docs/exec-plans/completed/2026-06-09-171515-skill-folder-model.md`). **Sibling** of
> `2026-06-09-093734-skill-documents.md` (PDF). This plan adds `office/SKILL.md` +
> `office/scripts/extract_office.py`. Split out of the documents plan: PDF (pymupdf4llm,
> page-chunk citations) and Office (OOXML, slide/sheet/heading structure) are different
> extraction backends and earn separate skills.

## Context

co is blind to Office documents the same way it was blind to PDFs: `file_read` rejects binary,
so a `.docx`/`.pptx`/`.xlsx` is an unconditional "I can't." This is on-mission ("Personal
Assistant CLI") — Office files are second only to PDFs as the local artifact a user hands an
assistant. It rides the same rails as the PDF skill: a lean dep + a thin subprocess script + a
bundled skill, **no new tool, no model-multimodal plumbing**.

This is the format-breadth half of the original `documents` plan, promoted to its own skill
(per the split decision). The PDF skill keeps pymupdf4llm for its page-chunk citation contract;
Office formats have no stable page model (slides, sheets, reflowing prose), so they get their
own backend and citation contract.

**Why a skill + script, not a tool** — identical to the PDF plan: zero model-visible surface,
zero `categories.py`/`toolset.py` churn, subprocess isolation (a crash in an OOXML parser on a
malformed file cannot touch the agent heap). Helper ships **inside the skill folder** at
`co_cli/skills/office/scripts/extract_office.py`, exposed to the agent as the `co-extract-office`
console entry point (`[project.scripts]`, exactly as `co` and the PDF skill's `co-extract-pdf`)
and invoked by the bare command `co-extract-office <file>` through `shell_exec` — reached only as
a subprocess, never imported into the agent process. `office/` and `office/scripts/` carry
docstring-only `__init__.py` so the `:main` entry-point module path import-resolves;
`extract_office.py` exposes a `main()` entry-point target.

**Why a console entry point, not `uv run python -m` (corrected at Gate 1, code-grounded by the
shipped `documents` skill).** `shell_exec` spawns via `asyncio.create_subprocess_exec` with a
**sanitized allowlist env** (`shell_env.py` — `PATH` propagates, `VIRTUAL_ENV`/`PYTHONPATH` do
**not**) and `cwd` = the user's `workspace_dir`, **not** co's repo. Under that env `uv run`
resolves its environment from `cwd` (the user's project, which has no `co-cli` dependency — and
on an installed wheel, no project env) → `ModuleNotFoundError`; plain `python -m` fails too (a
`uv tool install` puts only entry-point scripts on `PATH`, not the venv's `python`). A
`[project.scripts]` entry point sidesteps this — the installer writes a `PATH` wrapper into the
same `bin/` as `co`, working identically in dev checkout and installed wheel. The PDF skill
(`co-extract-pdf`, v0.8.340) proved this is the only invocation that resolves under `shell_exec`;
`office` mirrors it.

### Hermes parity reference (code-verified 2026-06-10)

Hermes does **not** use one unified converter for Office. It uses **format-specific** libraries:
`python-docx` for Word (`skills/.../ocr-and-documents/SKILL.md:16`), a dedicated `powerpoint`
skill on `python-pptx` (`skills/productivity/powerpoint/SKILL.md`), and mentions

backend (`mammoth`/`python-pptx`/`openpyxl` — ML-free, OQ1 resolved) and **diverges** only by
shipping the deps **by default** (out-of-box UX, as in the PDF skill). All extraction is
subprocess, matching co's design.

## Problem & Outcome

**Problem.** co cannot read `.docx`/`.pptx`/`.xlsx`. "Summarize this deck / contract /
spreadsheet" is an unconditional "I can't," blocked only by a missing dependency + a thin script.

**Failure cost:** the assistant silently cannot serve a top-tier local-document request; the
user must convert to text by hand and paste it back. No error teaches a workaround.

**Outcome.**
1. Lean, ML-free format-specific extraction deps added — `mammoth>=1.12.0` (docx),
   `python-pptx>=1.0.2` (pptx), `openpyxl>=3.1.5` (xlsx), latest stable as of 2026-06-11 —
   **no torch/CUDA, no onnxruntime/magika, no pandas** (OQ1 resolved).
2. `co_cli/skills/office/scripts/extract_office.py` — subprocess CLI module: docx/pptx/xlsx →
   markdown with format-appropriate structure markers (slides, sheets, headings), plain stderr
   + non-zero exit on errors.
3. `co_cli/skills/office/SKILL.md` — locate → extract → answer; routes `.pdf` to the
   `documents` skill, other binaries to `file_read`.
4. Real-file tests + committed fixtures (one tiny each: docx, pptx, xlsx); bundled gate green
   (`office` in `_BUNDLED_NAMES`).
5. Ships once it passes lint + the bundled gate + the new tests (folder-skill model already in code).

## Behavioral Constraints
1. **Substrate gate.** Body references only `file_search`, `web_fetch`, `file_read`,
   `shell_exec` (+ the bundled extractor via the `co-extract-office` console entry point), and
   routes `.pdf` to the `documents` skill. No `file_find`, no `uv run python -m`.
2. **Marginal-value gate.** Multi-step locate → extract → answer with a by-extension decision tree.
3. **Native body.** co tool names + co's own module only; no extraction-library or hermes script
   names leaking into the prose (the script wraps the libraries so the body stays native).
4. **Process isolation (hard).** Extraction runs via `shell_exec` subprocess (the
   `co-extract-office` console entry point), never imported into the agent process.
5. **Approval transparency.** First extraction prompts approval (`co-extract-office` not in
   safe_commands); session-subject auto-approval thereafter (`co_cli/context/orchestrate.py`).
   The body describes this honestly and does not oversell the safe_commands opt-in
   (`_FORBIDDEN_ARG_PATTERNS` still rejects `/`, `~`, `..` args, so non-workspace-root paths
   re-prompt by design).
6. **Graceful failure.** The script exits non-zero with a plain one-line stderr message (no
   traceback) for: missing file, unsupported extension, corrupt/unparseable OOXML, and the
   **password-protected/encrypted** case (its own distinct message), so the user learns the cause.
7. **Lean dependency.** OOXML extraction only via the format-specific libs (`mammoth`/
   `python-pptx`/`openpyxl`). Hard gate (verifiable, as in the PDF plan): the resolved tree
   carries **no torch/CUDA**, and — by choosing format-specific over markitdown — **no
   `onnxruntime`/`magika` and no `pandas`** either. There is no footprint "ceiling" to weigh: the
   backend is ML-free by construction.
8. **Read-only synthesis.** No write tools.
9. **Bundled, test-gated.** Lands in `co_cli/skills/office/SKILL.md`; passes lint R1–R4 + B1 +
   security scan; registered in `_BUNDLED_NAMES`.
10. **Lint-clean script (hard).** `extract_office.py` lives under `co_cli/` and is in no
   per-file-ignore, so it must emit via `sys.stdout.write`/`sys.stderr.write` (ruff **T20**
   forbids `print` there) and otherwise pass `scripts/quality-gate.sh lint` — same as the PDF
   skill's `extract_pdf.py`.

## High-Level Design

Three layers, mirroring the PDF skill:

1. **Dependency layer** — the OOXML extraction backend: **format-specific libraries** (OQ1
   resolved) — `mammoth` (docx), `python-pptx` (pptx), `openpyxl` (xlsx). ML-free by
   construction: the resolved tree carries no torch/CUDA, no `onnxruntime`/`magika`, no `pandas`
   (the markitdown-unified path was rejected precisely because it drags those in for a
   deterministic, extension-dispatched task — see OQ1). co's only divergence from hermes here is
   shipping these deps by default for out-of-box UX.

2. **Extraction layer** — `co_cli/skills/office/scripts/extract_office.py`, a stdlib-`argparse`
   CLI with a `main()` entry point, exposed to the agent as the `co-extract-office` console
   command and invoked `co-extract-office <path>` (see the Context section for the full
   spawn-env rationale — a `[project.scripts]` entry point resolves via `PATH` under
   `shell_exec`'s allowlist env, unlike `uv run python -m`). It **dispatches by extension**
   (`.docx`→`mammoth`, `.pptx`→`python-pptx`, `.xlsx`→`openpyxl`), and emits markdown
   with **format-appropriate structure markers**: `## Slide N` for pptx, `## Sheet <name>` for
   xlsx (sheets as markdown tables), and the document's own markdown headings for docx (no
   synthetic markers — docx has no page model).
   - **docx citation contract degrades to prose.** mammoth maps Word **heading styles** to
     markdown headings, but real-world docx often use direct formatting (bold/large text) with no
     heading styles — so the extraction may carry few or no `#` anchors. The contract is "native
     headings where they exist, otherwise flat prose"; the body must not promise heading citations
     for docx the way it does slide/sheet citations for pptx/xlsx (OQ2).
   - **xlsx output is bounded.** openpyxl serializes every cell; a large sheet would otherwise
     produce an unbounded markdown table that blows the agent's context. The script caps rows per
     sheet (default knob, e.g. 1000) and, when a sheet exceeds the cap, emits an explicit
     `[truncated: showing rows 1–N of M]` line — same "never let a capped read look complete"
     discipline as the PDF skill's `total_pages` notice. (Mirrors `extract_pdf.py`'s page cap.)
   - **password/encrypted detection sniffs the container, not the lib error.** An encrypted OOXML
     file is an OLE/CFB compound document (magic bytes `D0 CF 11 E0 A1 B1 1A E1`), **not** a Zip —
     so all three libs fail it with a generic "not a zip / bad file" error, indistinguishable from
     genuine corruption. To satisfy Constraint 6's *distinct* password message, the script checks
     the leading magic bytes up front: CFB header → "password-protected/encrypted" message; a Zip
     header (`PK`) that still fails to parse → "corrupt/unreadable" message.
   On any failure it prints a one-line message to
   stderr and exits non-zero (no traceback). Ships inside the skill folder, never imported into
   the agent process — reached only as a `shell_exec` subprocess.

3. **Skill layer** — `co_cli/skills/office/SKILL.md`, a Locate → Extract → Answer overlay.
   Locate uses `file_search` or a user path (Drive route optional + config-gated, as in the PDF
   skill). Extract branches by extension (`.docx`/`.pptx`/`.xlsx` → the script; `.pdf` → defer to
   the `documents` skill; URL → `web_fetch`; other local → `file_read`). Answer synthesizes with
   **slide / sheet / heading** citations from the structure markers. Registered bundled by adding
   `office` to `_BUNDLED_NAMES`.

The two skills are reciprocal: `documents` routes Office formats here; `office` routes `.pdf`
back. Each owns one backend and one citation contract.

## Open Questions
1. **Q (lead):** Unified `markitdown[docx,pptx,xlsx]` vs format-specific libs (`mammoth` for
   docx, `python-pptx` for pptx, `openpyxl` for xlsx)? **A (RESOLVED — format-specific):** Use the
   format-specific libraries. The skill dispatches by extension, so markitdown's content-sniffing
   (`magika`→`onnxruntime`) is dead weight, and its xlsx extra drags in `pandas` — an ML runtime
   and a heavy dep for a deterministic OOXML parse. Format-specific is ML-free, leaner, more
   auditable/reproducible, peer-proven (hermes), and gives co direct control over its own citation
   contract (OQ2 markers) instead of post-processing markitdown's output. This also removes the
   undefined footprint "ceiling": the backend is lean by construction, with `no torch/CUDA` the
   (trivially satisfied) hard bar.
2. **Q:** Citation contract for paginationless formats? **A (resolved):** pptx → `## Slide N`;
   xlsx → `## Sheet <name>` with the sheet as a markdown table (row-capped with an explicit
   truncation notice); docx → the document's own markdown headings **where Word heading styles
   exist** — mammoth only emits `#` anchors for real heading styles, so a directly-formatted docx
   degrades to flat prose with no anchors (no synthetic page markers — docx reflows). docx
