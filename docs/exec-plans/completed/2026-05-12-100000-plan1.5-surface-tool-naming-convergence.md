## Context

After Plan 1 shipped, three model-callable tiers existed:

| Tier | Search | Read body | Write | Status post-Plan 1 |
|---|---|---|---|---|
| session | `memory_search(channel='session', ...)` | `memory_read_session_turn(...)` (source-only, unregistered) | — | mixed |
| knowledge | `memory_search(channel='knowledge', ...)` | `file_read(path)` via path leaked from search result | `knowledge_manage(action=...)` | mixed |
| skill | `skill_search(query)` | `skill_view(name)` | `skill_manage(action=...)` | converged ✓ |

The skill tier is already converged on the hermes pattern (`_search` + `_view` + `_manage(action)`). The memory tiers are **not** — they still share one polymorphic `memory_search` tool with a `channel` arg, route body reads through the generic `file_read` (leaking filesystem paths into the model's prompt), and the verbatim session-turn reader is source-only.

### Peer audit findings

Both hermes and openclaw **reject** polymorphic cross-tier search. Convergence:

- **Hermes** (`tools/memory_tool.py`, `tools/session_search_tool.py`, `tools/skills_tool.py`): three distinct families, never unified. Polymorphic dispatch is per-family via `action=` on **write** surfaces (`memory(action=add|replace|remove)`, `skill_manage(action=create|edit|patch|delete|...)`). Read surfaces are one tool per family — `session_search(query)`, `skills_list`, `skill_view(name)`.
- **Openclaw** (`extensions/memory-core/src/tools.ts`, `src/agents/tools/sessions-*`): many small verb-per-tool tools — `memory_search`, `memory_get`, `memory_store`, `memory_forget`, `sessions_list`, `sessions_history`. No `action=` dispatch.

Both peers diverge on read/write granularity (hermes prefers polymorphic write, openclaw prefers many small writes) but **converge** on the structural decision: **session and knowledge are separate tool families, never sharing a `channel` discriminator**.

Hermes formalizes exactly one body-load verb across its surfaces: **`skill_view(name)`**. No `_read` tool exists in hermes for any name-addressed surface.

### Why now, not later

Plans 2/3/4 will ship bundled-skill bodies, lint messages, and protocol files that reference tool names. Shipping them against `memory_search(channel='knowledge', ...)` would lock the wrong shape into the workflow bodies; renaming after would require touching every Plan 2/3/4 artifact. One-day cost now vs. four-plan churn later.

## Problem & Outcome

**Problem.** Plan 1 left two tiers (session, knowledge) sharing one polymorphic tool with a `channel` arg, while the third tier (skill) was decomposed into three single-purpose tools. The asymmetry is a tier-conflation residue: the agent's prompt has to teach two different mental models depending on which tier it's hitting, and downstream plans would entrench this inconsistency. Body reads for knowledge artifacts leak filesystem paths through `file_read`; session verbatim reads are source-only and never reach the model.

**Outcome.**

1. **`memory_search` deleted.** Hard removal — no aliases. Split into two registered tools.
2. **`session_search(query, limit)`** — new registered read tool. Ranked FTS over past session transcripts, chunk-cited.
3. **`knowledge_search(query, kinds?, limit)`** — new registered read tool. Ranked FTS over `~/.co-cli/knowledge/*.md`. Drops the `channel` arg.
4. **`knowledge_view(name)`** — new registered read tool. Loads full artifact body by `filename_stem`. Mirrors `skill_view(name)` shape. Eliminates the path-via-`file_read` leak.
5. **`session_view(session_id, start, end)`** — promoted from source-only `memory_read_session_turn` to registered tool. Renamed for verb consistency.
6. **All five new/renamed tools registered in `_native_toolset.py`.** `memory_read_session_turn` and `memory_search` removed from the registry.
7. **No surface keeps a `channel`-style polymorphic discriminator.** Each tool serves exactly one tier. The single allowed polymorphism is `action=...` on `_manage` write surfaces (hermes pattern).
8. **Spec restructure.** `memory.md`, `memory-knowledge.md`, `memory-sessions.md`, `tools.md`, `system.md`, `core-loop.md`, `bootstrap.md`, `dream.md`, `04_tool_protocol.md`, `CLAUDE.md` — update tool names and drop cross-channel framing.
9. **Test churn.** Split `test_flow_memory_recall.py` into per-surface tests; promote canon-recall tests' equivalents; new tests for `session_search`, `knowledge_view`, `session_view`.

## Behavioural Constraints

1. **One tool per (surface, role).** A surface = `{session, knowledge, skill}`. A role = `{search, view, manage}`. Each (surface, role) pair is at most one registered tool. No tool covers multiple surfaces.
2. **No `channel`-style polymorphism on read surfaces.** `memory_search(channel=...)` is the anti-pattern this plan eliminates; nothing analogous should appear.
3. **`action=` is the only polymorphic discriminator allowed**, and only on `_manage` writers. (Hermes pattern, already in `knowledge_manage` and `skill_manage`.)
4. **`_view` is the universal body-load verb** for name-addressed surfaces (`skill_view`, `knowledge_view`) and for the location-addressed session turn reader (`session_view`). One verb — hermes pattern. `_read` is reserved for path-addressed generic file reads (`file_read`).
5. **`_search` returns ranked hits; `_view` returns full content.** Search results never carry full bodies inline — model loads bodies on demand via `_view`. (Exception inherited from Plan 1: canon is auto-injected, not searched.)
6. **Hard removal of `memory_search` and `memory_read_session_turn`.** No aliases. Calling the old names is a clear `AttributeError` at import time for any stale caller — fail loud.
7. **Approval-free read surfaces.** All five read tools (`session_search`, `knowledge_search`, `knowledge_view`, `session_view`, `skill_search`/`skill_view`) are `is_read_only=True`, `is_concurrent_safe=True`, no approval.
8. **Empty-query semantics preserved per surface.** `knowledge_search()` empty query → browse recent artifacts (replaces Plan 1's `_dispatch_all_channels` browse). `session_search()` empty query → browse recent session metadata. `skill_search('')` → `tool_error` (Plan 1 decision; bundled manifest covers browse).

## High-Level Design

### Target tool surface (post-Plan-1.5)

| Surface | Search | Read body | Write | Lifecycle |
|---|---|---|---|---|
| **session** | `session_search(query, limit=3)` | `session_view(session_id, start_line, end_line)` | — | read-only (append happens via `persist_session_history`, not model-callable) |
| **knowledge** | `knowledge_search(query, kinds?, limit=10)` | `knowledge_view(name)` | `knowledge_manage(action=create\|append\|replace\|delete, name, content?, kind?, section?)` | full lifecycle |
| **skill** | `skill_search(query, limit=5)` | `skill_view(name, file_path=None)` | `skill_manage(action=create\|edit\|patch\|delete\|install\|write_file\|remove_file, name, ...)` | full lifecycle |

**Total registered tool delta:** `memory_search` removed (−1); `session_search`, `knowledge_search`, `knowledge_view` added (+3); `memory_read_session_turn` removed (was unregistered, no delta) and `session_view` added (+1). **Net: +3.** Final registry 38 tools (was 35).

### Empty-query behaviour per surface

| Tool | Empty query | Rationale |
|---|---|---|
| `session_search()` | Recent N session metadata (titles, dates, ids) | Browse mode preserved from Plan 1's `_browse_recent`. |
| `knowledge_search()` | Recent N artifacts (title + kind + path + snippet) | Browse mode preserved from Plan 1's `_list_artifacts`. |
| `knowledge_view(name)` | Required arg; no empty-arg semantics | Name-addressed reader. |
| `session_view(session_id, start, end)` | All three args required | Location-addressed reader. |
| `skill_search(query)` | `tool_error` (unchanged) | Bundled manifest in static prompt handles browse. |
| `skill_view(name)` | Required arg; no empty-arg semantics | Name-addressed reader. |

### Result schemas

`session_search` hits (`channel` field dropped — single-surface tool, no discriminator needed):

```python
{
    "session_id": <8-char uuid>,
    "when": <YYYY-MM-DD>,
    "source": "session",
    "chunk_text": <snippet>,
    "start_line": <int>,
    "end_line": <int>,
    "score": <BM25>,
}
```

`knowledge_search` hits:

```python
{
    "kind": <user|rule|article|note>,
    "title": <str>,
    "snippet": <FTS5 snippet>,
    "score": <BM25>,
    "path": <str>,
    "filename_stem": <str>,
}
```

`knowledge_view` return:

```python
tool_output(
    <full body text>,
    ctx=ctx,
    name=<filename_stem>,
    kind=<artifact_kind>,
    path=<str>,
)
```

`session_view` return: unchanged from `memory_read_session_turn` (verbatim turn JSON slice).

## Open Questions

1. **Q:** Should `_grep_artifacts_fallback`'s helper (`grep_recall`) stay in `view.py` after the `read.py` → `view.py` rename, or move into `recall.py` since `recall.py` is its only caller?
   **Tentative answer:** Move into `recall.py` as a private `_grep_recall` helper. Removes a cross-file dependency and `view.py` ends up containing only model-callable readers.

2. **Q:** Should `knowledge_view` accept a `chunk_index` arg for very large artifacts, mirroring the chunked FTS5 layer?
   **Tentative answer:** No. Artifacts are typically small (<10KB); `spill_threshold_chars=inf` is appropriate. Chunked access would re-introduce path-like addressing the plan is removing.

3. **Q:** Should `session_search` keep the `role_filter` arg hermes uses, or stay query-only?
   **Tentative answer:** Stay query-only. Co-cli session JSONLs aren't role-tagged the way hermes's are; a `role_filter` would be a stub. Add later if a real use case appears.

4. **Q:** Should the bundled-skill manifest be expanded to also list bundled knowledge artifacts?
   **Tentative answer:** No (out of scope). The manifest pattern works for skills because there's a small, curated bundled set. Knowledge artifacts are user-generated and dynamic — `knowledge_search` is the right surface for them.

5. **Q:** Should `knowledge_manage(action='create', ...)` return the new artifact body inline (so callers don't need an immediate `knowledge_view` round-trip)?
   **Tentative answer:** No change in this plan. The current `knowledge_manage` return is `{action, path, artifact_id}` — adequate. Body-inline return is a future ergonomics improvement, not a naming-convergence issue.

## Deferred items

- **Compaction-aware skill manifest budget** (openclaw `formatSkillsCompact` pattern). Out of scope; bundled set is 1 today. Worth revisiting when bundled count > 10 or token cost > ~500.
- **`session_manage`.** No model-callable write surface for sessions — sessions are append-only at runtime. If a use case for model-driven session edits emerges (compaction-via-rewrite from the model side, etc.), add then.
- **Cross-tier discovery tool.** A `recall(query)` umbrella that hits all three surfaces in one call is the inverse of this plan; explicitly deferred. If discovery ergonomics suffer, revisit — but the peer audit shows neither hermes nor openclaw ship one.
- **Action-vocab convergence across `_manage` tools.** `knowledge_manage` uses `create/append/replace/delete`; `skill_manage` uses `create/edit/patch/delete/install/write_file/remove_file`. Different surfaces, different semantics; aligning vocab is cosmetic and would regress per-surface clarity.
