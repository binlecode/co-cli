## Context

The current `co_cli/agents/` package contains the agent **factory** and infrastructure — `build_agent`, `discover_delegation_tools`, instructions, MCP wiring, tool-call limits. It does not contain "the agent": the foreground CLI agent is constructed at runtime in `main.py`/`context/orchestrate.py`; foreground delegation agents are constructed in `tools/agents/delegation.py`. The singular package name was always slightly dishonest.

Plan 3.5 forces the issue by adding three new files for background agent orchestrators (`_runner.py`, `session_review.py`, `skill_curator.py`). They were originally slated for a sibling `co_cli/agents/` package — which would create a permanent one-letter-different naming trap (`agent/` vs `agents/`) that fails the cold-reader test.

This plan resolves the collision **before** 3.5 lands by renaming the existing singular package to the plural form. Plan 3.5's new files then join a single unified `co_cli/agents/` package without ambiguity.

1. **Zero behavioural change.** Every test passing before the rename MUST pass after. No logic edits.

## Failure Modes

1. **Missed import.** A stray `from co_cli.agents.X import Y` lingering after the rename → `ModuleNotFoundError` at first use.
   - Mitigation: verification grep (see §HLD) must return zero matches; full test suite must pass.

2. **Doc drift.** Specs/active plans continue to reference `co_cli/agents/` after the rename, misleading future readers.
   - Mitigation: in-scope; same sed pattern applied to `docs/specs/`, `docs/exec-plans/active/`, `agent_docs/`.

3. **Sed false positive on `co_cli.agents.*` itself.** The pattern `s|co_cli\.agent\.|co_cli.agents.|g` is safe (literal `agent.` boundary), but `s|co_cli/agents/|co_cli/agents/|g` would be a no-op if applied twice — idempotent. The dangerous case is sed touching `co_cli/agents/` (plural) — the trailing `s` in the source path naturally rules this out.
   - Mitigation: verification grep uses a character class `[^s/]` to confirm zero singular-`agent` references remain.

4. **`git mv` not detected as a rename.** Large enough content change to push past git's similarity threshold breaks `git log --follow`.
   - Mitigation: the rename + import-line updates per file are well below git's 50% similarity threshold. Verified post-commit by `git log --follow co_cli/agents/core.py` showing pre-rename history.

## High-Level Design

Three mechanical operations, single commit:

1. **Move the package.**
   ```bash
   git mv co_cli/agent/ co_cli/agents/
   ```

2. **Rewrite imports in Python files.**
   ```bash
   find co_cli tests -name "*.py" \
     -exec sed -i '' 's|co_cli\.agent\.|co_cli.agents.|g' {} +
   ```
   (`co_cli.agents.X` → `co_cli.agents.X`; covers both `from co_cli.agents.X import Y` and bare `co_cli.agents.core` references)

3. **Rewrite path references in active docs.**
   ```bash
   find docs/specs docs/exec-plans/active agent_docs -name "*.md" \
     -exec sed -i '' \
       -e 's|co_cli\.agent\.|co_cli.agents.|g' \
       -e 's|co_cli/agents/|co_cli/agents/|g' {} +
   ```

**Verification gate** (must return zero matches):
```bash
grep -rE 'co_cli[\./]agent[^s/]' \
  co_cli tests docs/specs docs/exec-plans/active agent_docs
```

The `[^s/]` character class excludes both `agents` (plural) and `agent/` boundary cases where the trailing `/` is the intended terminator (i.e., already-rewritten paths like `co_cli/agents/...`).
