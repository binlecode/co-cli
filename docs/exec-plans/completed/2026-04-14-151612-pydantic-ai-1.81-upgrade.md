# Plan: pydantic-ai 1.81 Upgrade

Task type: code-feature

## Context

co-cli pins `pydantic-ai==1.77.0` (exact). PyPI latest is `1.81.0`. The pydantic-ai changelog documents no breaking changes across all v1.x releases — the library is API-stable since v1.0.

The single high-ROI fix in the 1.78–1.81 window is from 1.80: **unknown tool calls no longer consume retry counters**. This directly affects co-cli's approval-resume flow.

**How the failure mode manifests today (1.77):**

During an approval-resume turn, `_approval_resume_filter` (`co_cli/agent/_native_toolset.py:36–50`) narrows visible tools to `resume_tool_names` (the approved tools) plus ALWAYS-visible tools. The model may also reference deferred tools it previously discovered via `search_tools` — those breadcrumbs are preserved across compaction by `_preserve_search_tool_breadcrumbs` (`co_cli/context/_history.py:497–503`). On a resume segment those previously-discovered deferred tools are not in scope. The SDK sees them as unknown tool calls and, under 1.77, increments the retry counter for each. In multi-tool approval flows, this can exhaust `config.tool_retries` silently, causing the run to fail or degrade before the approved work completes.

**Other changes in 1.78–1.81** are non-breaking additive features (SetToolMetadata, ToolDefinition.return_schema, Capability.apply, AnthropicCompaction, CapabilityOrdering). None are adopted in this upgrade — low ROI or out of scope per project principles.

No artifact hygiene issues found. No existing plan for this slug.

## Problem & Outcome

**Problem:** co-cli pins `pydantic-ai==1.77.0`, missing the retry-counter fix released in 1.80.

**Failure cost:** During approval-resume turns with previously-discovered deferred tools, the SDK silently burns retry budget on unknown-tool-call errors. In multi-tool approval flows, this can exhaust `config.tool_retries`, aborting the run before approved work completes. The user sees an unexpected failure with no actionable signal.

**Outcome:** Pin updated to `==1.81.0`; lockfile regenerated; full test suite passes; approval-resume flows no longer burn retry budget on deferred tools that are outside resume scope.

## Scope

- Bump `pydantic-ai==1.77.0` → `pydantic-ai==1.81.0` in `pyproject.toml`
- Regenerate `uv.lock` via `uv sync`
- Full test suite gate

**Out of scope:** adopting any new 1.78–1.81 APIs (SetToolMetadata, AnthropicCompaction, etc.).

## Behavioral Constraints

- The co-cli source must require zero modifications to pass `uv run pytest` under pydantic-ai 1.81.0; any API-compatibility breakage discovered during the upgrade must be resolved within TASK-1 before the pin is committed.
- Transitive dependency changes pulled in by `uv sync` are acceptable only if the full test suite (TASK-2) continues to pass.
- Do not adopt `AnthropicCompaction` or other server-side compaction — co-cli's compaction must remain self-contained.

## High-Level Design

Purely a dependency pin change. The retry-counter fix is delivered by the SDK at runtime — no co-cli source changes are required. The only files that change are `pyproject.toml` (pin bump) and `uv.lock` (lockfile regeneration).

## Testing

No new tests added. The fix is in SDK internals (`_agent_graph.py:1371`) — the retry counter decrement on unknown tool calls is removed at the SDK level. Co-cli cannot test SDK-internal retry accounting without a live LLM producing unknown tool calls in a controlled sequence (which would be an eval, not a pytest test, and requires mocking model outputs — prohibited by co-cli testing policy).

Regression coverage: existing `test_tool_registry.py` BC-4 tests verify `_approval_resume_filter` narrowing behavior; these continue to exercise the approval-resume wiring at the co-cli boundary level.

**Manual verification (post-ship):** Run a multi-tool approval scenario where a deferred tool is in scope and confirm the run completes without unexpected retry exhaustion. This is an informal smoke test, not automated — no specific eval script is required.
