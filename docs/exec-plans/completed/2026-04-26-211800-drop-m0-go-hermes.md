# Plan: Drop M0 and Adopt Hermes-Style Single-Trigger Compaction with Aggressive Shape

Task type: behavior change (remove a layer; collapse trigger surface to a single gated trigger; lower threshold and tail defaults to hermes-aligned values) + bundled correctness fixes that the layer removal makes load-bearing

## Context

Co-cli's compaction system today has two trigger points sharing a single ratio (`compaction_ratio = 0.80`):

- **M3** — `proactive_window_processor` (`compaction.py:443-458`), gated. Registered as the last `history_processor`. Fires before each `ModelRequestNode` when token pressure exceeds the ratio, and respects the anti-thrash gate (`consecutive_low_yield_proactive_compactions >= proactive_thrash_window`).
- **M0** — `pre_turn_window_compaction` (`compaction.py:461-485`), ungated. Called once from `co_cli/context/orchestrate.py:573` at `run_turn()` entry. Bypasses the thrash gate by passing `apply_thrash_gate=False` to the shared core.

Both wrappers go through `_compact_window_if_pressured` (`compaction.py:372-440`), which parameterizes the gate-bypass and ratio.

The two-trigger design carries a cost: when M3's thrash gate trips and state stays high, M0 fires every subsequent turn at the same ratio and runs the same summarization code path — burning ~300s/turn of summarizer cost on low-yield retries. M0's "ungated emergency safety net" semantic collapses into "every-turn-after-M3-thrashes tax."

This plan takes the hermes direction: **drop M0 entirely; adopt hermes's design philosophy at hermes-shaped defaults**. Hermes (`hermes-agent/agent/context_compressor.py`) runs at production scale with one trigger, hard stop on thrash, and an aggressive shape (threshold ~0.50, effective tail ~0.10 of context). Their answer to "system can't recover" is "user must `/new` or `/compress <topic>`," not "system keeps trying forever."

Co-cli cannot match hermes's exact values today because the cross-compaction working memory is still ephemeral — the LLM summary marker is regenerated from scratch each pass. Hermes's iterative summary updates (`_previous_summary` carried across compactions, `:208, 299, 705`) close that gap. The sibling plan **`2026-04-27-084804-compaction-tail-compensators.md`** lands the iterative-summary mechanism in co-cli; it is a hard prerequisite for this plan's aggressive shape.

The function-3 gap (action-trace legibility post-truncation) is **already closed** in co-cli today via `co_cli/context/_tool_result_markers.semantic_marker`, which mirrors hermes's `_summarize_tool_result` per-tool one-liner pattern (`[shell] ran \`cmd\` → exit 0, N lines`, etc.). With iterative summary added by the prerequisite plan, co-cli has both load-bearing compensators in place and can safely drop the tail size.

**Aggressive defaults this plan ships:**
- `compaction_ratio`: 0.80 → **0.65** (triggers earlier, more passes per session each at higher savings ratio).
- `tail_fraction`: 0.40 → **0.20** (cross-compaction memory now lives in the iterative summary marker, freeing tail tokens).
- Single trigger, single gate, fail-open.

Trade vs. today: lose autonomous forward progress on thrash, gain a coherent hermes-shape pipeline, eliminate the M0-thrash failure mode entirely, and make the failure surface honest ("auto-compaction stopped helping; act").

**Bundled correctness scope.** Removing M0 makes the M3 gate the only auto-compaction protection. Three correctness issues that today are masked by M0's routine firing become load-bearing post-removal and must land in the same plan:

1. **Thrash counter never resets on `/compact`.** The plan's banner text claims `/compact` helps; pre-existing code resets the counter only on M3 success, OR-1, and OR-2 — not on manual `/compact`. After removing M0, this gap means a successful `/compact` does not unstick the gate, contradicting the banner.
2. **Hint flag (`compaction_thrash_hint_emitted`) never re-arms.** Pre-existing one-shot-per-session behavior; with M0 routinely cycling the system through recovery, the user rarely saw a second thrash. Post-M0, a recurring thrash will not re-emit the inline hint even though the gate has un-tripped and re-tripped.
3. **Savings ratio mixes provider-reported and locally-estimated tokens.** Pre-existing bias (`compaction.py:409, 433-435` — `token_count = max(estimate_message_tokens(messages), reported)` is reused as the divisor of the savings ratio); with M3's gate now sole protection, a 5pp inflation can falsely clear the counter and prevent the gate from engaging.

Each of these is a one- or two-line fix. They live in this plan because the M0 layer was implicitly papering over their consequences.

## Problem & Outcome

**Problem 1 (M0 thrash-bug shape).** With a single ratio at 0.80 and M0 ungated, post-thrash sessions burn ~300s/turn of summarizer cost on low-yield M0 fires until either yield improves or `recover_overflow_history` (OR-1) fires on a provider context-length error.

**Problem 2 (semantic incoherence).** M0's `apply_thrash_gate=False` is documented (`compaction.py:467-472`) as a "safety net for sessions where M3 was suppressed." The threshold (0.80, same as M3) inverts that semantic — M0 fires routinely, not as a safety net.

**Problem 3 (structural value vs. cost).** M0's only unique behavior is "trigger once per turn at run_turn entry, ungated." That is one extra retry attempt per turn beyond what M3 already provides. The retry succeeds when dropped-content composition has shifted enough that summarization yields > 10% (case 7); fails otherwise (case 6). The case-7/case-6 ratio is conversation-shape dependent; without telemetry, the project cannot honestly claim case 7 dominates.

**Problem 4 (user signal honesty).** The current banner text ("Consider /compact to free space") and inline hint ("Run /compact to force a manual pass") suggest user actions that the gate-state machine does not actually credit. `/compact` running successfully does not reset the thrash counter, so the gate stays tripped; the inline hint is one-shot per session, so a recurring thrash event surfaces only the post-turn banner. Removing M0 makes these gaps visible because M0 is no longer running compaction every turn to mask them.

**Outcome:**

- One trigger: `proactive_window_processor`. Fires mid-turn (history-processor slot) when `ratio > compaction_ratio` (now 0.65). Anti-thrash gate engaged.
- Aggressive shape: `compaction_ratio = 0.65`, `tail_fraction = 0.20`. Each pass produces ~60% savings ratio at typical conversation density (vs. ~46% today), making the 10% thrash bar trivially clearable in normal operation. Post-compact state ≈ 0.26 of context; per-pass headroom ≈ 0.39 of context (slightly better than today's 0.37).
- When the gate trips, the system stops auto-compacting and surfaces an actionable user signal naming both `/compact` (force one more attempt) and `/new` (start fresh).
- Pre-turn lifecycle slot (`run_turn` entry) holds no compaction call. `pre_turn_window_compaction` removed.
- Provider context-length errors continue to be handled by `recover_overflow_history` and `emergency_recover_overflow_history` — these are the remaining safety nets.
- Banner and inline thrash hint texts updated to match hermes's "auto-compaction stopped, act now" framing.
- `_compact_window_if_pressured` collapses into `proactive_window_processor`; the `apply_thrash_gate` parameter (now permanently True) is removed.
- M3 wraps in fail-open `try/except Exception` matching the M0-style contract previously held only by `pre_turn_window_compaction` — now-only-trigger has no caller-side handler in the `history_processors` pipeline, so propagating exceptions would fail the turn unnecessarily.
- Successful manual `/compact` resets the thrash counter and re-arms the inline hint flag, so the user signal accurately reflects the system's auto-compaction state.
- Inline hint re-arms at every site that resets the counter (OR-1, OR-2, M3 success, manual `/compact`), so a recurring thrash event surfaces a fresh user signal.
- Savings ratio computed in consistent local-estimate units, so the gate's yield decision is bias-free against provider tokenization differences.
- Pydantic validators on `CompactionSettings` enforce the post-compact-state invariant (`tail_fraction < compaction_ratio`) and basic bounds (`0 < tail_fraction`, `compaction_ratio < 1.0`).

## Behavioral Constraints

1. **Hard prerequisite: iterative-summary plan ships first.** `2026-04-27-084804-compaction-tail-compensators.md` MUST land before this plan. Its iterative-summary mechanism is what makes `tail_fraction = 0.20` safe: cross-compaction working memory lives in the iterative summary marker, not in the tail. Shipping aggressive shape without that prerequisite would expose silent failure modes (decisions diluted across passes, model losing project-level continuity).
2. **Aggressive defaults.** `compaction_ratio = 0.65`, `tail_fraction = 0.20`. Validators on `CompactionSettings` enforce `tail_fraction < compaction_ratio` (post-compact-state invariant) and `0 < tail_fraction`, `0 < compaction_ratio < 1.0`. Existing settings.json files with explicit overrides keep working; the defaults change is opt-out via override.
3. **Single trigger.** After this plan, only one compaction trigger exists: `proactive_window_processor`, registered as a history processor in `agent/_core.py:142-147` (the list runs from line 142 `history_processors=[` to line 147 `]`; `proactive_window_processor` is the last entry at line 146). No other module in `co_cli/` calls a compaction trigger directly.
4. **Pre-turn lifecycle is compaction-free.** `run_turn()` no longer invokes any compaction function at entry. `reset_for_turn()` runs as before; the next line proceeds straight to `frontend.on_status`.
5. **Anti-thrash gate is the only gate.** When `consecutive_low_yield_proactive_compactions >= proactive_thrash_window`, `proactive_window_processor` returns the unchanged `messages`. No other code path fires compaction in this state.
6. **User signal on thrash is actionable and naming both options.** Inline thrash hint and post-turn banner both name `/compact` (force one more attempt) and `/new` (start fresh). Verbatim text in Design Note below.
7. **Fail-open contract on the only trigger.** `proactive_window_processor` wraps its body in `try/except Exception` and returns the unchanged `messages` on failure with a `log.warning(..., exc_info=True)`. `BaseException` (including `asyncio.CancelledError`) propagates. The wrap is defensive — it covers any future raise path from `_run_window_compaction`, `apply_compaction`, or the `extract_at_compaction_boundary` distiller hook (`compaction.py:246`, also fired from OR-2 at `compaction.py:330`), which today is documented as failure-tolerant but reaches non-trivial code (`drain_pending_extraction`, session-state mutation) that the wrap honestly contains.
8. **Counter cadence preserved on M3 path.** `consecutive_low_yield_proactive_compactions` still increments on a `< min_proactive_savings` yield (today at `compaction.py:437`) and resets on a `>= min_proactive_savings` yield (today at `compaction.py:439`) — the existing semantics on the M3 gated branch. After inlining, both lines move into `proactive_window_processor` unchanged (modulo the local-only savings inputs from Constraint #10). At the new shape, typical-pass savings ratios (~60%) clear the 10% bar comfortably; the gate trips primarily when the conversation is genuinely stuck, not from drift.
9. **Counter and hint flag reset on every successful unconditional compaction.** OR-1 (`compaction.py:297`), OR-2 (`compaction.py:333`), and the new `_cmd_compact` reset path all set `consecutive_low_yield_proactive_compactions = 0` AND `compaction_thrash_hint_emitted = False` after a successful compaction. The two flags are paired semantically: counter zero means "auto-compaction state is clean"; hint flag zero means "user signal can re-emit on next thrash event." They reset together. (M3 success path also resets, but only when savings clear `min_proactive_savings` — that branch lives at `compaction.py:439` inside `_compact_window_if_pressured` and moves into `proactive_window_processor` after inlining.)
10. **Savings ratio uses consistent local-estimate units.** Trigger-threshold check uses `max(estimate_message_tokens(messages), reported)`; savings-ratio computation uses `estimate_message_tokens(messages)` and `estimate_message_tokens(result)` only. The two computations consume different inputs because they answer different questions: "is the context full?" must be conservative against estimator drift; "did this run save tokens?" must compare apples to apples.
11. **`compaction_applied_this_turn` semantics preserved.** Set by `apply_compaction` after a successful compaction. Reset by `reset_for_turn()`. The flag continues to drive the stale-token suppression in the post-compaction trigger check and the session-branching in `main.py`.
12. **`_compact_window_if_pressured` removed.** Its body is inlined into `proactive_window_processor`. The `apply_thrash_gate` parameter (always True after this plan) is gone. No private helper that takes a ratio argument remains.
13. **Public surface contracts.** `proactive_window_processor` keeps its `(ctx, messages) -> messages` signature — pydantic-ai requires this exact shape for a registered history processor.
14. **No backwards-compat shim.** Settings.json files containing fields that no longer exist (none today; this plan does not remove config fields) raise Pydantic validation errors per existing `extra="forbid"` policy. Settings.json files setting `tail_fraction >= compaction_ratio` raise the new validator with an actionable message naming both fields and offending values.

## Design Note — Threshold Derivation at New Defaults

**`compaction_ratio = 0.65`** — the trigger threshold. Bounded below by post-compact-state arithmetic:

| | Today (0.80 / 0.40) | This plan (0.65 / 0.20) | Hermes (0.50 / 0.10 effective) |
|---|---|---|---|
| Trigger fires at | 80% of budget | 65% of budget | 50% of context |
| Tail target | 40% | 20% | 10% |
| Head | ~3% | ~3% | ~3% |
| Marker | ~3% | ~3% | ~5% (iterative) |
| Post-compact state | ~46% | ~26% | ~18% |
| Headroom per pass | ~34% | ~39% | ~32% |
| Savings ratio per pass | ~46% | ~60% | ~74% |
| Turns before re-trigger | 3–5 | 3–5 | 3–5 |

The new defaults give *better* per-pass headroom than today (0.39 vs. 0.37) and a much higher savings ratio (60% vs. 46%), making the 10% thrash bar trivially clearable in normal operation. The gate trips primarily when the conversation is genuinely stuck (recent content alone exceeds the dropable middle), not from drift.

**`tail_fraction = 0.20`** — the preserved-tail target. Co-cli's two compensators that make this safe:

1. **Per-tool semantic markers** (`_tool_result_markers.semantic_marker`, already in code) — older tool results are replaced with informative one-liners (`[shell] ran \`pytest\` → exit 0, 47 lines`), preserving action-trace legibility at low token cost.
2. **Iterative summary updates** (added by prerequisite `2026-04-27-084804-compaction-tail-compensators.md`) — cross-compaction working memory accumulates in `previous_compaction_summary` rather than being regenerated each pass. The tail no longer has to carry session-state continuity.

With both compensators in place, the 0.20 tail carries: latest reasoning chain (function 4), enough recent action trace at high density (function 3 — already covered), and the active-task anchor (function 1 — covered by `_anchor_tail_to_last_user`). Function 2 (cross-compaction memory) lives in the iterative marker.

**Validators.** `tail_fraction < compaction_ratio` is the post-compact-state invariant (head + marker + tail must leave headroom before re-trigger). Inverting the order would make every compaction immediately re-trigger.

## Design Note — Inlined `proactive_window_processor` Shape

```python
async def proactive_window_processor(
    ctx: RunContext[CoDeps],
    messages: list[ModelMessage],
) -> list[ModelMessage]:
    """Mid-turn compaction trigger — the only auto-compaction layer.

    Registered as the last history processor on the orchestrator agent. Fires
    before each ModelRequestNode when token pressure exceeds compaction_ratio.
    Anti-thrash gate engages after `proactive_thrash_window` consecutive low-
    yield runs; once tripped, the system stops auto-compacting and surfaces
    a user-actionable hint pointing at /compact and /new.

    Fail-open: any Exception returns ``messages`` unchanged so the agent loop
    proceeds. asyncio.CancelledError (via BaseException) propagates.

    Pre-turn lifecycle slot is intentionally compaction-free — see
    docs/specs/compaction.md for the design choice.
    """
    try:
        ctx_window = ctx.deps.model.context_window if ctx.deps.model else None
        budget = resolve_compaction_budget(ctx.deps.config, ctx_window)
        if budget <= 0:
            return messages
        cfg = ctx.deps.config.compaction

        reported = (
            0
            if ctx.deps.runtime.compaction_applied_this_turn
            else latest_response_input_tokens(messages)
        )
        # Trigger threshold uses max(local, reported) — biases toward earlier compaction.
        # Savings ratio uses local-only on both sides — apples-to-apples yield comparison.
        tokens_before_local = estimate_message_tokens(messages)
        token_count = max(tokens_before_local, reported)
        token_threshold = int(budget * cfg.compaction_ratio)

        if token_count <= token_threshold:
            return messages

        if (
            ctx.deps.runtime.consecutive_low_yield_proactive_compactions
            >= cfg.proactive_thrash_window
        ):
            log.info("Compaction: anti-thrashing gate active, skipping")
            if not ctx.deps.runtime.compaction_thrash_hint_emitted:
                from co_cli.display._core import console

                console.print(
                    "[dim]Compaction paused: recent passes freed too little context. "
                    "/compact to force one more pass, or /new for a fresh session.[/dim]"
                )
                ctx.deps.runtime.compaction_thrash_hint_emitted = True
            return messages

        result = await _run_window_compaction(ctx, messages, budget)
        if result is None:
            return messages

        tokens_after_local = estimate_message_tokens(result)
        savings = (
            (tokens_before_local - tokens_after_local) / tokens_before_local
            if tokens_before_local > 0
            else 0.0
        )
        if savings < cfg.min_proactive_savings:
            ctx.deps.runtime.consecutive_low_yield_proactive_compactions += 1
        else:
            # hint re-arms with counter — banner-text contract
            ctx.deps.runtime.consecutive_low_yield_proactive_compactions = 0
            ctx.deps.runtime.compaction_thrash_hint_emitted = False
        return result
    except Exception:
        log.warning("Mid-turn compaction failed — skipping", exc_info=True)
        return messages
```

Notes:
- The body is the inlined `_compact_window_if_pressured` with `apply_thrash_gate=True` flattened and the parameter removed.
- `tokens_before_local` is computed once and reused — no double-evaluation cost vs. today.
- The hint-flag reset is paired with the counter reset on M3's success path.
- The outer `try/except Exception` is the only fail-open wrap on this processor; `_gated_summarize_or_none` retains its inner catch for summarizer failures (orthogonal — that one tracks `compaction_skip_count` for the circuit breaker).

## Design Note — What This Plan Trades Away

**Lost capabilities:**
1. Automated forward progress on M3 thrash. Sessions where M3's gate trips will not auto-recover; the user must `/compact` or `/new`.
2. M0's "topic-shift retry" win (case 7 in the assessment). When a topic shift makes previously-uncompressible content compressible, the system will not auto-detect this until the user manually triggers `/compact`.
3. The pre-turn lifecycle slot as a defensive checkpoint. Today's `pre_turn_window_compaction` provides one extra "before turn starts" sanity check that disappears.

**Gained capabilities:**
1. Bounded session cost. M0-thrash failure mode is eliminated by construction — it cannot happen because the layer doesn't exist.
2. Honest user signal. When auto-compaction stops, the user knows immediately; no background 300s/turn drift. The `/compact` action the banner suggests actually works as advertised — running it clears the gate state.
3. Bias-free gate decisions. Savings ratio now compares apples to apples, so the gate engages and disengages on accurate yield measurements rather than provider-tokenization-inflated values.
4. Recurring-event signal. The inline thrash hint re-emits on subsequent gate trips after a recovery — surfaces the issue every time it happens, not just the first.
5. Defensive containment of distiller errors. The fail-open wrap protects the agent loop from any future raise path through `extract_at_compaction_boundary` or other downstream.
6. Simpler mental model. One ratio, one trigger, one gate. Documentation, debugging, and review surface contract.
7. Hermes-aligned shape. With the prerequisite iterative-summary plan in place, co-cli converges on hermes's production-validated single-trigger-aggressive-shape design (modulo head expansion, which is deferred). Cross-compaction memory lives in the iterative marker; action trace lives in `semantic_marker` per-tool one-liners (already in code); single trigger fires earlier and more often, each pass with higher savings ratio.
8. Per-pass economics improve. ~60% savings per typical pass at the new defaults (vs. ~46% today) means the 10% thrash bar is trivially clearable in normal operation; gate trips primarily when the conversation is genuinely stuck, not from drift.
