# Phase 3.5 — Write-capable delegated children + child→parent approval propagation

**Parent milestone:** `2026-06-24-234633-loop-decoupling-milestone.md` (Phase 3.5). **Design:** `2026-06-24-234633-loop-decoupling-design.md` §6.12. **Depends on:** Phase 3 (inline approval, completed) + Phase 2.5 (read-mostly delegation, completed).

This phase reaches the Option-B end state: the delegated child gains **write-capable tools** (shell, file-write) and any approval-required call inside the child **surfaces on the parent's terminal** via Phase 3's inline-approval mechanism — the child asks the user, never bypasses the gate. It is **additive capability #2 of the milestone** (the other is 2.5); owned-path-only, so the cutover parity gate on existing flows is undisturbed.

## Context

Phases 1, 2, 2.5, 3 are shipped (all in `completed/`). The two pieces this phase bridges already exist:

- **Inline approval collector (Phase 3):** `collect_inline_approvals(tool_calls, deps, frontend) -> ApprovalResolution` (`co_cli/agent/approval.py:132`), run pre-fan-out in the orchestrator step loop (`loop.py:300`). It detects both triggers — catalog `is_approval_required=True` (read off the **global** `deps.tool_catalog`, `approval.py:157`) and shell's dynamic `evaluate_shell_command` gate (`approval.py:76-90`) — prompts via `frontend.prompt_approval` (frontend `None` → `"n"`), honors auto-approval + remember-choice, and returns `denials` (denied calls → `ToolReturnPart`) + `approved_ids` (the in-body raisers' gate). Headless (`frontend is None`) auto-denies.
- **Read-mostly delegation (Phase 2.5):** `delegate` tool (`co_cli/tools/system/delegate.py`) → `delegate_to_child(parent_deps, task)` (`co_cli/agent/delegation.py:82`) → `fork_deps(parent_deps, share_dispatch_sem=False)` → `run_standalone_owned(DELEGATE_CHILD_SPEC, child_deps, task, settings=parent_deps.model.settings)` (`loop.py:398`). The child runs a read-mostly surface (`_CHILD_TOOL_NAMES`, `delegation.py:47` — all `is_approval_required=False`, `delegate` excluded) and returns only `result.summary`; the child's transcript never enters parent history (the context-isolation contract).

**The two gaps that make the child read-only today:**

1. **The child surface excludes every write/approval tool.** `_CHILD_TOOL_NAMES` (`delegation.py:47-59`) is read/search/web/memory/session/todo only. `shell_exec` (`tools/shell/execute.py:20`, dynamic gate), `file_write` and `file_patch` (`tools/files/write.py:269,324`, both catalog `is_approval_required=True`) are absent.

2. **The subagent driver never prompts.** `run_standalone_owned`'s only dispatch call is `dispatch_tools(calls, deps, cap_state=cap_state, frontend=None)` (`loop.py:495`) — no collector runs, and `frontend=None` means a catalog-gated call would (if present) hit no gate at all on the owned path (`FunctionToolset.call_tool` has no approval check — Phase-3 finding), while shell's in-body `raise ApprovalRequired` (`execute.py:80`, fires when `not ctx.tool_call_approved`) would crash the child turn. So a write-capable child today would either act **unprompted** (catalog tools) or **crash** (shell). Subagents are deliberately approval-free because daemons run them headless.

**The frontend is not on `deps`.** Phase 3 keeps `frontend` an explicit parameter (`run_turn_owned(frontend=...)` → `dispatch_tools(frontend=...)` → `collect_inline_approvals(..., frontend)`); it is **not** stored on `deps` or `deps.runtime`. A tool body receives only `ctx` (`RunContext`), whose `ctx.deps` carries no frontend. So `delegate` (running inside `dispatch_tools`) has **no channel to the parent's prompt surface** — the missing seam this phase must add. `deps.runtime` already holds the analogous per-turn `status_callback` (`deps.py:205`, set in `run_turn_owned` at `loop.py:188`, cleared by `reset_for_turn` at `loop.py:186`).

**`fork_deps` resets runtime and copies approval rules (`deps.py:456,434`):** the child gets `CoRuntimeState(agent_depth=parent+1)` (so it will **not** inherit a `runtime.frontend` — the frontend must be passed explicitly into `run_standalone_owned`), and `session_approval_rules=list(base...)` (a **copy** — see Open Questions on remember-choice scope). `tool_catalog` is shared by reference (`deps.py:453`), so the child collector reads the same global catalog the orchestrator does.

**Daemon path must stay byte-for-byte unchanged.** `run_standalone` (`run.py:50`) calls `run_standalone_owned(spec, deps, prompt)` with no frontend; the dream-reviewer and other daemons run headless and approval-free. Any change to `run_standalone_owned` must default to today's behavior when no frontend is passed.

## Problem & Outcome

**Problem:** the delegated child can only read. A long-running small-model loop that wants to isolate a subtask's context (the 2.5 win) cannot delegate any subtask that *acts* — write a file, run a command — because the child has no write tools and the driver has no way to ask the user before a sensitive action. So delegation is half a capability: context isolation for gather, but not for do.

**Outcome:** the orchestrator can delegate a subtask that **acts**. The child runs shell / file-write in its isolated context; when a child tool call needs approval (catalog-gated or shell's dynamic gate), the prompt surfaces **inline on the parent's terminal** via Phase 3's collector; approve → the child proceeds, deny → the denial is fed back to the child (which adapts and summarizes), headless → auto-deny. The child's transcript still never enters parent history (context isolation preserved); only the distilled summary returns. The user's trust boundary holds across the delegation seam — the child never bypasses the gate.

**Failure cost:** without it, write subtasks cannot be isolated — the parent must do all acting inline, so any multi-step act-and-verify subtask (edit N files, run the build, fix, re-run) pollutes the parent's context with the full tool transcript, exactly the window pressure 2.5 was built to relieve. The trust boundary is the harder cost: a naïve "just give the child write tools" without propagation would let a child act **unprompted** (catalog tools hit no owned-path gate) or **crash** (shell) — a silent breach of the user's approval gate. This phase exists to make write-capable delegation *safe*, not just possible.

## Scope

**In:**
- A **frontend-reach seam**: `deps.runtime.frontend` (per-turn, set in `run_turn_owned`, cleared by `reset_for_turn`) so the `delegate` tool body can reach the parent's prompt surface — the justified `runtime` carve-out (like `clarify_answers` / `revealed_tools`).
- **Approval propagation in `run_standalone_owned`**: a `propagate_approvals` flag + a `frontend` param; when propagating (the delegated path), run `collect_inline_approvals` before each dispatch and feed `denials`/`approved_ids` into `dispatch_tools` — even when `frontend is None` (headless parent → auto-deny, G1-1). When not propagating (daemon), unchanged.
- **Widening the delegated child surface** to write-capable tools: `shell_exec`, `file_write`, `file_patch`. Single `DELEGATE_CHILD_SPEC` becomes full-power; `delegate` tool docstring + child instructions updated.
- `delegate_to_child` reads `parent_deps.runtime.frontend` and passes it through.
- A real-Ollama owned-path flow test (the gate): child write surfaces approval on the parent terminal; deny-blocks-side-effect + approve-executes + headless-auto-denies + context isolation preserved.

**Out:**
- **Display of the child's routine tool activity on the parent terminal** — the child's normal tool calls stay silent (opaque-delegation UX from 2.5 preserved); **only approval prompts surface** (D-3). `dispatch_tools` is still called `frontend=None` inside the child; the collector alone gets the real frontend.
- **Broader-trust child tools** beyond shell + file write/patch — `gmail` send, `memory_manage`, `system/skills` manage, `user_profile` write, `tasks` control (the other catalog `is_approval_required=True` tools) are deliberately excluded; they are not "do the work" primitives and widen the trust surface. Re-open as scope if a need pulls them.
- **Cross-scope remember-choice propagation** — `"a"` inside a child writes the child's *copied* approval rules, not the parent session's (Open Questions). Documented limitation, not fixed here.
- **Nested / multi-level delegation** — depth cap stays 1 (`delegate` absent from the child surface; `agent_depth` backstop). Unchanged from 2.5.
- **The graph path** — delegation is owned-path-only.
- **Deleting any deferred/graph machinery** — Phase 5 owns deletion.

## Behavioral Constraints

- **The child never bypasses the approval gate.** Every approval-required child call (catalog `is_approval_required=True` OR shell's dynamic `REQUIRE_APPROVAL`) prompts on the parent's frontend before executing. A delegated child with write tools but no frontend (headless parent) **auto-denies** — never acts unprompted — because the collector still runs on the delegated path (`propagate_approvals=True`) and auto-denies when `frontend is None` (G1-1). This holds for `file_write`/`file_patch` too, which have no in-body raise and are gated solely by the collector.
- **Context isolation preserved (the 2.5 contract).** The child's intermediate tool calls/results do not enter the parent's history — only the `delegate` summary. Widening the surface does not change this.
- **Deny = continue, not hard-halt (parity with Phase 3 D-A).** A denied child call becomes a denial `ToolReturnPart` fed back into the **child's** history; the child loop continues and (typically) calls `final_result` summarizing what it could/could not do. The milestone's "deny → child halts and returns that to the parent" is satisfied in effect: the parent receives a summary reflecting the denial. No special child-halt branch — reuse dispatch's denial mechanism.
- **Daemon path byte-for-byte unchanged.** `run_standalone` passes `propagate_approvals=False` (the default) → no collector runs → identical to today. Verified by the existing daemon/subagent regression tests.
- **Additive only.** No existing eval calls `delegate`; owned-vs-graph parity on existing flows is unchanged. The graph path is untouched. Existing approval evals (`eval_approval_discipline`, `eval_bounded_autonomy`) remain green (the orchestrator collector is unchanged).
- **Interrupt propagation unchanged.** A parent interrupt cancels the in-flight child (`CancelledError` propagates through `delegate_to_child` → `run_standalone_owned`) — already true from 2.5.

## High-Level Design

```
parent orchestrator step (owned loop)
  parent collector runs pre-fan-out for the PARENT's calls   # delegate itself is is_approval_required=False → not prompted
  dispatch_tools: concurrent batch, THEN sequential (delegate is is_concurrent_safe=False → runs alone)
    delegate(ctx, task)                                       # ctx.deps == parent_deps
      → delegate_to_child(parent_deps, task)
          frontend = parent_deps.runtime.frontend             # ← the new seam (set by run_turn_owned)
          depth guard (agent_depth >= 1) → refuse
          child_deps = fork_deps(parent_deps, share_dispatch_sem=False)   # own sem; agent_depth+1
          run_standalone_owned(DELEGATE_CHILD_SPEC, child_deps, task,
                               settings=parent_deps.model.settings,
                               propagate_approvals=True, frontend=frontend)   # frontend may be None (headless parent)
            ── child step loop ──
              calls = tool calls (shell_exec / file_write / file_patch / reads …)
              if propagate_approvals:                         # delegated child → gate every step
                  resolution = collect_inline_approvals(calls, child_deps, frontend, origin_label=…)
                  # frontend set → prompts on PARENT terminal; frontend None → auto-denies (never acts unprompted)
              else:                                           # daemon (run_standalone) → no collector, unchanged
                  resolution = empty
              dispatch_tools(calls, child_deps, cap_state=…, frontend=None,   # child tool DISPLAY stays silent
                             denials=resolution.denials, approved_ids=resolution.approved_ids)
              # approved write executes; denied write → denial part fed back → child adapts → final_result
          return result.summary                               # the ONLY thing entering parent history
```

**Why `runtime.frontend` is the seam (D-1).** The `delegate` tool body has only `ctx`; the parent's frontend is reachable only through `ctx.deps`. Phase 3 deliberately keeps `frontend` off `deps` for the orchestrator (explicit param), but a *tool body* cannot receive an explicit frontend param through the tool protocol. `deps.runtime` is the per-turn scratch space that already carries the analogous `status_callback`; adding `frontend` there is the same "explicitly justified tool reach-in" the `CoRuntimeState` docstring (`deps.py:174`) anticipates for `clarify_answers` / `revealed_tools`. Set it in `run_turn_owned` next to `status_callback`, clear it in `reset_for_turn`. Because `fork_deps` resets runtime, the child does **not** inherit it through `child_deps.runtime` — the frontend is threaded **explicitly** into `run_standalone_owned`, keeping the propagation path one-directional and auditable.

**Why dispatch display stays silent but the collector gets the real frontend (D-3).** Context isolation is a **history** contract (the parent model's window), not a **display** contract. The 2.5 child is silent on the terminal (the user sees one `delegate` step, then its summary). Surfacing *every* child tool call would re-introduce the terminal spam 2.5 removed, for the common read/gather delegation. But the **trust boundary** must surface: an approval prompt cannot be silent. So the child loop passes the real frontend **only to the collector** (which prompts) and keeps `frontend=None` on `dispatch_tools` (no `on_tool_start`/`on_tool_complete` rendering). The approval subject (the command / file path) gives the user enough to decide; the routine activity stays opaque. Net: the user sees the `delegate` step running, then — only if the child hits a gate — an inline approval prompt for the concrete action, then the summary.

**Why no contention on the single prompt future.** `delegate` is `is_concurrent_safe=False`, so dispatch runs it in the **sequential** phase, after the concurrent batch (`dispatch.py:301-306`) — no sibling tool is mid-execution when the child prompts. The parent's own collector already ran (pre-fan-out) and finished. So the frontend's single `_prompt_future` (`display/core.py`) is free when the child's collector prompts. The strictly-sequential collector (`approval.py` no-`gather` invariant) holds inside the child too.

**Why widen one spec, not add a second tool.** The design (§6.12) specifies one `delegate` tool reaching Option-B full power. Widening `DELEGATE_CHILD_SPEC` keeps a single delegation capability; the model chooses whether a subtask needs writes, and the approval gate protects the user. The read/gather use (2.5) still works — the child simply doesn't call write tools. The child instructions gain a line on its write capability; the `delegate` docstring drops "It cannot write".

**Daemon parity (and why it is NOT keyed on `frontend is None` — G1-1).** The daemon path is `run_standalone` (`run.py`), which passes `propagate_approvals=False` (the default) → the collector is skipped entirely → `run_standalone_owned` calls `dispatch_tools(frontend=None)` exactly as today. The discriminator is **`propagate_approvals`, not `frontend is None`**, because the delegated child can *also* have `frontend is None` (a headless parent) yet must still gate: keying on the frontend value would route that headless-delegated child into the daemon skip-branch and let its `file_write` execute unprompted (the catalog gate lives only in the collector — `dispatch.py:69-71`). On the delegated path the collector therefore **always runs**; with `frontend is None` it auto-denies (`approval.py:169`). The daemon child surfaces (dream reviewer etc.) carry no approval-required tools regardless, so the daemon skip is doubly safe — but the delegated path no longer *relies* on that non-local invariant.

**Known limitation (re-raise trigger):** remember-choice (`"a"`) inside a child calls `remember_tool_approval(subject, child_deps)`, which appends to the child's **copied** `session_approval_rules` (`fork_deps` copies the list, `deps.py:434`) — it does **not** persist to the parent session. So "always allow" chosen inside a delegated child holds for the rest of that child run but not across delegations or back in the parent. **This is a deliberate default, not a cost concession (PO-m-2): a child's approval scope is subtask-local by design** — a blanket "always allow shell rm" earned inside one bounded delegated subtask leaking back to the parent session is itself a trust-boundary question, not obviously desirable. **Re-raise** if users hit repeated prompts for the same subject across delegations — the fix is a scoped decision about sharing approval rules into in-turn (not daemon) forks.
