# Skill Port — `documents` (PDF extraction over `shell_exec`)

Task type: code + skill content

> **Built on the folder-skill model** — every skill is a `<name>/SKILL.md` folder that can
> carry bundled assets. Shipped in **v0.8.332**
> (`docs/exec-plans/completed/2026-06-09-171515-skill-folder-model.md`). This plan adds
> `documents/SKILL.md` + `documents/scripts/extract_pdf.py` on top of it.

## Context

The most critical of co's three real functional tooling gaps vs hermes
(`docs/reference/RESEARCH-tools-gaps-co-vs-hermes.md` §2.1, §5): co cannot read a local
PDF — `file_read` rejects binary. This is squarely on-mission ("Personal Assistant CLI"):
PDFs are the most common local artifact a user hands an assistant. It is also the
cheapest of the three to close (a lean dep + a thin script + a skill — no new tool, no
model-multimodal plumbing). This plan was split out of the retired
`2026-05-27-133104-skill-porting-mission-converged.md`; the other two gaps (vision,
interactive terminal) have their own plans.

**Why a skill + script, not a tool.** The primary reason is that it adds **zero new
model-visible surface and zero `categories.py`/`toolset.py` churn** — it rides the existing
ALWAYS `shell_exec` + bundled-skill rails. The secondary benefit is process isolation:
extraction runs in a **subprocess** via `shell_exec`, so a memory spike or crash in pymupdf
on a large/malformed PDF cannot touch the agent heap (in-process `asyncio.to_thread` shares
it; a subprocess does not). A dedicated `document_extract` tool was rejected on both counts.

**Where the script lives — inside the skill folder, never `scripts/`.** `scripts/` at the
repo root is build-time-only dev tooling (quality-gate, audits, validators) and hatchling
ships **only `co_cli/`** in the wheel, so a bundled skill referencing `scripts/extract_pdf.py`
would find nothing on an installed system. With the folder-skill model (the prerequisite
plan), the helper ships **inside the skill's own folder** at
`co_cli/skills/documents/scripts/extract_pdf.py` and is invoked as a **module** —
`uv run python -m co_cli.skills.documents.scripts.extract_pdf <pdf>` — which resolves via
`sys.path` regardless of the agent's `work_dir` or the install location. It is still reached
**only** through `shell_exec` (never imported into the agent process), so the
subprocess-isolation boundary is unchanged. `documents/` and `documents/scripts/` become
packages (docstring-only `__init__.py`) so the `-m` invocation resolves.

## Problem & Outcome

**Problem.** co is blind to local PDFs. "Summarize this PDF" is an unconditional "I can't,"
even though the capability is blocked only by a missing dependency + a thin script.

**Failure cost:** the assistant silently cannot serve its single most common local-document
request; the user must extract text by hand outside co and paste it back — defeating the
"personal assistant over your files" purpose. No error teaches them a workaround exists.

**Outcome.**
1. `pymupdf` + `pymupdf4llm` added as lean deps (no PyTorch/marker-pdf).
2. `co_cli/skills/documents/scripts/extract_pdf.py` — subprocess CLI module: PDF → markdown
   with page-break markers, plain stderr + non-zero exit on errors.
3. `co_cli/skills/documents/SKILL.md` — locate → extract → answer, PDF-only v1, via `shell_exec`.
4. Real-PDF test + committed fixture; bundled gate green (`documents` in `_BUNDLED_NAMES`).
5. Ships once it passes lint + the bundled gate + the new test (folder-skill model already in code).

## Behavioral Constraints
1. **Substrate gate.** Body references only `file_search`, `web_fetch`, `file_read`,
   `shell_exec` (+ the bundled `documents/scripts/extract_pdf.py` via `-m`). No `file_find`.
2. **Marginal-value gate.** Multi-step locate → extract → answer with a decision tree.
3. **Native body.** co tool names only; no `terminal`/`extract_pymupdf.py`/`web_extract`.
4. **Process isolation (hard).** Extraction runs via `shell_exec` subprocess (`python -m …`),
   never imported into the agent process.
5. **Approval transparency.** First extraction prompts approval (`python` not in safe_commands).
   The opt-in `shell.safe_commands += "uv run python -m co_cli.skills.documents.scripts.extract_pdf"`
   only auto-approves the module command — `shell_policy.py`'s `_FORBIDDEN_ARG_PATTERNS` rejects any
   arg containing `/`, `~`, or `..`, so a PDF anywhere but the workspace root still prompts by
   design (path-traversal guard, not a bug). The realistic ergonomic win is session-subject
   auto-approval: once approved this session, the same subject is not re-prompted
   (`co_cli/context/orchestrate.py`). The body must describe this accurately and not promise the
   opt-in eliminates the prompt for typical paths.
6. **Graceful failure.** The script exits non-zero with a plain stderr message (no traceback)
   for missing file / non-PDF / corrupt PDF; the **encrypted/password-protected** case gets its
   own distinct message ("PDF is password-protected") so the user learns the recoverable cause,
   per the Failure-cost thesis.
7. **Lean dependency.** `pymupdf` + `pymupdf4llm` only; no PyTorch/marker-pdf pulled.
8. **Read-only synthesis.** No write tools.
9. **Bundled, test-gated.** Lands in `co_cli/skills/documents/SKILL.md`; passes lint
   R1–R4 + B1 + security scan; registered in `_BUNDLED_NAMES`.

## High-Level Design

Three layers, bottom-up:

1. **Dependency layer** — `pymupdf` (the C-backed engine, ~25 MB) + `pymupdf4llm` (markdown
   adapter that emits page-break markers). Added to `[project].dependencies`; `uv sync`
   regenerates `uv.lock`. The lean-dep constraint is verifiable: marker-pdf/PyTorch must
   not appear in the resolved tree.

2. **Extraction layer** — `co_cli/skills/documents/scripts/extract_pdf.py`, a stdlib-`argparse`
   CLI invoked as a module: `uv run python -m co_cli.skills.documents.scripts.extract_pdf
   <path> [--pages RANGE]`. Module invocation resolves via `sys.path` from any `work_dir` and
   on an installed wheel — unlike a repo-relative file path. `--pages RANGE` mirrors hermes's
   `extract_pymupdf.py` shape (`0-4`, `2`, `0-1,3`); the script **parses it into an explicit
   0-based list** before calling pymupdf4llm — `to_markdown(pages=…)` takes a list of 0-based
   page numbers, **not** a range string. To guarantee the citation contract regardless of
   library internals, the script calls `to_markdown(path, pages=<list>, page_chunks=True)`
   (which returns per-page chunks) and emits its own explicit `## Page N` marker before each
   page, then prints the concatenated markdown to stdout. On any failure it prints a one-line
   message to stderr and exits non-zero (no traceback). It ships **inside the skill folder**
   (the folder-skill model makes this shippable) and is **never imported into the agent
   process** — the agent reaches it only as a `shell_exec` subprocess, the isolation boundary.
   `documents/` and `documents/scripts/` carry docstring-only `__init__.py` so `-m` resolves.

3. **Skill layer** — `co_cli/skills/documents/SKILL.md`, a prompt overlay with a Locate → Extract
   → Answer decision tree. Locate uses `file_search` or a user-supplied path; if Google is
   configured, `google_drive_search` is an *optional* additional route (it is DEFERRED +
   config-gated, invisible when Google is unconfigured), so the body degrades to
   `file_search` + user path as the primary, no-Google path. Extract branches (URL →
   `web_fetch`; local `.pdf` → `shell_exec` driving the script; local `.docx`/`.pptx`/`.xlsx`
   → defer to the `office` skill; other local → `file_read`); Answer synthesizes with
   **page N** citations from the script's `## Page N` markers. The
   skill is registered as bundled by adding `documents` to `_BUNDLED_NAMES` in the
   bundled-library test, which is what makes the gate enforce it.

The approval prompt on first extraction is a feature, not friction: it surfaces that co is
about to run Python on the user's file. Repeat use within a session is auto-approved by
subject; the body explains this honestly (see Constraint 5) rather than overselling the
safe_commands opt-in.
