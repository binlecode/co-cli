## Execution order (document-stack plans)

dependency edges (verified against each plan's prerequisites):

1. **`skill-documents`** (this plan) ‚Äî creates `extract_pdf.py`, the `co-extract-pdf` entry
   point, and `documents/SKILL.md`. **No prerequisites.** Everything else extends these.
2. **`skill-office`** (`2026-06-10-114024-skill-office.md`) ‚Äî **after** this plan. Reciprocal
   routing: `office`'s body routes `.pdf` ‚Üí `documents`, so `documents` must exist first; once
   `office` lands, the Office-format branch here can hard-route to it (until then this plan
   degrades honestly ‚Äî see High-Level Design layer 3). Independent of the vision/scanned track.
3. **`vision-input`** (`2026-05-28-150239-vision-input.md`) ‚Äî provides `image_view`.
   Independent of the document plans; can ship anytime, but is a **hard prerequisite of #4**.
4. **`scanned-pdf-vision`** (`2026-06-10-144745-scanned-pdf-vision.md`) ‚Äî tier 2, ships
   **last**. Explicitly gated on **both** this plan (it adds `--render` to `extract_pdf.py` and
   keys off Constraint 7's scanned signal) **and** `vision-input` (`image_view`). It also edits
   `documents/SKILL.md`.

**Serialization note:** plans #2 and #4 both edit `documents/SKILL.md` (different branches of
the same decision tree ‚Äî Office routing vs the scanned branch). Sequence them

## Context

The most critical of co's three real functional tooling gaps vs hermes
(`docs/reference/RESEARCH-tools-gaps-co-vs-hermes.md` ¬ß2.1, ¬ß5): co cannot read a local
PDF ‚Äî `file_read` rejects binary. This is squarely on-mission ("Personal Assistant CLI"):
PDFs are the most common local artifact a user hands an assistant. It is also the
cheapest of the three to close (a lean dep + a thin script + a skill ‚Äî no new tool, no
model-multimodal plumbing). This plan was split out of the retired
`2026-05-27-133104-skill-porting-mission-converged.md`; the other two gaps (vision,
interactive terminal) have their own plans.

**Why a skill + script, not a tool.** The primary reason is that it adds **zero new
model-visible surface and zero `categories.py`/`toolset.py` churn** ‚Äî it rides the existing
ALWAYS `shell_exec` + bundled-skill rails. The secondary benefit is process isolation:
extraction runs in a **subprocess** via `shell_exec`, so a memory spike or crash in pymupdf
on a large/malformed PDF cannot touch the agent heap (in-process `asyncio.to_thread` shares
it; a subprocess does not). A dedicated `document_extract` tool was rejected on both counts.

**Where the script lives ‚Äî inside the skill folder, never `scripts/`.** `scripts/` at the
repo root is build-time-only dev tooling (quality-gate, audits, validators) and hatchling
ships **only `co_cli/`** in the wheel, so a bundled skill referencing `scripts/extract_pdf.py`
would find nothing on an installed system. With the folder-skill model (the prerequisite
plan), the helper ships **inside the skill's own folder** at
`co_cli/skills/documents/scripts/extract_pdf.py`. It is still reached **only** through
`shell_exec` (never imported into the agent process), so the subprocess-isolation boundary
is unchanged. `documents/` and `documents/scripts/` become packages (docstring-only
`__init__.py`) so the module path import-resolves.

**How the agent invokes it ‚Äî a console entry point, not `uv run python -m`.** The agent
reaches the script through a `[project.scripts]` console entry point
`co-extract-pdf = "co_cli.skills.documents.scripts.extract_pdf:main"`, declared in
`pyproject.toml` exactly as `co` itself is (`pyproject.toml:29-30`). The skill body calls the
bare command `co-extract-pdf <path>`. **Why not `uv run python -m ‚Ä¶`** ‚Äî code-grounded in the
spawn path (verified 2026-06-10): `shell_exec` ‚Üí `shell_backend.run_command` ‚Üí
`asyncio.create_subprocess_exec` runs with a **sanitized allowlist env** (`shell_env.py:12-23`)
that propagates `PATH` but **not** `VIRTUAL_ENV` or `PYTHONPATH`, and the subprocess `cwd` is
the user's `workspace_dir` (`execute.py:78,85`), **not** co's repo. Under that env, `uv run`
resolves its environment from `cwd` ‚Äî i.e. the *user's* project, which has no `co-cli`
dependency (and on an installed wheel, no project env at all) ‚Üí `ModuleNotFoundError`. It only
works in the one narrow case of a developer sitting inside the co checkout ‚Äî the very
install-location dependency the in-folder placement was chosen to escape. Plain `python -m ‚Ä¶`
fails too: a `uv tool install` puts only entry-point scripts on `PATH`, not the tool venv's
`python`. A `[project.scripts]` entry point sidesteps all of this ‚Äî the installer (uv sync /
uv tool install / pipx / pip) writes a wrapper into the same `bin/` as `co`, hard-coding the
correct interpreter; it depends on nothing but `PATH` containing that `bin/`, which is
guaranteed because the user launched `co` from there. Works identically in dev checkout and
installed wheel. The `__init__.py` files are still required (the `:main` module path must
import-resolve); `extract_pdf.py` exposes a `main()` function as the entry-point target.

## Problem & Outcome

**Problem.** co is blind to local PDFs. "Summarize this PDF" is an unconditional "I can't,"
even though the capability is blocked only by a missing dependency + a thin script.

**Failure cost:** the assistant silently cannot serve its single most common local-document
request; the user must extract text by hand outside co and paste it back ‚Äî defeating the
"personal assistant over your files" purpose. No error teaches them a workaround exists.

**Outcome.**
1. `pymupdf4llm` added as a lean dep (pulls `pymupdf==1.27.2.3` transitively); `pymupdf`
   declared explicitly (version-unconstrained) because the script imports it directly. No
   PyTorch/marker-pdf.
2. `co_cli/skills/documents/scripts/extract_pdf.py` ‚Äî subprocess CLI with a `main()` entry
   point: PDF ‚Üí markdown with page-break markers, plain stderr + non-zero exit on errors.
   Output goes through `sys.stdout.write`/`sys.stderr.write` (not `print`) ‚Äî the script is
   under `co_cli/` and ruff **T20** forbids `print` there (Code Accuracy table). Exposed to the
   agent as the `co-extract-pdf` console entry point (`[project.scripts]`), invoked via
   `shell_exec`.
3. `co_cli/skills/documents/SKILL.md` ‚Äî locate ‚Üí extract ‚Üí answer, PDF-only v1, via `shell_exec`.
4. Real-PDF test + committed fixture; bundled gate green (`documents` in `_BUNDLED_NAMES`).
5. Ships once it passes lint + the bundled gate + the new test (folder-skill model already in code).

## Behavioral Constraints
1. **Substrate gate.** Body references only `file_search`, `web_fetch`, `file_read`,
   `shell_exec` (+ the bundled extractor via the `co-extract-pdf` console entry point). No
   `file_find`, no `uv run python -m`.
2. **Marginal-value gate.** Multi-step locate ‚Üí extract ‚Üí answer with a decision tree.
3. **Native body.** co tool names only; no `terminal`/`extract_pymupdf.py`/`web_extract`.
4. **Process isolation (hard).** Extraction runs via `shell_exec` subprocess (the
   `co-extract-pdf` console entry point), never imported into the agent process.
5. **Approval transparency.** First extraction prompts approval (`co-extract-pdf` not in
   safe_commands). The opt-in `shell.safe_commands += "co-extract-pdf"` only auto-approves the
   bare command ‚Äî `shell_policy.py`'s `_FORBIDDEN_ARG_PATTERNS` rejects any arg containing `/`,
   `~`, or `..` (verified `shell_policy.py:11-27`), so a PDF anywhere but the workspace root
   still prompts by design (path-traversal guard, not a bug). The realistic ergonomic win is
   session-subject auto-approval: once approved this session, the same subject is not
   re-prompted (`co_cli/context/orchestrate.py`). The body must describe this accurately and
   not promise the opt-in eliminates the prompt for typical paths.
6. **Graceful failure.** The script exits non-zero with a plain stderr message (no traceback)
   for missing file / non-PDF / corrupt PDF; the **encrypted/password-protected** case gets its
   own distinct message ("PDF is password-protected") so the user learns the recoverable cause,
   per the Failure-cost thesis.
7. **Scanned-PDF detection seam (handoff, not OCR).** When extraction yields empty/near-empty
   text (an image-only / scanned PDF), the script emits a **distinct, machine-detectable signal**
   ‚Äî a dedicated exit code (or a single sentinel marker line, not blank stdout) meaning *"no
   extractable text layer ‚Äî likely scanned."* This is the seam the future **tier-2** path
   (render pages ‚Üí `image_view`, the vision plan's consumer interface) keys off; here it only
   surfaces the condition so the skill body can route to that path or suggest `web_fetch`,
   **never silently answer from a blank extraction.** The threshold (e.g. chars-per-page floor)
   is the script's call; no OCR/rendering happens in this plan.
8. **Lean dependency.** `pymupdf` + `pymupdf4llm` only; no PyTorch/marker-pdf pulled.
9. **Read-only synthesis.** No write tools.
10. **Bundled, test-gated.** Lands in `co_cli/skills/documents/SKILL.md`; passes skill-lint
   R1‚ÄìR4 + B1 + security scan; registered in `_BUNDLED_NAMES`.
11. **Lint-clean script (hard).** `extract_pdf.py` passes `scripts/quality-gate.sh lint`
   (ruff over `co_cli/`). It is **not** in any per-file-ignore, so it must use
   `sys.stdout.write`/`sys.stderr.write` rather than `print` (ruff **T20**), and otherwise
   satisfy the same ruleset as the rest of `co_cli/`.

## High-Level Design

Three layers, bottom-up:

1. **Dependency layer** ‚Äî `pymupdf4llm` (the markdown adapter that emits page chunks) +
   `pymupdf` (the C-backed engine). **Why both, when pymupdf4llm alone would install pymupdf?**
   pymupdf4llm pins `pymupdf==1.27.2.3` exactly as an install dep (verified on PyPI), so pymupdf
   lands transitively regardless. We still declare `pymupdf` **explicitly** because the script
   *imports it directly* ‚Äî `pymupdf.open(path).needs_pass` is how Constraint 6's distinct
   "password-protected" message is produced (pymupdf4llm exposes no encryption check), and
   `doc.page_count` validates `--pages`. Convention: declare what you import; relying on a
   transitive dep for a direct import is fragile (a future pymupdf4llm could drop/rename it).
   **Caveat:** declare `pymupdf` **without a version constraint** so it defers to pymupdf4llm's
   exact pin ‚Äî a conflicting co pin would break resolution. If the final script ends up *not*
   importing pymupdf directly, drop it and depend on pymupdf4llm alone. Either way the lean
   constraint is verifiable: marker-pdf/PyTorch must not appear in the resolved tree.

2. **Extraction layer** ‚Äî `co_cli/skills/documents/scripts/extract_pdf.py`, a stdlib-`argparse`
   CLI with a `main()` entry point, exposed to the agent as the `co-extract-pdf` console
   command and invoked `co-extract-pdf <path> [--pages RANGE]`. A `[project.scripts]` entry
   point resolves via `PATH` from any `work_dir` and on an installed wheel ‚Äî unlike `uv run
   python -m` (which resolves the env from the user's `cwd`, where `co_cli` is not installed)
   or a repo-relative file path. See the Context section for the full spawn-env rationale
   (`shell_env.py` allowlist: `PATH` yes, `VIRTUAL_ENV`/`PYTHONPATH` no). `--pages RANGE`
   mirrors hermes's `extract_pymupdf.py` shape (`0-4`, `2`, `0-1,3`); the script **parses it
   into an explicit 0-based list** before calling pymupdf4llm ‚Äî `to_markdown(pages=‚Ä¶)` takes a
   list of 0-based page numbers, **not** a range string. To guarantee the citation contract
   regardless of library internals, the script calls `to_markdown(path, pages=<list>,
   page_chunks=True)` (which returns per-page chunks) and emits its own explicit `## Page N`
   marker before each page, then writes the concatenated markdown to stdout via
   `sys.stdout.write` (not `print` ‚Äî ruff T20). On any failure it writes a one-line message to
   stderr and exits non-zero (no traceback). It ships **inside the skill folder** (the
   folder-skill model makes this shippable) and is **never imported into the agent process** ‚Äî
   the agent reaches it only as a `shell_exec` subprocess, the isolation boundary. `documents/`
   and `documents/scripts/` carry docstring-only `__init__.py` so the `:main` module path
   import-resolves.

3. **Skill layer** ‚Äî `co_cli/skills/documents/SKILL.md`, a prompt overlay with a Locate ‚Üí Extract
   ‚Üí Answer decision tree. Locate uses `file_search` or a user-supplied path; if Google is
   configured, `google_drive_search` is an *optional* additional route (it is DEFERRED +
   config-gated, invisible when Google is unconfigured), so the body degrades to
   `file_search` + user path as the primary, no-Google path. Extract branches (URL ‚Üí
   `web_fetch`; local `.pdf` ‚Üí `shell_exec` driving the script; local `.docx`/`.pptx`/`.xlsx`
   ‚Üí defer to the `office` skill; other local ‚Üí `file_read`); the local `.pdf` branch calls
   `co-extract-pdf <path>` via `shell_exec`. **Office-format degradation:** the `office` skill
   ships on its own active plan (`2026-06-10-114024-skill-office.md`) and may not be loaded when
   `documents` lands first (see Execution order). The body must degrade honestly for
   `.docx`/`.pptx`/`.xlsx` ‚Äî name the format and that Office support is the `office` skill's job
   ‚Äî rather than silently dead-end or hand off to a skill the loader can't find. Answer
   synthesizes with **page N** citations from the script's `## Page N` markers. The skill is
   registered as bundled by adding `documents` to `_BUNDLED_NAMES` in the bundled-library test,
   which is what makes the gate enforce it.

The approval prompt on first extraction is a feature, not friction: it surfaces that co is
about to run Python on the user's file. Repeat use within a session is auto-approved by
subject; the body explains this honestly (see Constraint 5) rather than overselling the
safe_commands opt-in.

## Spec sync ‚Äî merge into `docs/specs/skills.md` on delivery

`sync-doc` runs post-delivery (see Out of scope). These are the **specific runtime facts**
this plan establishes that must land in `docs/specs/skills.md` so the bundled-skill-with-assets
model is documented canon, not tribal knowledge:

1. **Skill folders may bundle an executable asset** invoked as a shell subprocess ‚Äî the
   `documents` skill is the first to do so (`co_cli/skills/documents/scripts/extract_pdf.py`).
2. **Bundled skill scripts are exposed via a `[project.scripts]` console entry point**
   (`co-extract-pdf`), never `uv run python -m`. Rationale is a runtime constraint, not a
   preference: `shell_exec` spawns with a **sanitized allowlist env** (`shell_env.py` ‚Äî `PATH`
   propagates, `VIRTUAL_ENV`/`PYTHONPATH` do not) and `cwd` = the user's `workspace_dir`, so
   only a `PATH`-resolved console command resolves `co_cli` reliably on dev checkout *and*
   installed wheel. This is the **convention for any future skill-bundled script.**
3. **Skill-bundled scripts live under `co_cli/` and obey the full `co_cli/` ruff ruleset**
   (notably T20 ‚Äî output via `sys.stdout.write`/`sys.stderr.write`, not `print`), because
   per-file-ignores do not cover `co_cli/skills/*/scripts/`.
4. **Approval semantics for skill-driven shell commands** ‚Äî first run prompts; the
   `shell.safe_commands` opt-in plus the `_FORBIDDEN_ARG_PATTERNS` path-traversal guard means
   only workspace-root bare paths auto-approve; session-subject auto-approval is the real
   ergonomic win (Constraint 5).
