# Capability Enhancement ‚Äî vision input (`image_view`)

Task type: code

## Context

A capability enhancement that extends co's surface in a direction it cannot currently
reach, sourced from the cross-review against hermes-agent
(`docs/reference/RESEARCH-tools-gaps-co-vs-hermes.md` ¬ß2.1, ¬ß5). It is **additive** ‚Äî it
opens a new input modality on top of existing, proven plumbing. Vision is the largest raw
capability hole vs hermes (a whole missing modality), though it ranks **Low** on
mission-frequency, below the `documents` (PDF) gap.

- **Vision input**: co has no way to feed pixels to a model. It can read a PDF's text
  (`documents` skill) but cannot look at a screenshot, photo, diagram, or rendered chart.
  Greenfield ‚Äî a repo-wide grep confirms **no `BinaryContent` / `ImageUrl` / multimodal
  content** is constructed anywhere in `co_cli/` today.

> tool, (B) clipboard-paste UX, (C) a bounded image store + `/images` command. **PO review

> commands, input-layer + banner + session-boundary changes) around an already-working
> tool. Phase A *is* the capability; pixels reach the model the moment `image_view` works.
> B and C are deferred to a follow-up plan, gated on shipping A and observing whether

> workstream (`task_write`/`task_close`) had already moved to

### Why it rides proven infrastructure
- **Vision model** reuses the `judge_model` precedent: `LlmSettings` already carries an
  optional pinned secondary model (`judge_model: str | None`, `config/llm.py:232`) that
  inherits provider/host/api_key, and `build_judge_model` clones settings via
  `llm.model_copy(update={"model": ‚Ä¶})` (`llm/factory.py:94`). Vision adds the same shape.
- **Pixels-to-model** rides `pydantic_ai.messages.ToolReturn.content`
  (`str | Sequence[UserContent] | None`, confirmed at `pydantic_ai/messages.py:867` in
  1.81.0; `UserContent`‚Üí`MultiModalContent`‚Üí`BinaryContent`) ‚Äî a tool can *return*
  `BinaryContent` and the model sees real pixels next turn. Same mechanism as hermes's
  "native fast path".
- **Capability probe** rides `probe_ollama_model` (`bootstrap/check.py:68`), which already
  reads `/api/show` at bootstrap; the `vision` capability can be surfaced from the
  already-fetched payload with no new request.

**Probe-confirmed premise (2026-06-02):** the default agentic Ollama model is *itself*
vision-capable ‚Äî `qwen3.6:35b-a3b-agentic` reports `vision` via `/api/show` and a live
`Agent(model).run([prompt, BinaryContent])` returned a correct read of a test PNG. So the
agent-model-is-the-vision-model path (already true for Gemini) applies to Ollama too;
`vision_model` is a **fallback-only escape hatch** for genuinely text-only Ollama models.

### Design alignment with hermes (grounded, not copied)
Studied `hermes-agent/tools/vision_tools.py`. co copies the **shape and mechanism**: one
capability-gated tool that, when the model is vision-capable, attaches pixels via the
tool-result envelope (native path), else returns a text description from an aux model
(fallback). co attaches pixels via the **tool-result channel**, not the user turn ‚Äî
keeping history/compaction text-only (hermes does the same).

## Problem & Outcome

**Problem.** Any image input has nowhere to go ‚Äî `file_read` rejects binary and no path
feeds pixels to a model.

**Failure cost:** the assistant is blind to an entire input modality ‚Äî screenshots, photos,
diagrams, charts. The user cannot ask "what's in this image?" at all, and no error hints at
a workaround because there is no path to one.

**Outcome.** **`image_view(path, prompt)`** ‚Äî reads a local image and, when the resolved
model is vision-capable, returns the **pixels** via `ToolReturn.content` so the agent model
sees them next turn; when only a text-only agent model + a pinned `vision_model` fallback
is available, returns a **text description** instead. Capability-gated: never sends bytes to
a blind model and fakes a result. Replayed history stays text-only (the pixel-bearing
`UserPromptPart` is elided on replay so no base64 accumulates).

## Behavioural Constraints
1. **Honest capability gate.** When no vision-capable model resolves (text-only Ollama
   agent model + no `vision_model`), `image_view` is unavailable (`check_fn` false) with a
   remediation message. On a blind host, `tool_view("image_view")` returns the remediation
   rather than unlocking a tool that never materializes. Never send bytes to a blind model
   and present the result as if it saw the image.
2. **Pixels via tool-result, not user turn.** Real pixels reach the model only through
   `ToolReturn.content`, which pydantic-ai materializes as a separate **`UserPromptPart`**.
   The user turn / history / compaction stay text-only: a new history processor elides
   multimodal `UserContent` from non-tail `UserPromptPart`s on replay so transcripts don't
   carry base64 bloat across turns.
3. **Single request, no agent loop.** One multimodal request per `image_view` (fallback
   path), bounded by the same per-call timeout discipline (`llm/call.py`) as other calls.
4. **Local-file boundary.** `image_view` `path` resolved + read-boundary-confined like
   `file_read` (`fs_guards.enforce_read_boundary`); reject escapes.
5. **Size + media-type guards.** Cap at ~20 MB; accept only `image/{png,jpeg,webp,gif}`;
   reject others clearly (PDFs ‚Üí `documents` skill). *Downstream consumer:* the future
   **scanned-PDF tier-2** path (a separate plan, gated on this one) will render PDF pages to
   local PNGs and feed them **through `image_view`** ‚Äî so PDF rendering lives in tier-2, not
   here, and `image_view` stays image-only. The local-path + PNG contract here is exactly what
   tier-2 builds on; no change needed in this plan.
6. **No new hard dependency** ‚Äî pydantic-ai's existing `BinaryContent` / `ToolReturn`.

## High-Level Design

### A1 ‚Äî `build_vision_model` (factory) + capability detection
Resolution order ‚Äî *use the agent model when it can see; pin a fallback only when it can't*:
```python
def build_vision_model(llm: LlmSettings, agent_vision_capable: bool) -> LlmModel | None:
    if llm.uses_gemini():
        return build_model(llm)                                   # natively multimodal
    if llm.uses_ollama() and agent_vision_capable:
        return build_model(llm)                                   # agent model sees pixels
    if llm.uses_ollama() and llm.vision_model:
        return build_model(llm.model_copy(update={"model": llm.vision_model}))  # fallback pin
    return None                                                   # text-only Ollama, no fallback
```
Note `model_copy` (pydantic), **not** `dataclasses.replace` ‚Äî matches `build_judge_model`
(`factory.py:94`). **Detection (no new IO):** extend `probe_ollama_model` to also surface
the `vision` capability from the same `/api/show` payload it already reads for `num_ctx`.
This changes its return shape (`int | None` ‚Üí a struct/tuple carrying num_ctx + vision), so
the existing `bootstrap/core.py` caller is updated in the same task. The probe degrades to
`vision=False` on any error ‚Üí honest gate.

### A2 ‚Äî `image_view` tool (dual path via `ToolReturn.content`)
```python
@agent_tool(visibility=DEFERRED, is_concurrent_safe=True,
            check_fn=_vision_available)
async def image_view(ctx, path: str, prompt: str = "Describe this image.") -> ToolReturn:
    # 1. resolve + read-boundary-confine path; stat-size guard (~20 MB)
    # 2. media_type from suffix; reject non-image
    # 3. data = Path(path).read_bytes()
    # 4. native = deps.agent_vision_capable_or_gemini; vm = deps.vision_model
    # 5a. NATIVE: build a RAW ToolReturn directly (tool_output cannot carry content):
    #        return ToolReturn(
    #          return_value="<short note: image attached; answer using vision>",
    #          content=[prompt, BinaryContent(data=data, media_type=media_type)])
    # 5b. FALLBACK: one multimodal request to vm via llm/call.py (settings_noreason) ‚Üí text;
    #        return tool_output(text, ctx=ctx, path=path, media_type=media_type)
```
The native branch **bypasses `tool_output`** (which has no `content` param, `tool_io.py:245`)
and constructs a raw `ToolReturn`; its `return_value` is a short note (no spill needed). The
fallback branch reuses `llm/call.py` (OTel span + per-call timeout), not a bare
`Agent(model).run`.

### A3 ‚Äî history processor: elide multimodal `UserPromptPart` on replay
A new processor registered in `spec.history_processors` (`agent/build.py:48`) that scans
**`UserPromptPart`** (not `ToolReturnPart` ‚Äî the existing processors only cover those) and,
for any non-tail part, strips/summarizes multimodal `UserContent` (the pixels produced by
`image_view`'s native path) down to a text placeholder (e.g. `[image elided]`). The most
recent turn's pixels are preserved so the model can still answer; older ones are dropped so
base64 does not accumulate across turns.

## Resolved decisions
1. **Scope = Phase A only (2026-06-09, PO review).** Clipboard paste (B) and image store (C)
   are deferred to a follow-up plan gated on shipping A and observing path-only ergonomics.
2. **Fallback single-request API surface.** The describe path reuses `co_cli/llm/call.py`
   (OTel spans + per-call timeout), not a bare `Agent(model).run`.
3. **Fallback reframed as a thin escape hatch**, not a co-equal path ‚Äî the default agent
   model is already vision-capable (Gemini + `qwen3.6` agentic), so the fallback fires only
   for a deliberately-pinned text-only Ollama model. Kept because it completes the honest
   gate at near-zero cost (one config field + 3 lines).
4. **Reasoning vs noreason settings.** The describe path uses `settings_noreason` (a
   description is not reasoning), falling back to `settings` when the resolved vision model
   has no noreason entry.
