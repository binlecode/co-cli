# Capability Enhancement — vision input (`image_view` + clipboard paste)

Task type: code

## Context

A capability enhancement that extends co's surface in a direction it cannot
currently reach, sourced from the cross-review against hermes-agent
(`docs/reference/RESEARCH-tools-gaps-co-vs-hermes.md` §2.1, §5). It is
**additive** — it does not fix a regression; it opens a new mode on top of
existing, proven plumbing:

- **Vision input**: co has no way to feed pixels to a model. It can read a PDF's
  text (documents skill) but cannot look at a screenshot, photo, diagram, or
  rendered chart. This is greenfield — a repo-wide grep confirms **no
  `BinaryContent` / `ImageUrl` / multimodal content** is constructed anywhere in
  `co_cli/` today (neither user-turn nor tool-result).

> **Split note (2026-05-28):** this plan originally bundled a second workstream,
> interactive task drive (`task_write`/`task_close`). That work moved to
> `2026-05-28-200025-toolgap-interactive-terminal.md` (Phase 2). This plan is
> vision-only.

> **Scope expansion (2026-06-02):** the plan grew from a single path-only tool to
> the full hermes-aligned shape: a dual-path `image_view` tool **plus** clipboard
> paste UX **plus** image-storage management. Rationale: a path-only tool does not
> serve the real use case (drop a screenshot, ask about it). See "Design alignment
> with hermes" below. This **supersedes** an earlier mid-design pick to attach
> pixels onto the user turn (`current_input`) — hermes deliberately avoids that
> channel; co follows hermes and attaches pixels via the **tool-result** channel.

### Why it rides proven infrastructure

- **Vision model** reuses the `judge_model` precedent: `LlmSettings` already
  carries an optional pinned secondary model name (`judge_model: str | None`) that
  inherits provider/host/api_key. Vision adds the same shape (`vision_model`).
- **Pixels-to-model** rides `pydantic_ai.messages.ToolReturn.content`
  (`str | Sequence[UserContent] | None`, verified in pydantic-ai 1.81) — a tool
  can *return* `BinaryContent` and the model sees real pixels on its next turn.
  co's `tool_output` already builds `ToolReturn`; it just does not populate
  `content` today. This is the same mechanism hermes uses (its "native fast path").
- **Image dir** rides the `USER_DIR`-derived constant pattern in
  `co_cli/config/core.py` (alongside `SESSIONS_DIR`, `MEMORY_DIR`, …) — never a
  hardcoded `~/.co-cli` (the known pitfall; tests override `CO_HOME`).

### Design alignment with hermes (grounded, not copied)

Studied `hermes-agent/tools/vision_tools.py` + `cli.py`. co copies the **shape and
mechanism**, not the backend wiring (co's model is a pydantic-ai object):

1. **One tool, capability-gated dual path** (`vision_analyze`): when the main model
   is vision-capable, attach pixels via the tool-result envelope → model sees them
   next turn ("native fast path"); otherwise an aux model returns a text
   description ("fallback path"). → co's `image_view`.
2. **Clipboard paste UX** (`cli.py`): `_try_attach_clipboard_image` **saves** the
   clipboard image to a persistent dir (`~/.hermes/images/clip_<ts>_<n>.png`),
   tracks `_attached_images: list[Path]`, renders `[📎 image attached]` badges,
   supports auto-attach on image-only paste + an explicit `/paste`, and patches
   bracketed-paste torn-sequence recovery. → co's input layer.
3. **Save + describe + path-for-escalation** (`_preprocess_images_with_vision`):
   on submit, hermes prepends a text description **and the saved file path** to the
   user message, so the agent can re-examine real pixels later via `vision_analyze`.
   The file is **persistent** (`should_cleanup=False` for local/clipboard files);
   only remote-URL downloads use a deleted temp.

**Where co intentionally diverges / improves:**
- **Describe step.** Hermes pre-describes with a *separate cheap aux model*. co's
  agent model *is* the vision model, so pre-describing with it is redundant and
  expensive. co's default: inject the **path** (+ short description only when a
  distinct cheap `vision_model` is configured); the agent escalates to real pixels
  on demand via `image_view`. (Open Question 2.)
- **Storage lifecycle.** Hermes never prunes its saved-image dir — it grows
  unbounded (confirmed: retention logic exists only for sessions, not images). co
  adds bounded management: prune-oldest at the new-session boundary, a banner
  warning when the dir is bloated, and a `/images` command to monitor/manage.
- **Remote URLs / video** stay out of scope (co keeps the tighter surface).

## Problem & Outcome

**Problem.** Any image input has nowhere to go — `file_read` rejects binary, the
REPL input is text-only, and no path feeds pixels to a model.

**Outcome.**
1. **`image_view(path, prompt)`** — reads a local image and, when the resolved
   model is vision-capable, returns the **pixels** via `ToolReturn.content` so the
   agent model sees them on its next turn; when only a text-only agent model + a
   pinned `vision_model` fallback is available, returns a **text description**
   instead. Capability-gated; never sends bytes to a blind model and fakes a result.
2. **Clipboard paste** — pasting an image (or `/paste`) saves it to
   `<USER_DIR>/images/`, shows an attachment badge, and on submit injects the saved
   **path** (+ optional description) into the turn so the agent can `image_view` it.
3. **Storage management** — the image dir is bounded: oldest entries pruned at the
   new-session boundary, a banner warning when it exceeds a size threshold, and a
   `/images` command to list/monitor/clear.

## Behavioural Constraints
1. **Honest capability gate.** When no vision-capable model resolves (text-only
   Ollama agent model + no `vision_model`), `image_view` is unavailable (`check_fn`
   false) with a remediation message. Never send bytes to a blind model and present
   the result as if it saw the image.
2. **Pixels via tool-result, not user turn.** Real pixels reach the model only
   through `ToolReturn.content`. The user turn / history / compaction stay
   text-only. **History portability:** stale `BinaryContent` in older
   `ToolReturnPart`s must be elided/summarized on replay so transcripts don't carry
   base64 bloat across turns (settle against `co_cli/context/history_processors.py`).
3. **Single request, no agent loop.** One multimodal request per `image_view`,
   bounded by the same per-call timeout discipline as other model calls.
4. **Persistent, bounded image store.** Saved clipboard/pasted images live under
   `<USER_DIR>/images/` (via the config constant, never a hardcoded path) and
   persist so the path is stable for re-examination — but the dir is bounded
   (prune-oldest at session boundary; banner warning; `/images` clear). Saved
   images are not world-readable (dir `0o700`).
5. **Local-file boundary.** `image_view` `path` resolved + workspace-confined like
   `file_read`; reject escapes. The `<USER_DIR>/images/` dir is an allowed read
   root for saved attachments.
6. **Size + media-type guards.** Cap at ~20 MB; accept only
   `image/{png,jpeg,webp,gif}`; reject others clearly (PDFs → documents skill).
7. **No new hard dependency** — pydantic-ai's existing `BinaryContent` /
   `ToolReturn`. Clipboard read uses a soft/optional dependency or platform shell;
   absence degrades to "no clipboard image found", never a crash.

## High-Level Design

### A1 — `build_vision_model` (factory) + capability detection
Resolution order — *use the agent model when it can see; pin a fallback only when
it can't*:
```python
def build_vision_model(llm: LlmSettings, agent_vision_capable: bool) -> LlmModel | None:
    if llm.uses_gemini():
        return build_model(llm)                        # natively multimodal
    if llm.uses_ollama() and agent_vision_capable:
        return build_model(llm)                        # agent model sees pixels (qwen3.6)
    if llm.uses_ollama() and llm.vision_model:
        return build_model(replace(llm, model=llm.vision_model))  # text-only → fallback pin
    return None                                        # text-only Ollama, no fallback
```
**Detection (no new IO).** Extend `probe_ollama_model` to also surface the `vision`
capability from the same `/api/show` payload it already reads for `num_ctx`;
`agent_vision_capable` is computed at bootstrap and passed in, keeping the factory
pure. The probe degrades to false on any error → honest gate.

### A2 — `image_view` tool (dual path via `ToolReturn.content`)
```python
@agent_tool(visibility=DEFERRED, is_read_only=True, is_concurrent_safe=True,
            check_fn=_vision_available)
async def image_view(ctx, path: str, prompt: str = "Describe this image.") -> ToolReturn:
    # 1. resolve + confine path (allow workspace + <USER_DIR>/images/); stat-size guard
    # 2. media_type from suffix; reject non-image
    # 3. data = Path(path).read_bytes()
    # 4. vm, native = ctx.deps.vision_model, ctx.deps.agent_vision_capable_or_gemini
    # 5a. NATIVE: model sees pixels — return ToolReturn(
    #        return_value="<short note: image attached, answer using vision>",
    #        content=[prompt, BinaryContent(data=data, media_type=media_type)])
    # 5b. FALLBACK: one multimodal request to vm (settings_noreason) → text;
    #        return tool_output(text, ctx=ctx, path=path, media_type=media_type)
```
Settle the fallback single-request call site against `co_cli/llm/call.py` in
TASK-A2 (reuse its OTel-span + timeout plumbing over a bare `Agent(model).run`).

### B1 — paste interception + attachment state
Mirror hermes: on paste, if the gesture is image-only (`not pasted_text.strip()`)
check the clipboard; `save_clipboard_image(<USER_DIR>/images/clip_<ts>_<n>.png)`,
append to a per-session `attached_images: list[Path]`, render a `[📎 …]` badge row,
and recover torn bracketed-paste sequences. `/paste` triggers the same explicitly.

### B2 — submit injection (save + path, describe optional)
On submit with attachments, prepend to the user text (hermes
`_preprocess_images_with_vision` analog, adapted):
```
[The user attached an image: <path>.
 Use image_view to look at it.]            # default: path-only
```
When a distinct cheap `vision_model` is configured, optionally also run a one-shot
description and inject it (hermes parity). The agent escalates to real pixels via
`image_view(<path>)` → A2 native path.

### C1 — bounded image store + banner warning
`ImageStore` over `<USER_DIR>/images/`: `usage_bytes()`, `prune_oldest(bound)`.
Prune-oldest runs at the new-session boundary (`_cmd_new` + bootstrap). Banner gets
a `build_images_line` that warns when `usage_bytes()` exceeds a threshold (e.g.
"Images: 412 MB ⚠ /images to manage").

### C2 — `/images` command
`BUILTIN_COMMANDS["images"]` → list saved images with sizes/ages + total usage;
`/images clear` prunes all (or oldest-N). Handler in `co_cli/commands/images.py`,
registered in `core.py` next to `/sessions`.

## Resolved decisions (2026-06-02, Gate 1)
1. **Fallback single-request API surface — RESOLVED.** The describe-path call
   reuses `co_cli/llm/call.py` plumbing (consistent OTel spans + per-call timeout)
   rather than a bare `Agent(model).run`. Wire it in A2.
2. **Describe-on-paste vs path-only — RESOLVED: path-only default.** Submit injects
   the saved **path** only; the agent escalates to real pixels via `image_view`.
   A one-shot description is injected *only* when a distinct cheap `vision_model` is
   configured (pre-describing with the heavy agent model is redundant/expensive).
3. **Prune bound — RESOLVED: total-size, 500 MB.** Both the prune-oldest bound and
   the banner-warning threshold key off `usage_bytes()` against a 500 MB ceiling
   (single constant). Prune-oldest deletes until under the ceiling at the
   new-session boundary; the banner warns when over it.
4. **Reasoning vs noreason settings — RESOLVED.** The describe path uses
   `settings_noreason` (description is not reasoning), falling back to `settings`
   when the resolved vision model has no noreason entry.
5. **Single plan, phased delivery — RESOLVED.** Paste (B) and storage (C) stay in
   this plan rather than a follow-up — they are the point of the feature. They ship
   in dependency phases (see Shipping order); each task is independently
   verifiable.
