## Context

The OTel SDK exists to ship traces to external collectors over a standard wire format — co-cli doesn't do that and has no plan to. For a personal local-only CLI, the investment doesn't earn its keep.

## Problem & Outcome

**Problem.** Three compounding issues with the current observability stack:

1. **Infrastructure mismatch.** OTel SDK, two span exporters, and the HTML trace viewer exist for use cases (external collectors, browser-based deep post-mortem) that a personal local CLI does not have.

2. **Framework lifecycle awkwardness.** pydantic-ai's `Agent.instrument_all()` creates `invoke_agent` spans whose activation lives only inside `agent.iter()` and is ended in a `finally` block before `agent.run()` returns. Attaching co-cli attributes "to the framework span" from caller code is impossible — `trace.get_current_span()` after the call returns whatever enclosing span exists, never the `invoke_agent` span.

3. **Business logic coupling.** 24 manual span sites + 5 attribute-attach sites = 29 OTel touchpoints embedded in business code across 13 files. Contributors must understand OTel mechanics to read or modify core agent / context / memory code.

**Failure cost (concrete):**
- ~5 OTel SDK packages carried for ~5% of OTel's surface used.
- Two span exporters + a SQLite schema + an HTML renderer maintained for co-cli's own data displayed back to co-cli's own tools.
- Trace bloat from manual-span-wraps-framework-span duplication.
- 29 code sites that any future observability change has to traverse.

**Outcome.** OTel is retired. Replaced by:

- Single structured JSON log file with `RotatingFileHandler` (50 MB × 5 backups by default).
- `@trace` decorator for the small set of co-cli operations that need explicit span boundaries.
- `ObservabilityCapability(AbstractCapability[CoDeps])` emitting records on agent/model/tool lifecycle via pydantic-ai's hook surface (correct signatures throughout, error hooks included for stack cleanup).
- `co tail` reads the JSON log via tail-style follow — live, append-only, summary + `--detail`. No tree mode.
- `co trace <trace_id>` — new CLI command. Reads completed records for one trace, builds parent/child structure in memory, renders an indented tree, exits. Snapshot, not live.
- HTML viewer (`co_cli/observability/viewer.py`, `co traces` command) deleted entirely.

Traceability is preserved via record fields: `trace_id`, `span_id`, `parent_span_id`, `session_id`, `attributes`, `events`. OTel's intrusiveness is eliminated. The investment matches the use case.

## Behavioral Constraints

- Every operation that currently emits a span must continue to emit a log record. No silent visibility loss. Concretely: agent runs, model requests, tool executions, `co.turn`, `co.dream.*`, `co.knowledge.knowledge_manage.*`, `compaction.proactive_check`, bootstrap spans, `background_task_execute` — all must appear in the new log.
- Each record must include: `ts`, `schema_version`, `session_id`, `trace_id`, `span_id`, `parent_span_id` (nullable), `name`, `kind`, `start_ts`, `duration_ms`, `status`, `status_msg` (nullable), `attributes`, `events`. Every field has a justified consumer (see §1 below).
- The `@trace` decorator must auto-detect sync vs async via `inspect.iscoroutinefunction()`. One decorator name for both.
- Inside a `@trace`-decorated function, `current_span()` must return a span handle whose `.set_attribute(k, v)` and `.add_event(name, attrs)` calls mutate the active span dict. Business code must never need to import `opentelemetry` or any successor library.
- `parent_span_id` derived from a `contextvars.ContextVar` stack. The stack must survive `asyncio.gather`, `asyncio.create_task`, and `asyncio.to_thread` correctly. For `loop.run_in_executor` (raw thread executor without ContextVar propagation), call sites must wrap with `contextvars.copy_context().run(...)` or use the documented helper from `tracing.py`. Verified by test.
- `ObservabilityCapability` hook methods must return the correct types: `before_run -> None`, `after_run -> AgentRunResult`, `on_run_error -> AgentRunResult` (re-raise by default), `before_model_request -> ModelRequestContext`, `after_model_request -> ModelResponse`, `on_model_request_error -> ModelResponse` (re-raise by default), `before_tool_execute -> ValidatedToolArgs`, `after_tool_execute -> Any`, `on_tool_execute_error -> Any` (re-raise by default). Error hooks must pop the span stack on exception path so `parent_span_id` doesn't leak across runs.
- **Attribute parity contract.** The following attributes that `tail.py --detail` currently reads from OTel spans must be populated by `ObservabilityCapability` on the equivalent log record:
  - Agent record (`kind="agent"`): `agent.role`, `agent.model`, `agent.request_limit`, `agent.requests_used`, `final_result` (string content of agent output)
  - Model record (`kind="model"`): `gen_ai.input.messages` (JSON-serialized list of message dicts, structure preserved including `role`, `parts[].type`, `parts[].content`, and `parts[].type == "thinking"` chunks), `gen_ai.output.messages` (same shape, including thinking parts), `gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`, `gen_ai.request.model`, `gen_ai.response.finish_reason`
  - Tool record (`kind="tool"`): `gen_ai.tool.call.arguments` (JSON string), `gen_ai.tool.call.result` (JSON string, size-capped per existing spill policy), `co.tool.source`, `co.tool.requires_approval`, `co.tool.result_size`
  Attribute names retained verbatim from current OTel-emitted names so `tail.py` rendering code is unchanged at the attribute-read level.
- **Redaction parity.** `settings.observability.redact_patterns` continues to work. Redaction applies to string values inside `attributes` and `events[*].attributes` recursively (one level deep into nested JSON-string values where the value parses as JSON — apply per-leaf string regex). Implemented in `tracing.py`'s record emission path before write. Verified by `tests/test_flow_observability_redaction.py` rewritten to read JSON log lines.
- `RotatingFileHandler` defaults: 50 MB per file, 5 backups. Path: `~/.co-cli/logs/co-cli-spans.jsonl`. Configurable via existing `settings.observability` fields (`log_max_size_mb`, `log_backup_count`) — reuse existing config schema; no new settings keys.
- Log rotation safety: `RotatingFileHandler.rolloverAt` is synchronous, holds the handler's lock, briefly closes the current file and opens a new one. Concurrent writes from other threads queue on the lock. Records in flight at rollover are not lost (stdlib guarantee). `tail.py` follow-mode detects rotation via inode change (`os.stat().st_ino`) and re-opens the new file from offset 0 — not via size comparison.
- `pyproject.toml` must contain no `opentelemetry-*` dependency after this change. `uv tree | grep -i opentelemetry` returns empty post-ship.
- `co tail` summary mode: re-read interval ≤100 ms; tail position seek via file offset; rotation via inode tracking.
- `co trace <id>` reads the log and the rotated backup files (matching glob `co-cli-spans.jsonl*`) to find all records of one trace. Builds tree, renders, exits. No follow.
- No functional behavior change to agent / tool / model execution. Agent's responses, decisions, and side effects must be identical before and after.
- Schema versioning: every log record carries `"schema_version": 1`. Schema bumps allowed in future; readers must check.

## High-Level Design

### 1. Log record schema — every field justified

JSON-line format, one record per span close. Spans emit on close, not on open — records are self-contained.

```json
{
  "ts": "2026-05-16T19:30:00.123456Z",
  "schema_version": 1,
  "session_id": "a1b2c3d4",
  "trace_id": "t_e5f6g7h8",
  "span_id": "s_i9j0k1l2",
  "parent_span_id": "s_xxxx",
  "name": "co.turn",
  "kind": "agent",
  "start_ts": "2026-05-16T19:30:00.000123Z",
  "duration_ms": 123.456,
  "status": "OK",
  "status_msg": null,
  "attributes": { ... },
  "events": [ { "ts": "...", "name": "...", "attributes": {...} } ]
}
```

| Field | Justification (named consumer) |
|---|---|
| `ts` | Emission timestamp — used by `co tail` follow-mode for ordering across rotation, used in summary line render |
| `schema_version` | Forward compatibility — readers can refuse incompatible versions |
| `session_id` | Filter / correlation in `co tail`; matches `co chat` session boundary |
| `trace_id` | Group records into a trace for `co trace <id>` and for `--trace` filter |
| `span_id` | Parent linkage target; uniqueness within a trace |
| `parent_span_id` | Tree assembly in `co trace <id>` (nullable for trace roots) |
| `name` | Display in summary line, filterable |
| `kind` | Display color, `--tools-only` / `--models-only` filters, replaces OTel span kind with co-cli vocabulary (`agent`, `model`, `tool`, `co`) — four values, no `co.system` |
| `start_ts` | Sibling ordering in tree view (record `ts` is close-time, not start-time) |
| `duration_ms` | Summary line render, performance triage |
| `status` | `OK` / `ERROR` — display, filter on errors |
| `status_msg` | Error context — null on OK; populated by `on_*_error` hooks. Distinct from python application-log error handler (which captures exception traceback in the application log file, not in span records) |
| `attributes` | Per-record payload (the long tail of useful data) |
| `events` | Nested mini-records for "thing happened with attributes but no duration to measure" — converted from prior span-with-near-zero-duration patterns (see §6 disposition table) |

### 2. `tracing.py` — decorator, contextvars, log setup

Public module — no leading underscore — because consumed from across packages.

API surface:
- `setup_log(log_path: Path, *, max_size_mb: int, backup_count: int, redact_patterns: list[str]) -> None`
- `set_session_context(session_id: str) -> None`
- `clear_session_context() -> None`
- `new_trace() -> str` — generates a fresh 16-hex `trace_id`, sets the contextvar, returns it. Existing span stack is preserved.
- `current_span() -> Span` — returns top-of-stack span proxy (or no-op proxy if stack is empty)
- `trace(name: str | None = None)` — decorator
- `run_with_context(coro_or_fn)` — helper for `loop.run_in_executor` to propagate ContextVars via `contextvars.copy_context()`

ContextVars:
```python
_SESSION_ID: ContextVar[str | None] = ContextVar("co_session_id", default=None)
_TRACE_ID: ContextVar[str | None] = ContextVar("co_trace_id", default=None)
_SPAN_STACK: ContextVar[tuple[dict, ...]] = ContextVar("co_span_stack", default=())
```

`@trace(name=None)`:
- Detects sync/async via `inspect.iscoroutinefunction(func)`
- Name defaults to `f"co.{module_basename}.{func_name}"` if not given
- On entry: generates `span_id`, captures `parent_span_id` from `_SPAN_STACK` top, captures `trace_id` from `_TRACE_ID` (lazy-generate if None), pushes new span dict onto stack
- On exit (normal): pops, computes `duration_ms`, emits record via spans logger with `status="OK"`
- On exit (exception): pops, sets `status="ERROR"`, `status_msg=str(exc)`, emits, re-raises
- Static `attributes=` parameter dropped (no current consumer per PO-M-3)

`Span` proxy:
- `set_attribute(key, value)` — mutates `attributes` dict on top-of-stack span
- `add_event(name, attributes=None)` — appends to `events` list with `ts=now`
- `set_status(status, msg=None)` — overrides default OK/ERROR
- No-op variant returned when stack empty (defensive — observability code never raises into business code)

Redaction: implemented in record-emission helper, recursively walks `attributes` and `events[*].attributes`. For each string value, applies regex patterns. For string values that parse as JSON, parses and recurses one level (to catch secrets inside `gen_ai.input.messages` content). Depth-1 only — bounded.

Log setup:
- One handler — `RotatingFileHandler` writing to `~/.co-cli/logs/co-cli-spans.jsonl`. JSON formatter outputs one record per line.
- Dedicated logger `co_cli.observability.spans` with `propagate=False` (records go ONLY to this handler, not to the application log).
- The application log (`co-cli.jsonl` via `file_logging.py`) is unchanged and unrelated to span records. Two files, two purposes: app log for Python `logging` records, span log for trace records. The prior plan's "one file" framing was wrong — Core Dev flagged this contradiction.

### 3. `capability.py` — `ObservabilityCapability`

```python
class ObservabilityCapability(AbstractCapability[CoDeps]):

    async def before_run(self, ctx):
        _push_span(name=f"invoke_agent {ctx.agent.name}", kind="agent",
                   attrs={"agent.role": ctx.agent.name, ...})

    async def after_run(self, ctx, *, result):
        _pop_span(status="OK", attrs={
            "agent.requests_used": result.usage().requests,
            "final_result": str(result.output),
            ...
        })
        return result

    async def on_run_error(self, ctx, *, error):
        _pop_span(status="ERROR", status_msg=str(error))
        raise error

    async def before_model_request(self, ctx, request_context):
        _push_span(name=f"chat {request_context.model.model_name}", kind="model",
                   attrs={"gen_ai.input.messages": _serialize_messages(request_context.messages), ...})
        return request_context

    async def after_model_request(self, ctx, *, request_context, response):
        _pop_span(status="OK", attrs={
            "gen_ai.output.messages": _serialize_response(response),
            "gen_ai.usage.input_tokens": response.usage.input_tokens,
            ...
        })
        return response

    async def on_model_request_error(self, ctx, *, request_context, error):
        _pop_span(status="ERROR", status_msg=str(error))
        raise error

    async def before_tool_execute(self, ctx, *, call, tool_def, args):
        _push_span(name=f"tool {call.tool_name}", kind="tool",
                   attrs={"gen_ai.tool.call.arguments": json.dumps(args), ...})
        return args

    async def after_tool_execute(self, ctx, *, call, tool_def, args, result):
        _pop_span(status="OK", attrs={
            "gen_ai.tool.call.result": _truncate(result),
            "co.tool.result_size": len(str(result)),
            ...
        })
        return result

    async def on_tool_execute_error(self, ctx, *, call, tool_def, args, error):
        _pop_span(status="ERROR", status_msg=str(error))
        raise error
```

Wired alongside `CoToolLifecycle` in `co_cli/agent/build.py` via `capabilities=`. Ordering: `ObservabilityCapability` placed **before** `CoToolLifecycle` so its `before_*` hooks open spans first and its `after_*` hooks close last — span boundaries enclose `CoToolLifecycle`'s timing accurately.

### 4. `co tail` — live, append-only

```
co tail                  — follow co-cli-spans.jsonl, render summary line per record as it appears
co tail --last N         — print last N records and exit (no follow)
co tail --detail         — append per-record detail block (per the in-flight tail-detail-tree-modes plan, adapted)
co tail --tools-only     — kind == "tool"
co tail --models-only    — kind == "model"
co tail --trace ID       — trace_id == ID
co tail --no-follow      — print recent and exit
```

Implementation: open `co-cli-spans.jsonl`, seek to end (or `--last N` lines back), loop:
- Read available lines, parse, render
- If no new lines: stat the file; if inode changed (rotation), open the new file at offset 0; else sleep ~100 ms

The three helpers previously imported from `viewer.py` (`extract_span_attrs`, `format_duration`, `get_span_type`) are small — inline into `tail.py` and delete `viewer.py`. `get_span_type` is replaced by reading the new `kind` field directly.

**No tree mode.** `--tree` is removed from this plan and from the in-flight `tail-detail-tree-modes` plan (that plan shrinks to `--detail` only, or is retired if `--detail` arrives here first).

### 5. `co trace <trace_id>` — snapshot tree

New Typer command. Implementation ~80 lines:

```python
@app.command()
def trace(trace_id: str):
    """Render the full tree of one trace as a snapshot."""
    records = _read_records_for_trace(trace_id)  # globs co-cli-spans.jsonl* and rotated backups
    if not records:
        console.print(f"[yellow]No records for trace {trace_id}[/yellow]")
        return
    tree = _build_tree(records)  # group by parent_span_id, sort siblings by start_ts
    _render_tree(console, tree)
```

No follow mode. No depth cap (completed traces are bounded — cap was needed for live tail's "endless tree" risk, doesn't apply here). No filters (one trace, render all of it).
