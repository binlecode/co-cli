## Context

Hermes's skill system self-evolves through three mechanisms:

1. **In-session drift patching (model-driven).** Model loads a skill, executes, hits an issue not covered, patches the skill before finishing the task. Encoded in `skill_manager_tool.py` docstring: *"If you used a skill and hit issues not covered by it, patch it immediately."*

2. **Post-session skill review (autonomous fork).** After every N tool calls (default 10), hermes forks a second `AIAgent` with the full conversation history, scoped to only the skills toolset, `max_iterations=8`. It scans for: user-correction patterns, drift the foreground model missed, reusable procedures worth saving. Patches and creates skills autonomously. Reports `💾 <summary>` to the user. See `hermes-agent/run_agent.py:3240-3576`, `_SKILL_REVIEW_PROMPT`.

3. **Curator (consolidation + state machine).** Pure state transitions (active → stale at 30 days unused → archived at 90 days). Forked agent scoped to skills toolset (`max_iterations=9999`, very large). Consolidates prefix-clustered narrow skills into class-level umbrellas. Trigger: startup, when `now - last_run > interval_hours` (default 168 hours = 7 days). See `hermes-agent/agent/curator.py:1-870`, `hermes-agent/tools/skill_usage.py:1-457`.

**What co-cli has.** Mechanism 1 — shipped in Plan 3 as the `## Drift` reflex in `06_skill_protocol.md` + `skill_manage(action='patch')`.

**What co-cli is missing.** Mechanisms 2 and 3. The model improves skills only when it notices issues mid-turn. Most session learnings are lost: by the time it's worth saving, the model has moved past the relevant context. There's no second pass, no autonomous library hygiene, and no usage data to prioritize maintenance.

### Trigger model: turn-boundary, not session-end

Hermes fires its review **at turn boundaries** once `_iters_since_skill >= N` (default every 10 tool calls). Each fire covers the full session history available at that point. Co-cli ports this directly:

- Counter `_iters_since_session_review` on session state, bumped per tool call.
- After each `_finalize_turn`, check counter. If threshold reached, fire review as `asyncio.create_task(...)` — fire-and-forget, doesn't block the next user turn. Reset counter.
- Status surfaces via `Frontend.on_status()` when the background task completes.
- On REPL exit (`_drain_and_cleanup`), await any pending review task with a short timeout, then cancel.

This is a meaningful behavioral choice. Earlier drafts of this plan deferred mid-session triggering to a future revision; G1 review (this turn) pushed it back into scope as required for true hermes parity. Per-turn cadence catches learnings while context is fresh; session-end-only loses fidelity in long sessions.

The existing session-end dream cycle (`_maybe_run_dream_cycle`) remains unchanged — it consolidates memory artifacts, complementing (not competing with) the turn-boundary review that extracts new artifacts.

## Problem & Outcome

**Problem.** Co-cli's skill library evolves only when the model notices something mid-turn (the `## Drift` reflex) or when the user explicitly asks for a save. The longer-horizon improvements — patterns that emerge across multi-turn sessions, drift the model didn't catch in-flight, deprecated/superseded skills cluttering the index — are never captured. Without usage data, the library has no signal for what's valuable vs vestigial.

**Failure cost.** Skill quality plateaus. The library grows monotonically (every `## Create` adds, nothing prunes). `skill_search` results degrade. The model can't differentiate a well-used skill from a stale one. User-driven improvements (corrections, style preferences) are absorbed into memory but not encoded into skills, so the next session re-learns them.

**Outcome.** Three new mechanisms make the library self-improving:

1. **Usage tracking sidecar** (`~/.co-cli/skills/.usage.json`) — per-skill counters and timestamps: `use_count`, `view_count`, `patch_count`, `created_at`, `last_used_at`, `last_viewed_at`, `last_patched_at`, `state` (`active`|`stale`|`archived`), `pinned`. Bumped at hook points in `skill_view` and `skill_manage`. Filtered to agent-created skills (bundled and URL-installed skills excluded — they're upstream-managed).

2. **Turn-boundary combined session review** — new `_post_turn_hook(deps, message_history, turn_tool_call_count)` fired after `_finalize_turn` (line 135 in `co_cli/main.py`) when the per-session tool-call counter exceeds `skills.review_nudge_interval` (default 10). Hook bumps counter, checks threshold + single-in-flight, then spawns `asyncio.create_task(_run_review_bg(deps, history_copy))`. `_run_review_bg` calls `run_session_review(deps, history)` under an outer `asyncio.wait_for` timeout. The forked agent has `delegation="session_reviewer"` (covers both skill AND knowledge tools — single combined pass per hermes's `_COMBINED_REVIEW_PROMPT`). `max_iterations=8`. Identifies drift, reusable procedures, user corrections worth memorializing. Patches/creates skills AND knowledge artifacts autonomously. Never blocks the next user turn; status surfaces via `Frontend.on_status()` when complete.

3. **Curator** — state machine + consolidation. Pure state transitions run at startup if `(now - last_run > interval_hours) AND (idle_seconds > curator_min_idle_hours * 3600)`. Stale set when `last_used_at > stale_after_days` ago; archived (file moved to `.archive/`) when `> archive_after_days`. Pinned skills opt out. After transitions, a forked agent runs consolidation — scoped to `delegation="skill_curator"`, `max_iterations=100` (configurable up to 999 for large libraries) — merging prefix-clustered narrow skills into broader umbrellas. Adapted to flat-file: merged content goes inline as labeled subsections. Per-run reports written as BOTH JSON (`run.json`) AND markdown (`run.md`) for human readability.

4. **CLI control surface.** `/skills curator status|run|pause|resume|restore <name>`, `/skills usage [<name>]`, `/skills pin <name>` / `/skills unpin <name>`, `/skills review run` (force-fire a combined review on demand).

5. **Protocol acknowledgment** in `06_skill_protocol.md` — short `## Background review` section telling the model the review and curator exist (so it doesn't duplicate work) and that pinning is the opt-out.

## Behavioural Constraints

1. **Agent-created filter is absolute.** Usage tracking, state transitions, and curator consolidation apply only to user-skills-dir skills *without* `source-url`. Bundled skills (under `co_cli/skills/`) and URL-installed skills are upstream-managed. Predicate: `is_agent_created(name, deps)` checks both location and `source-url` absence.

2. **Pinned skills opt out.** A `pinned: true` field in the sidecar (not in skill frontmatter — to avoid polluting tracked bundled-skill files) exempts a skill from all auto-state-transitions and curator consolidation. Set/cleared via `/skills pin <name>` / `/skills unpin <name>`.

3. **Archive is recoverable, not deleted.** Archived skills move to `~/.co-cli/skills/.archive/<name>.md`. Restoration via `/skills curator restore <name>`. The curator never deletes.

4. **Approval bypass is scoped to forked deps.** `CoRuntimeState.auto_approve_skill_ops` and `auto_approve_knowledge_ops` both default False. Set True only inside `fork_deps_for_reviewer()` / `fork_deps_for_curator()` factories (reviewer sets both; curator sets only skill_ops). Approval-subject builders check the relevant flag and short-circuit the gate when True. Foreground tool calls always require approval.

5. **Review, curator, and combined-tool review are opt-in by default.** `skills.review_enabled: bool = False`, `skills.curator_enabled: bool = False`. Rationale: a forked agent writing to the skill/knowledge library autonomously is a strong default; users opt in explicitly. `skills.usage_tracking_enabled: bool = True` (cheap, no LLM cost).

6. **Errors never propagate.** Review and curator failures call `logger.warning("...", exc_info=True)` and return. Per-turn review failures must not block the next user turn. Curator failures must not block REPL startup. Pattern copied from `_maybe_run_dream_cycle`.

7. **Iteration budgets are bounded.** Review agent: `max_iterations=8`. Curator agent: `max_iterations=100` (configurable up to 999 — hermes uses 9999). Hard limits passed as `UsageLimits(request_limit=budget)` via `_run_agent_standalone`.

8. **Forks use the foreground model.** Same model object passed through `fork_deps`; same credentials, same provider. Cost surfaces in the same tracing/log stream as foreground turns.

9. **Usage tracking is best-effort.** Sidecar writes are atomic (`tempfile + os.replace`); failures are logged but never block the underlying skill operation. The sidecar can be deleted at any time — it rebuilds from zero counts on next bump.

10. **Reviewer/curator prompts frame the model as a maintainer with explicit preference order.** Prompts include hermes's "Be ACTIVE" framing (*"most sessions produce at least one update; a pass that does nothing is a missed learning opportunity, not a neutral outcome"*) plus the 3-tier preference order (UPDATE LOADED skill → UPDATE EXISTING UMBRELLA → CREATE NEW UMBRELLA). Tier 3 from hermes (ADD SUPPORT FILE under umbrella) collapses for co-cli's flat-file model — supplementary detail goes inline as labeled subsections. Session-specific names (`fix-pr-123`, `debug-X-today`) are forbidden — only class-level taxonomies. **Never call `skill_manage(action='delete')` or `knowledge_manage(action='delete')` from a fork** — for skills, the curator's state machine handles cleanup via archive; for knowledge, prefer `replace` over `delete` (knowledge has no archive surface today).

11. **Turn-boundary review concurrency model.** Review fires as `asyncio.create_task(...)` from the post-turn hook. The next user turn begins immediately; the review runs concurrently. If a new review trigger fires while the previous review is still running, the new trigger is skipped (single in-flight review per session). The hook also checks `background_curator_task` — if the startup curator is still running, the review is skipped to avoid concurrent writes to the skill library. On REPL exit, cleanup awaits any pending review with a short timeout (5s default), then cancels.

12. **Curator idle gating.** Curator triggers only when BOTH `(now - last_run_at) > curator_interval_hours` AND `(now - session.last_user_input_at) >= curator_min_idle_hours` (default 2 hours). Idle-tracking is per-session state in `deps.session.last_user_input_at`, bumped on every user prompt. On REPL startup, idle is effectively infinite (no foreground turns yet) — curator fires immediately if interval gate passes.

13. **Per-run reports persist as JSON + markdown.** Both reviewer and curator write `run.json` (machine) AND `run.md` (human) to per-agent top-level directories: `~/.co-cli/session-reviews/<timestamp>/` and `~/.co-cli/curator-runs/<timestamp>/`. JSON is the structured audit trail; markdown summarizes for `/skills curator status` history view. Reports are tier-equivalent to `sessions/` and `knowledge/` (durable run artifacts, not observability logs) — kept out of `logs/` (which is for OTEL/file logging). The directory name is a sortable timestamp `YYYYMMDD-HHMMSS` (e.g. `20260513-094221/`); the pydantic-ai `run_id` is recorded as a field inside `run.json` for trace correlation. If two fires collide within the same second, append a short hash suffix (`20260513-094221-a3f2/`).

14. **Combined review handles skills and knowledge in one fork.** Per hermes's `_COMBINED_REVIEW_PROMPT`, a single forked agent has BOTH skill and knowledge toolsets. Single review pass updates both surfaces, treating user-corrections as candidates for memory artifacts AND skill patches. Avoids the cost of two separate forks and the prompt confusion of duplicated triggers.

## Failure Modes

1. **Reviewer over-creates narrow skills.** The model treats every session as a save opportunity, polluting the library.
   - Mitigation: `SESSION_REVIEW_PROMPT` explicitly forbids session-specific names; only class-level umbrellas. Curator consolidates narrow siblings on the next pass. Behavioral test: review on a "fixed-pr-123" style session asserts no skill named after the PR is created.

2. **Curator merges skills the user values.**
   - Mitigation: `pinned: true` opt-out via sidecar; archive (recoverable) only, never delete; per-run reports persisted for audit; `restore` command for human reversal.

3. **Approval bypass leaks into foreground.**
   - Mitigation: `auto_approve_skill_ops` lives on `CoRuntimeState` (which `fork_deps` resets); set only inside `fork_deps_for_reviewer()` / `fork_deps_for_curator()` factories. No public parameter. Unit test asserts the flag is False after a plain `fork_deps()` call, True only after the named-factory calls.

4. **Turn-boundary review delays the next user turn.**
   - Mitigation: review is spawned via `asyncio.create_task(...)`, never awaited by the foreground turn loop. User-perceived latency is zero; the review runs concurrently and reports when done.

4a. **Two reviews fire concurrently on a fast-moving session.**
   - Mitigation: BC11 — single in-flight review per session. New trigger while previous is running is skipped; counter still resets so the next eligible trigger gets a fresh window.

4b. **Pending review still running at REPL exit.**
   - Mitigation: `_drain_and_cleanup` awaits the pending task with a short timeout (default 5s), then cancels. The cancellation is graceful — partial writes are atomic, so no half-applied edits.

5. **Sidecar corruption.**
   - Mitigation: atomic writes; on parse failure, log and start fresh (the sidecar is observable state, not authoritative — counts rebuild from next operation).

6. **Curator blocks first response at startup.**
   - Mitigation: spawned as `asyncio.create_task(...)`, never awaited by the REPL bootstrap. REPL becomes interactive immediately; status surfaces when complete. At startup `last_user_input_at` is None — `_idle_seconds` returns inf so the idle gate passes; the curator runs concurrently with the user's first turn (writes are atomic; BC11 prevents a concurrent reviewer fork from racing the curator).

7. **Curator fires while user is mid-task.**
   - Mitigation: idle gating (BC12). Curator skips if `(now - session.last_user_input_at) < curator_min_idle_hours` (default 2h). On startup, idle is effectively infinite — first run is unblocked, but subsequent in-REPL `/skills curator run` invocations bypass the gate intentionally.

8. **Reviewer patches a skill the foreground agent is about to use.**
   - Mitigation: review writes are atomic; foreground reads from `deps.skill_commands` (refreshed after each write). Race is brief and benign — at worst, the foreground reads a slightly-newer version. Skill file format is stable across edits.

9. **Reviewer or curator hangs.**
   - Mitigation: outer `asyncio.wait_for` with timeout. On timeout, cancel the task and log. The agent's own iteration budget is the inner limit.

10. **Multiple curator runs concurrently** (two REPLs launched within seconds).
    - Mitigation: curator state file write is atomic and uses optimistic concurrency: read state, check `should_run_now`, write back with new `last_run_at`; if the on-disk `last_run_at` differs from the value read, abort (the loser of the race). Implementation in `co_cli/skills/curator.py`.

11. **`source-url` collision with agent-created skill.**
    - Mitigation: `is_agent_created` checks BOTH location (under `user_skills_dir`) AND `source-url` absence. A skill with `source-url` is upstream-managed even if user copied it. Behavioral test covers both branches.

12. **Status callback surfaces while user is typing.**
    - Mitigation: existing `TerminalFrontend._status_live` buffering (see `co_cli/display/core.py:282–285`) already handles this — status messages buffer during active input and flush when input ends. No new infrastructure; rely on the established pattern.

13. **Background task does sync setup before first `await`, briefly blocking the REPL.**
    - Mitigation: `run_session_review` and `maybe_run_curator` should `await asyncio.sleep(0)` at the very start to yield control immediately. Agent construction (`build_agent`) then runs without blocking the next user prompt. Implementer detail; verified by the latency-budget test in TASK-3.

## High-Level Design

### Usage tracking sidecar (`co_cli/skills/usage.py`)

Schema (`~/.co-cli/skills/.usage.json`):

```json
{
  "version": 1,
  "skills": {
    "<name>": {
      "use_count": 0,
      "view_count": 0,
      "patch_count": 0,
      "created_at": "2026-05-12T14:48:40Z",
      "last_used_at": null,
      "last_viewed_at": null,
      "last_patched_at": null,
      "state": "active",
      "pinned": false
    }
  }
}
```

API:
```python
def bump_view(deps: CoDeps, name: str) -> None: ...
def bump_use(deps: CoDeps, name: str) -> None: ...
def bump_patch(deps: CoDeps, name: str) -> None: ...
def record_create(deps: CoDeps, name: str) -> None: ...
def forget(deps: CoDeps, name: str) -> None: ...
def set_pinned(deps: CoDeps, name: str, pinned: bool) -> None: ...
def is_agent_created(name: str, deps: CoDeps) -> bool: ...
def read_records(deps: CoDeps) -> dict: ...
def write_records(deps: CoDeps, data: dict) -> None: ...  # atomic
```

### Foreground vs background agent runners

Two helpers, one symmetric pair. The split is which context the agent runs inside:

### Standalone agent runner (`co_cli/agents/_runner.py`)

`_run_agent_in_turn` is foreground-only — it dereferences `ctx.deps`, calls `_merge_turn_usage(ctx, ...)` to roll child usage into the parent turn's accumulator, and raises `ModelRetry` on failure (a pydantic-ai signal meaningful only inside a parent tool-call loop). Reviewer and curator both run OUTSIDE any tool-call context, so they need a different runner:

```python
# co_cli/agents/_runner.py
from copy import copy
from typing import Any

from opentelemetry import trace as otel_trace
from pydantic_ai.usage import RunUsage, UsageLimits

from co_cli.deps import CoDeps

_TRACER = otel_trace.get_tracer("co-cli.agents")


async def _run_agent_standalone(
    agent: Any,
    prompt: str,
    deps: CoDeps,
    budget: int,
    model_settings: Any,
    *,
    role: str,
) -> tuple[Any, RunUsage, str]:
    """Run a background agent (reviewer or curator).

    Differs from delegation.py:_run_agent_in_turn in three ways:
      1. Takes `deps` (already forked by caller) instead of `ctx` — no foreground RunContext.
      2. Does NOT merge usage into `turn_usage` — background tasks are independent
         budget centers; folding into a foreground turn would mis-attribute cost.
      3. Does NOT raise ModelRetry on failure — caller wraps with try/except for
         logger.warning swallowing. ModelRetry is foreground-protocol-specific.
    """
    with _TRACER.start_as_current_span(role) as span:
        span.set_attribute("agent.role", role)
        span.set_attribute("agent.model", str(agent.model))
        span.set_attribute("agent.request_limit", budget)
        result = await agent.run(
            prompt,
            deps=deps,
            usage_limits=UsageLimits(request_limit=budget),
            model_settings=model_settings,
            metadata={"session_id": deps.session.session_path.stem[-8:], "role": role},
        )
        usage = result.usage()
        span.set_attribute("agent.requests_used", usage.requests)
        return result.output, copy(usage), result.run_id
```

Both `run_session_review` and `maybe_run_curator` import and call `_run_agent_standalone`. `delegation.py:_run_agent_in_turn` (renamed in this plan) stays semantically untouched — foreground delegation tools (`web_research`, `knowledge_analyze`, `reason`) continue to use it via the new name.

```python
async def _post_turn_hook(deps: CoDeps, message_history: list[ModelMessage], turn_tool_call_count: int) -> None:
    """Called after _finalize_turn before returning to the REPL prompt."""
    if deps is None:
        return
    settings = deps.config.skills
    if not settings.review_enabled:
        return

    # Bump counter; check threshold.
    deps.session.iters_since_review += turn_tool_call_count
    if deps.session.iters_since_review < settings.review_nudge_interval:
        return

    # Single in-flight: skip if previous review OR startup curator is still running.
    # (Curator and reviewer both write to the skill library — avoid concurrent writes.)
    for attr in ("background_review_task", "background_curator_task"):
        task = getattr(deps.session, attr, None)
        if task is not None and not task.done():
            return  # counter NOT reset; subsequent triggers will retry once task completes

    deps.session.iters_since_review = 0
    deps.session.background_review_task = asyncio.create_task(
        _run_review_bg(deps, list(message_history))  # defensive copy: history may mutate
    )

async def _run_review_bg(deps: CoDeps, message_history: list[ModelMessage]) -> None:
    """Background review runner. Errors are logged and swallowed."""
    logger = logging.getLogger(__name__)
    try:
        timeout = deps.config.skills.review_timeout_seconds
        result = await asyncio.wait_for(
            run_session_review(deps, message_history),
            timeout=timeout,
        )
        cb = deps.runtime.background_status_callback
        if result.summary and cb is not None:
            cb(f"💾 {result.summary}")
    except asyncio.TimeoutError:
        logger.warning("Session review timed out")
    except asyncio.CancelledError:
        logger.info("Session review cancelled")
        raise
    except Exception:
        logger.warning("Session review failed", exc_info=True)
```

### Output schemas (reviewer / curator)

The reviewer and curator both run with `output_type=...` to return structured output. `AgentOutput` (the existing foreground-delegation precedent at `co_cli/tools/agents/delegation.py:23`) is just `result: str`, which is too thin: TASK-3 acceptance requires the per-run JSON report to record `actions (skill AND knowledge writes)`, and the curator needs a parallel audit trail. We keep both schemas flat (no nested action objects) for structured-output reliability, and let empty lists mean "nothing happened" so the model can omit them.

`SessionReviewOutput` (LLM output) and `SessionReviewResult` (orchestrator return type) live in `co_cli/agents/session_review.py`; `CuratorOutput` lives in `co_cli/agents/skill_curator.py`:

Notes:
- `summary` is the only field the foreground status callback consumes (`💾 <summary>` / `💾 Curator: <summary>`); the list fields are written into `run.json` by `_write_review_report` / `_write_curator_report` for audit trail.
- `CuratorOutput` deliberately splits `skills_merged` from `skills_consolidated`: hermes (and this plan's §Background, point 3) describes two distinct archiving paths — the state machine archives by age (logged via `transitions` in `_write_curator_report`), the consolidation agent archives by merge (this field). - `SessionReviewResult` is a frozen dataclass (not a `BaseModel`): it is the orchestrator's return type, not LLM output, so it doesn't need pydantic-ai's structured-output machinery.

```python
@dataclass(frozen=True)
class StateTransition:
    name: str
    from_state: str
    to_state: str

def apply_state_transitions(records: dict, settings: SkillsSettings, now: datetime) -> list[StateTransition]:
    """Pure: returns list of transitions; caller applies."""
    stale_cutoff = now - timedelta(days=settings.curator_stale_after_days)
    archive_cutoff = now - timedelta(days=settings.curator_archive_after_days)
    transitions = []
    for name, record in records.get("skills", {}).items():
        if record.get("pinned"):
            continue
        last_used = _parse_iso(record.get("last_used_at") or record["created_at"])
        current_state = record.get("state", "active")
        if last_used < archive_cutoff:
            target = "archived"
        elif last_used < stale_cutoff and current_state == "active":
            target = "stale"
        elif last_used >= stale_cutoff and current_state == "stale":
            target = "active"
        else:
            continue
        if target != current_state:
            transitions.append(StateTransition(name, current_state, target))
    return transitions
```

```python
def should_run_now(
    state: dict,
    settings: SkillsSettings,
    now: datetime,
    idle_seconds: float,
    *,
    bypass_time_gate: bool = False,
) -> bool:
    if not settings.curator_enabled: return False
    if state.get("paused"): return False
    # Idle gate: skip if user is actively working.
    if idle_seconds < settings.curator_min_idle_hours * 3600:
        return False
    if bypass_time_gate:  # for /skills curator run
        return True
    last_run = state.get("last_run_at")
    if last_run is None: return True
    return (now - _parse_iso(last_run)) > timedelta(hours=settings.curator_interval_hours)
```

On startup, `idle_seconds = float("inf")` (no foreground turns yet) — the idle gate always passes. On in-REPL `/skills curator run`, `idle_seconds = now - session.last_user_input_at`; if the user just typed, the gate will skip — `/skills curator run --force` (a future addition) would bypass.

```python
async def maybe_run_curator(deps: CoDeps, *, bypass_time_gate: bool = False) -> None:
    """Called from _chat_loop() in main.py as a background task, or from /skills curator run."""
    await asyncio.sleep(0)  # yield before any sync I/O (Failure Mode 13)
    state = read_curator_state(deps)
    idle_seconds = _idle_seconds(deps.session)  # inf at startup; now - last_user_input_at in REPL
    if not should_run_now(state, deps.config.skills, _utcnow(), idle_seconds, bypass_time_gate=bypass_time_gate):
        return

    logger = logging.getLogger(__name__)
    try:
        # Phase 1: pure state transitions
        records = read_records(deps)
        transitions = apply_state_transitions(records, deps.config.skills, _utcnow())
        for t in transitions:
            records["skills"][t.name]["state"] = t.to_state
            if t.to_state == "archived":
                archive_skill(deps, t.name)
        write_records(deps, records)

        child_deps = fork_deps_for_curator(deps)
        agent = build_agent(

                prompt=prompt,
                deps=child_deps,
                budget=deps.config.skills.curator_max_iterations,
                model_settings=deps.model.settings,
                role="curator",
            ),
            timeout=timeout,
        )

        # Phase 3: persist report + state

        state["last_run_at"] = _utcnow().isoformat()
        state["last_run_summary"] = output.summary
        state["run_count"] = state.get("run_count", 0) + 1
```

### Test pattern

Real construction of `CoDeps` with bundled + user skills loaded; real `skill_manage` calls in the foreground path; **stubbed agent factory** for reviewer and curator agent fork tests (inject a deterministic tool-call sequence by monkey-patching `_run_agent_standalone` to return a canned `(output, usage, run_id)` triple, so the test is independent of the model). End-to-end tests for full reviewer/curator flows use the stubbed factory plus real filesystem + real sidecar to verify state transitions and reports.

Usage tracking tests use real filesystem (`tmp_path` for `~/.co-cli` override via `CO_HOME`).

Curator race-detection tests spawn two `maybe_run_curator` tasks concurrently and assert exactly one wins (the loser sees the updated `last_run_at` and exits cleanly).

- `scripts/quality-gate.sh lint` after each task.

## Open Questions

1. **Q:** Should `pinned` live in skill frontmatter or the sidecar?  
   **Tentative answer:** Sidecar. Frontmatter for bundled skills would pollute tracked files. Sidecar is purely user-state. Per-user `pinned` semantics make sense (a skill someone pins on machine A shouldn't be pinned on machine B).

2. **Q:** Should the reviewer have its own tools (`skill_curator_search`, etc.) or share `skill_search`/`skill_view`/`skill_manage` with the foreground?  
   **Tentative answer:** Share. Adding curator-specific tools doubles the surface and adds maintenance burden. Approval bypass via `runtime.auto_approve_skill_ops` is the right scoping primitive — narrower than tool duplication.

3. **Q:** What happens if `skills.curator_enabled` flips True for the first time and the library has 500 agent-created skills with all `created_at` set to the same recent date?  
   **Tentative answer:** Phase 1 (pure transitions) is harmless — no skill is stale yet. Phase 2 (consolidation) will scan all 500. Bounded by `curator_max_iterations=100` (default; tunable up to 999) and `curator_timeout_seconds=600`. The agent only patches what it has budget for; subsequent runs continue the work.

4. **Q:** Should reviewer/curator usage cost surface in the same place as foreground usage?  
   **Tentative answer:** Yes — same model, same provider, same tracing. Logged distinctly (`source=session_review` / `source=curator` in the trace records).

5. **Q:** Should usage tracking apply to bundled skills (`use_count`/`view_count`) even though they can't be state-transitioned?  
   **Tentative answer:** No for v1. Bundled-skill usage stats would be useful telemetry but bloat the agent-created sidecar. Add a separate read-only counter table later if needed.

6. **Q:** Should the `## Drift` reflex be softened now that the reviewer catches misses?  
   **Tentative answer:** No. In-flight patching is still the right primary reflex (cheaper, faster, more accurate context). The reviewer is the safety net, not a replacement.

7. **Q:** Is `asyncio.create_task(maybe_run_curator(deps))` at bootstrap safe — what if bootstrap completes and the event loop ends before the curator finishes?  
   **Tentative answer:** The REPL's main event loop persists for the session. The curator task is scheduled into that loop. If the user exits the REPL before curator completes, cleanup awaits the task with a short timeout, then cancels. Task handle stored on `deps.session.background_curator_task` for cleanup to find.

8. **Q:** Does pydantic-ai support our `output_type=SessionReviewOutput` structured-output pattern in the delegation path?  
   **Tentative answer:** Yes — `web_research` already uses `output_type=AgentOutput` (delegation.py:213). Same pattern applies.
