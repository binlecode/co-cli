## Context

Co-cli's skill library has no per-skill usage telemetry today. Users can't see which skills they actually use, can't mark important skills as exempt from future autonomous mutation, and the system has no data layer for any future autonomous library hygiene.

This plan adds:
1. **Usage tracking sidecar** (`~/.co-cli/skills/.usage.json`) — per-skill counters and timestamps, filtered to agent-created skills (bundled and URL-installed excluded — they're upstream-managed).
2. **Manual user control** — `/skills usage` for visibility, `/skills pin <name>` / `/skills unpin <name>` for opt-out from future autonomous changes.

No LLM passes, no background tasks, no autonomous mutation. Pure mechanical hygiene.

## Problem & Outcome

**Problem.** No visibility into skill usage. No user control over future autonomous library changes. No data layer for any later hygiene mechanism.

**Failure cost.** Without per-skill usage data, the team has no signal for "which skills matter?" — needed before any autonomous pruning can be safe. Without a pin mechanism, users have no opt-out path against future autonomous mutation.

**Outcome.**
1. Sidecar tracks `use_count`, `view_count`, `patch_count`, `created_at`, `last_used_at`, `last_viewed_at`, `last_patched_at`, `state` (default `active`), `pinned` (default `false`). Bumped at hook points in `skill_view` and `skill_manage`.
2. `/skills usage` prints a table; `/skills usage <name>` prints one record.
3. `/skills pin <name>` / `/skills unpin <name>` toggles the `pinned` flag.

## Behavioural Constraints

1. **Agent-created filter is absolute.** Usage tracking applies only to user-skills-dir skills *without* `source-url`. Bundled skills (under `co_cli/skills/`) and URL-installed skills are upstream-managed. Predicate: `is_agent_created(name, deps)` checks both location and `source-url` absence.

2. **Pinned flag is a forward-compat data point.** 3.5a writes it; nothing reads it for behavior in 3.5a. 3.5b's curator will honor it. Setting/clearing `pinned` in 3.5a is a no-op in terms of system behavior; the value is captured for 3.5b consumption.

3. **Usage tracking is best-effort.** Sidecar writes are atomic (`tempfile + os.replace`); failures are logged but never block the underlying skill operation. The sidecar can be deleted at any time — it rebuilds from zero counts on next bump.

4. **No new asynchronous behavior.** 3.5a adds zero background tasks, zero LLM calls, zero coroutines beyond what the existing tools already use. All hook work is synchronous filesystem I/O.

## Failure Modes

1. **Sidecar corruption.**
   - Mitigation: atomic writes; on parse failure, log and start fresh (the sidecar is observable state, not authoritative — counts rebuild from next operation).

2. **`source-url` collision with agent-created skill.**
   - Mitigation: `is_agent_created` checks BOTH location (under `user_skills_dir`) AND `source-url` absence. A skill with `source-url` is upstream-managed even if a user copied it locally. Behavioral test covers both branches.

3. **`/skills pin` on a bundled or URL-installed skill.**
   - Mitigation: CLI handler rejects with an explanatory error. Pin/unpin is meaningful only for agent-created skills.

## High-Level Design

### Usage tracking sidecar (`co_cli/skills/usage.py`)

Schema (`~/.co-cli/skills/.usage.json`):

```json
{
  "version": 1,
  "skills": {
    "<name>": {
      "use_count": 0,
      "view_count": 0,
      "patch_count": 0,
      "created_at": "2026-05-12T14:48:40Z",
      "last_used_at": null,
      "last_viewed_at": null,
      "last_patched_at": null,
      "state": "active",
      "pinned": false
    }
  }
}
```

API:
```python
def bump_view(deps: CoDeps, name: str) -> None: ...
def bump_use(deps: CoDeps, name: str) -> None: ...
def bump_patch(deps: CoDeps, name: str) -> None: ...
def record_create(deps: CoDeps, name: str) -> None: ...
def forget(deps: CoDeps, name: str) -> None: ...
def set_pinned(deps: CoDeps, name: str, pinned: bool) -> None: ...
def is_agent_created(name: str, deps: CoDeps) -> bool: ...
def read_records(deps: CoDeps) -> dict: ...
def write_records(deps: CoDeps, data: dict) -> None: ...  # atomic
```

Hook points (in `co_cli/tools/system/skills.py`):
- `skill_view` success path → `bump_view(deps, name)`. In co-cli's flat-file model, view IS use — no separate slash-invoke path.
- `_skill_create` success → `record_create(deps, name)`
- `_skill_edit`, `_skill_patch` success → `bump_patch(deps, name)`
- `_skill_delete` success → `forget(deps, name)`
- `_skill_install` success → if installed from a local path (no `source-url`), `record_create`; URL installs skipped (upstream-managed).

All hooks are best-effort; failures logged via `logger.debug` and swallowed.

### Config (`co_cli/config/skills.py`)

```python
from pydantic import BaseModel, Field

class SkillsSettings(BaseModel):
    """Skills hygiene + (3.5b) self-evolution configuration."""

    # Dynamic knobs (lowercase snake_case, prefixed by feature area).
    usage_tracking_enabled: bool = Field(default=True)
```

**Naming convention** (held consistently across 3.5a/3.5b/3.5c):
- Dynamic Pydantic fields → lowercase snake_case, prefixed by feature area: `usage_tracking_*`, `review_*`, `curator_*`.
- Static module-level constants → UPPER_CASE, prefixed by the same feature area.
- 3.5a has zero static constants. 3.5b adds these (anticipated):
  ```python
  REVIEW_MAX_ITERATIONS = 8
  REVIEW_TIMEOUT_SECONDS = 120
  CURATOR_MIN_IDLE_HOURS = 2
  CURATOR_STALE_AFTER_DAYS = 30
  CURATOR_ARCHIVE_AFTER_DAYS = 90
  CURATOR_MAX_ITERATIONS = 100
  CURATOR_TIMEOUT_SECONDS = 600
  ```
- 3.5b adds dynamic: `review_enabled`, `curator_enabled`, `curator_interval_hours`.
- 3.5c adds dynamic: `review_nudge_interval`.

Total dynamic surface across all three plans: **4 knobs**. The principle: configurability is a contract; each knob is a test, a doc, a debugging variable. Knob earns its keep only when users will demonstrably want different values for legitimate reasons.

Wired into `co_cli/config/core.py:Settings`:
```python
skills: SkillsSettings = Field(default_factory=SkillsSettings)
```

## Open Questions

1. **Q:** Should `pinned` live in skill frontmatter or the sidecar?  
   **Answer:** Sidecar. Frontmatter for bundled skills would pollute tracked files. Sidecar is purely user-state. Per-user `pinned` semantics make sense (a skill someone pins on machine A shouldn't be pinned on machine B).

2. **Q:** Should usage tracking apply to bundled skills (`use_count`/`view_count`) even though they can't be state-transitioned?  
   **Answer:** No for 3.5a. Bundled-skill usage stats would be useful telemetry but bloat the agent-created sidecar. Add a separate read-only counter table later if needed.

- **Bundled-skill usage telemetry** (`view_count` for bundled skills, surfaced for library tuning).
- **Sidecar retention/rotation** (currently grows monotonically with skill count; cap or rotate if it becomes problematic).
