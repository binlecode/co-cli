# Plan: Canon Kind Merge — Unify Canon into Artifact Recall

Task type: `refactor`

## Context

The three-channel recall model (sessions | knowledge | canon) is architecturally inconsistent. Sessions are genuinely episodic (raw transcripts, append-only) and belong as a separate channel. Canon is not structurally different from knowledge artifacts — it is persistent, distilled, and reusable — but is treated as a third channel with its own dispatch function (`_search_canon_channel()`), its own result shape (`channel: "canon"`, `body` field, `role` field), and its own isolated config cap (`character_recall_limit`).

**Current-state validation — no doc/source inconsistencies found.** `docs/specs/memory.md` §4 accurately describes the current canon channel behavior. The plan obsoletes §4 and §5 of that spec; `sync-doc` will handle the spec update post-delivery.

The `limit` parameter formerly controlled the artifact search budget as a single count. With three prioritized passes, per-pass count caps plus a full-chunk size guard on the waterfall replaces it — otherwise priority kinds could consume the entire budget and starve background kinds.

**Workflow hygiene:** No stale exec-plans found for this slug.

## Problem & Outcome

**Problem:** Canon is dispatched via `_search_canon_channel()` — a parallel code path that duplicates FTS/hybrid search logic already in `_search_artifacts()`. It cannot be filtered via the `kinds` parameter, cannot participate in a unified priority/cap structure, and produces a result shape inconsistent with artifact results.

**Failure cost:** Callers cannot request `kinds=['canon']`; canon results are not part of artifact priority budgeting; adding new per-kind caps requires adding new dispatch functions instead of constants.

**Outcome:** `_search_canon_channel()` is deleted. Canon flows through `_search_artifacts()` as `kind='canon'`. Four module-level constants govern priority caps and waterfall dual-caps. The two-channel model (sessions + artifacts) replaces the three-channel model.

## Scope

- **In:** `ArtifactKindEnum`, `sync_dir()` kind auto-set, `_search_artifacts()` rewrite with three passes (canon priority, user priority, waterfall), `_search_canon_channel()` deletion, `memory_search()` merge cleanup, `_format_search_display()` adaptation, `character_recall_limit` deprecation comment, test rewrites, eval update.
- **Out:** Session channel (untouched), `list_artifacts()` (canon excluded from browse/list intentionally — read-only, not user-managed), canon frontmatter files (kind injected at index time, no file edits needed), hybrid/FTS5 backend logic (unchanged).

## Behavioral Constraints

1. Canon hits return full body — `snippet` field carries full chunk content (not FTS5 40-token fragment). This holds because `no_chunk=True` stores the entire MD body as chunk[0]; fetching chunk[0] content gives the full body.
2. Canon pass: at most `_ARTIFACTS_CANON_CAP = 3` items per `memory_search` call.
3. User pass: at most `_ARTIFACTS_USER_CAP = 3` chunk results per `memory_search` call.
4. Waterfall pass: stops at `_ARTIFACTS_WATERFALL_CHUNK_CAP = 5` count OR `_ARTIFACTS_WATERFALL_SIZE_CAP = 2000` cumulative full-chunk-content chars (fetched via `get_chunk_content()`), whichever triggers first. The FTS5 `snippet` (displayed fragment) is NOT used for size measurement — full chunk content at up to `chunk_size=600` chars is.
5. When `personality` is None or `store` is None, canon results are empty — same guard as current.
6. Canon items carry `channel='artifacts'`, `kind='canon'` in result dicts.
7. `kinds=['canon']` isolates canon-only results; `kinds=['user']` correctly excludes canon.
8. Waterfall searches only kinds not covered by priority passes: when `kinds` is None, waterfall searches `['rule', 'article', 'note']`; when `kinds` is specified, waterfall searches `kinds - {'canon', 'user'}`.
9. Grep fallback (store=None): knowledge artifacts still retrieved by grep; `'canon'` silently excluded from grep kinds since canon files live in the package, not `knowledge_dir`.
10. `_SESSIONS_CHANNEL_CAP = 3` and session channel behavior are unchanged.

## High-Level Design

### Pass structure inside `_search_artifacts()`

```
_search_artifacts(ctx, query, kinds, limit):
    results = []

    # Canon priority pass
    if kinds is None or 'canon' in kinds:
        canon_hits = store.search(sources=['canon'], limit=_ARTIFACTS_CANON_CAP)
        for hit in canon_hits:
            body = store.get_chunk_content('canon', hit.path, 0)
            results.append({channel:'artifacts', kind:'canon', snippet:body, ...})

    # User priority pass
    if kinds is None or 'user' in kinds:
        user_hits = store.search(sources=['knowledge'], kinds=['user'], limit=_ARTIFACTS_USER_CAP)
        results.extend(user_hits as channel='artifacts' dicts)

    # Waterfall pass (rule / article / note — or caller-specified non-priority kinds)
    # list(set(...) - set) avoids TypeError; empty guard prevents invalid SQL (kind IN ())
    waterfall_kinds = list(set(kinds or ['rule', 'article', 'note']) - {'canon', 'user'})
    if waterfall_kinds:  # only call store.search when non-empty
        candidates = store.search(sources=['knowledge'], kinds=waterfall_kinds,
                                  limit=_ARTIFACTS_WATERFALL_CHUNK_CAP)
        total_chars = 0
        for r in candidates:
            full_content = store.get_chunk_content(r.source, r.path, r.chunk_index or 0) or ''
            if total_chars >= _ARTIFACTS_WATERFALL_SIZE_CAP:
                break
            results.append({channel:'artifacts', ..., snippet:r.snippet})
            total_chars += len(full_content)

    return results
```

### `_format_search_display()` canon rendering

Canon items (identified by `kind='canon'`) are separated from other artifact results and rendered with full body under a `**Character canon:**` header — same visual output as today, driven by `kind` instead of `channel`.

### Grep fallback path

When `store` is None, the existing `load_artifacts()` + `grep_recall()` path continues. Canon kind is filtered out of `grep_kinds` since canon files are not in `knowledge_dir`. No priority/waterfall structure is applied in the degraded path.

### TASK-2 — Recall layer: unified `_search_artifacts()`, remove `_search_canon_channel()`, update `memory_search()` and `_format_search_display()`

**files:**
- `co_cli/tools/memory/recall.py`
- `tests/test_flow_canon_recall.py`

**prerequisites:** [TASK-1]

**what:**
1. Add four constants at module top: `_ARTIFACTS_CANON_CAP`, `_ARTIFACTS_USER_CAP`, `_ARTIFACTS_WATERFALL_CHUNK_CAP`, `_ARTIFACTS_WATERFALL_SIZE_CAP`.
2. Delete `_search_canon_channel()`.
3. Rewrite `_search_artifacts()` with the three-pass structure described in the design section. Waterfall kinds: `waterfall_kinds = list(set(kinds or ['rule', 'article', 'note']) - {'canon', 'user'})`. Guard: only call `store.search(sources=['knowledge'], kinds=waterfall_kinds, ...)` when `waterfall_kinds` is non-empty — an empty list would produce `kind IN ()` which is invalid SQL and raises `sqlite3.OperationalError`. For size accumulation: fetch `store.get_chunk_content(r.source, r.path, r.chunk_index or 0)` for each waterfall candidate; accumulate its length against `_ARTIFACTS_WATERFALL_SIZE_CAP`; put `r.snippet` (FTS5 fragment) in the result dict for display. For grep fallback: filter out `'canon'` from kinds before calling `load_artifacts()`; if only `'canon'` was requested, return `[]`.
4. In `memory_search()`: remove `canon_results = _search_canon_channel(ctx, query)` and remove canon from the result merge. The existing `knowledge_results` assignment becomes `_search_artifacts(ctx, query, kinds, limit)` which now includes all three passes.
5. In `_format_search_display()`: replace `char_canon_results = [r for r in all_results if r["channel"] == "canon"]` with `canon_artifact_results = [r for r in artifact_results if r.get("kind") == "canon"]` and `non_canon_artifact_results = [r for r in artifact_results if r.get("kind") != "canon"]`. Render non-canon artifacts first (snippet truncated), then canon under `**Character canon:**` header with full `snippet` (no truncation).
6. Rewrite `tests/test_flow_canon_recall.py`: tests call `_search_artifacts()` directly (not `_search_canon_channel()`); assert `result[0]["channel"] == "artifacts"`, `result[0]["kind"] == "canon"`, full body in `result[0]["snippet"]`; assert cap honored (len <= `_ARTIFACTS_CANON_CAP`); assert empty when store=None or personality=None.

**done_when:** `uv run pytest tests/test_flow_canon_recall.py tests/test_flow_bootstrap_canon.py -x` passes. Canon hits carry `kind='canon'` and `len(snippet) > 200` (full body, not FTS snippet — add comment: `# real tars humor file body is ~700 chars; FTS5 snippet() would cap at ~200`).

**success_signal:** `memory_search(ctx, "humor deadpan")` returns canon hits in `results` with `channel='artifacts'`, `kind='canon'`, and full scene text in `snippet`; rendered output still shows `**Character canon:**` section with full scene body.

### TASK-3 — Waterfall dual-cap test

**files:**
- `tests/test_flow_artifacts_waterfall_cap.py`

**prerequisites:** [TASK-2]

**what:** New test file. Two scenarios, two test functions:

Scenario A (count cap): Seed 6 rule artifacts with short bodies (~100 chars each, well under `_ARTIFACTS_WATERFALL_SIZE_CAP` in aggregate). Index with `chunk_size=600`. Call `_search_artifacts()` with `kinds=['rule']` and a matching query. Assert `len(results) <= _ARTIFACTS_WATERFALL_CHUNK_CAP` (count cap fires).

Scenario B (size cap): Seed 6 note artifacts each with a 500-char body (use `chunk_size=600` in the test store config so each artifact produces exactly one chunk of ~500 chars). Call `_search_artifacts()` with `kinds=['note']` and a matching query. Assert `len(results) == 4` — full-chunk size accumulation reaches 2000 (4 × 500) before the count cap of 5 fires (checked via `get_chunk_content()`). Note: the FTS5 `snippet` in results remains a short fragment; this scenario verifies that size is measured against full chunk content, not snippets.

**done_when:** `uv run pytest tests/test_flow_artifacts_waterfall_cap.py -x` passes; `test_waterfall_count_cap` and `test_waterfall_size_cap` both present and green; `assert len(results) == 4` in the size-cap test is explicit.
