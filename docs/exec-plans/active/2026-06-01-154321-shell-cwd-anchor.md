## Context

The `file-search-roots` review surfaced that `shell_exec` anchored its cwd to a *different* variable
than the file write tools. **That divergence has already been fixed** (atomic change shipped
2026-06-01, outside any plan):

- `shell_exec` now resolves both its `workdir` boundary and its default cwd against
  `ctx.deps.workspace_dir` — the single intended write/cwd anchor — and always passes an explicit
  cwd, so a configured `workspace_path` takes effect even when no `workdir` is given
  (`co_cli/tools/shell/execute.py`).
- `ShellBackend` is now **stateless** — the `workspace_dir` constructor param + field were removed
  (orphaned once `shell_exec` stopped reading them and always passes an explicit cwd);
  `run_command` takes cwd per call (`cwd=None` → subprocess inherits the parent cwd). This was the
  near-orphan flagged below as Q1 — **resolved inline** during the post-fix anti-pattern cleanup
  rather than deferred, so there is now exactly one source of truth for the cwd
  (`co_cli/tools/shell_backend.py`, `co_cli/bootstrap/core.py` back to `ShellBackend()`).
- Regression coverage: `tests/test_flow_shell_exec.py` (runs `pwd` in workspace, sub-dir scoping,
  traversal-escape rejection — real subprocess, no mocks).

This plan does **not** re-litigate that fix. After Q1 was resolved inline, **one** decision remains
open: `cd` persistence (Q2). Q3 is recorded as YAGNI.

### Peer survey (corrected after a deeper second read)

The decision question is narrow: **does a model-issued `cd` in one call carry to the next call?**
On that precise question the peers split — the majority are stateless, not persistent. (An earlier
loose first-pass read recorded "both peers persist"; that conflated *sandbox/filesystem reuse* with
*cwd tracking `cd`*. Corrected below.)

| Repo | Mechanism | Model-issued `cd` persists across calls? | Evidence |
|------|-----------|------------------------------------------|----------|
| openclaw | fresh subprocess per call; cwd snapshot fixed at session creation, never updated post-run | **No — stateless** | `bash-tools.exec-runtime.ts:640`; no `session.cwd = …` after exec |
| hermes-agent | per-call cwd = `overrides.get("cwd") or config["cwd"]` — a fixed task cwd; no `pwd` re-read. Only an **external** orchestrator (RL/ACP) can change it via `register_task_env_overrides`. Sandbox/filesystem persists, the cwd does not track `cd`. | **No** (for model `cd`) | `terminal_tool.py:1739`, `register_task_env_overrides` (~`:936`) |
| fork-claude-code | global cwd state + `AsyncLocalStorage`; re-reads `pwd -P` after each cmd and writes drift back | **Yes — true auto-persistence** | `cwd.ts`, `Shell.ts:218,395` |
| co-cli (post-fix) | fresh `sh -c` per call, cwd fixed to `deps.workspace_dir` | **No — stateless per call** | `shell/execute.py`, `shell_backend.py` |

**So 2A (stateless) is the majority peer pattern.** openclaw and hermes both leave a model `cd`
local to its own call — exactly co's behavior. Only fork-claude-code truly persists `cd` drift, and
it pays for it with global/async-local cwd state it needs for *concurrent-agent isolation* — a
problem co does not have. co is local-first, single-workspace, approval-first.

## Problem & Outcome

**Problem:** one open question remains after the anchor fix + inline cleanup; not blocking:

1. **(RESOLVED inline — orphan removed.)** `ShellBackend.workspace_dir` was a second source of truth
   for "the workspace": `shell_exec` (sole caller of `run_command`) stopped reading it post-fix, so
   it survived only as a never-triggered `cwd or self.workspace_dir` fallback. Removed during the
   post-fix cleanup — `ShellBackend` is now stateless and `deps.workspace_dir` is the lone anchor.
   No further work; left here for traceability.

2. **co has no `cd` persistence.** Every `shell_exec` call starts fresh at `workspace_dir` (or the
   given `workdir`). A model that runs `cd build && ...` in one call sees no effect on the next.
   This matches openclaw and hermes (both stateless for model `cd`); only fork-claude-code
   auto-persists. So it is a minor capability gap, not a bug, and co is already aligned with the
   peer majority. Whether co *wants* full persistence is a product
   call (it trades simplicity/predictability for shell-session realism).

**Outcome:** a decided, documented shell-cwd model — single source of truth for the anchor, and an
explicit ruling on `cd` persistence (adopt with a defined mechanism, or reject with rationale
recorded so it stops resurfacing).

**Failure cost:** none silent. Without this, the `cd`-persistence question keeps re-surfacing in
reviews. Nothing breaks.

## Open questions (resolve at Gate 1)

1. **Remove `ShellBackend.workspace_dir`? — RESOLVED (A, done inline).** Removed during the post-fix
   cleanup: `run_command`'s `cwd=None` inherits the parent cwd via `create_subprocess_exec`, and
   `shell_exec` always passes an explicit cwd, so the fallback was dead for the only caller. All
   `ShellBackend(...)` sites already used the no-arg form. `deps.workspace_dir` is now the lone
   anchor; the backend is stateless. No Gate-1 action needed.

2. **Adopt `cd` persistence across `shell_exec` calls? — SETTLED: 2A (reject; stay stateless).**
   - **(A) Reject — stay stateless per call. ✓ CHOSEN.** Matches co's simplicity/predictability bias;
     each call is reproducible and approval-legible (the model can't silently relocate the shell), and
     it matches the peer majority (openclaw + hermes are stateless for a model `cd`). Documented in
     `docs/specs/tools.md` so it does not re-surface. See TASK-2.
   - **(B) Adopt with an explicit mechanism.** *(Not chosen.)* Persist resolved cwd after each
     foreground command (re-read `pwd -P`, write back to session state), with a BC-1 re-containment
     check on every persisted cwd or drift escapes the boundary. Bigger change; revisit only if a
     concrete workflow needs interactive `cd` persistence.

3. **Is the shell anchor ever legitimately *different* from `workspace_dir`?** (e.g. a future
   `shell_workdir` config for running builds outside the file-write root.) Default expectation: **no**
   — the `file-search-roots` design settled `workspace_dir` as the single write/cwd anchor and
   per-root `mode: rw|ro` was ruled YAGNI. Confirm or open as its own future plan; do not pre-build.
