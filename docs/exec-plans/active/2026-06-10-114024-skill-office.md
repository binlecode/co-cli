# Skill — `office` (docx / pptx / xlsx extraction over `shell_exec`)

Task type: code + skill content

> **Built on the folder-skill model** (every skill is a `<name>/SKILL.md` folder that can carry
> bundled assets), shipped in **v0.8.332**
> (`docs/exec-plans/completed/2026-06-09-171515-skill-folder-model.md`). **Sibling** of
> `2026-06-09-093734-skill-documents.md` (PDF). This plan adds `office/SKILL.md` +
> `office/scripts/extract_office.py`. Split out of the documents plan: PDF (pymupdf4llm,
> page-chunk citations) and Office (OOXML, slide/sheet/heading structure) are different
> extraction backends and earn separate skills.

## Context

co is blind to Office documents the same way it was blind to PDFs: `file_read` rejects binary,
so a `.docx`/`.pptx`/`.xlsx` is an unconditional "I can't." This is on-mission ("Personal
Assistant CLI") — Office files are second only to PDFs as the local artifact a user hands an
assistant. It rides the same rails as the PDF skill: a lean dep + a thin subprocess script + a
bundled skill, **no new tool, no model-multimodal plumbing**.

This is the format-breadth half of the original `documents` plan, promoted to its own skill
(per the split decision). The PDF skill keeps pymupdf4llm for its page-chunk citation contract;
Office formats have no stable page model (slides, sheets, reflowing prose), so they get their
own backend and citation contract.

**Why a skill + script, not a tool** — identical to the PDF plan: zero model-visible surface,
zero `categories.py`/`toolset.py` churn, subprocess isolation (a crash in an OOXML parser on a
malformed file cannot touch the agent heap). Helper ships **inside the skill folder** at
`co_cli/skills/office/scripts/extract_office.py`, invoked as a module
`uv run python -m co_cli.skills.office.scripts.extract_office <file>` (resolves via `sys.path`
on any `work_dir` / installed wheel), reached only through `shell_exec`. `office/` and
`office/scripts/` carry docstring-only `__init__.py` so `-m` resolves.

### Hermes parity reference (code-verified 2026-06-10)

Hermes does **not** use one unified converter for Office. It uses **format-specific** libraries:
`python-docx` for Word (`skills/.../ocr-and-documents/SKILL.md:16`), a dedicated `powerpoint`
skill on `python-pptx` (`skills/productivity/powerpoint/SKILL.md`), and mentions
`pip install "markitdown[pptx]"` only for plain pptx text extraction (`powerpoint/SKILL.md:233`).
None are installed by default — all are manual `pip`. **co's divergence:** ship the dep by
default (out-of-box UX, as in the PDF skill) and prefer a **single unified backend** if its
footprint clears the lean bar — see Open Question 1 (markitdown-unified vs format-specific).
All extraction is subprocess, matching co's design.

## Problem & Outcome

**Problem.** co cannot read `.docx`/`.pptx`/`.xlsx`. "Summarize this deck / contract /
spreadsheet" is an unconditional "I can't," blocked only by a missing dependency + a thin script.

**Failure cost:** the assistant silently cannot serve a top-tier local-document request; the
user must convert to text by hand and paste it back. No error teaches a workaround.

**Outcome.**
1. A lean OOXML extraction dependency added (markitdown office extras, or format-specific libs
   per OQ1) — **no torch/CUDA**.
2. `co_cli/skills/office/scripts/extract_office.py` — subprocess CLI module: docx/pptx/xlsx →
   markdown with format-appropriate structure markers (slides, sheets, headings), plain stderr
   + non-zero exit on errors.
3. `co_cli/skills/office/SKILL.md` — locate → extract → answer; routes `.pdf` to the
   `documents` skill, other binaries to `file_read`.
4. Real-file tests + committed fixtures (one tiny each: docx, pptx, xlsx); bundled gate green
   (`office` in `_BUNDLED_NAMES`).
5. Ships once it passes lint + the bundled gate + the new tests (folder-skill model already in code).

## Behavioral Constraints
1. **Substrate gate.** Body references only `file_search`, `web_fetch`, `file_read`,
   `shell_exec` (+ the bundled `office/scripts/extract_office.py` via `-m`), and routes `.pdf`
   to the `documents` skill. No `file_find`.
2. **Marginal-value gate.** Multi-step locate → extract → answer with a by-extension decision tree.
3. **Native body.** co tool names + co's own module only; no `markitdown`/hermes script names
   leaking into the prose (the script wraps the library so the body stays native).
4. **Process isolation (hard).** Extraction runs via `shell_exec` subprocess (`python -m …`),
   never imported into the agent process.
5. **Approval transparency.** First extraction prompts approval (`python` not in safe_commands);
   session-subject auto-approval thereafter (`co_cli/context/orchestrate.py`). The body describes
   this honestly and does not oversell the safe_commands opt-in (`_FORBIDDEN_ARG_PATTERNS` still
   rejects `/`, `~`, `..` args, so non-workspace-root paths re-prompt by design).
6. **Graceful failure.** The script exits non-zero with a plain one-line stderr message (no
   traceback) for: missing file, unsupported extension, corrupt/unparseable OOXML, and the
   **password-protected/encrypted** case (its own distinct message), so the user learns the cause.
7. **Lean dependency.** OOXML extraction only; **no torch/CUDA** in the resolved tree (verifiable
   gate, as in the PDF plan). markitdown's `magika`/onnxruntime (~tens of MB) and the xlsx-extra
   `pandas` are acknowledged, not torch-scale; OQ1 weighs the format-specific lean alternative.
8. **Read-only synthesis.** No write tools.
9. **Bundled, test-gated.** Lands in `co_cli/skills/office/SKILL.md`; passes lint R1–R4 + B1 +
   security scan; registered in `_BUNDLED_NAMES`.

## High-Level Design

Three layers, mirroring the PDF skill:

1. **Dependency layer** — the OOXML extraction backend. Primary candidate: `markitdown` with
   `docx,pptx,xlsx` extras (one dep, unified markdown output). The lean constraint is the same
   verifiable bar as the PDF plan: marker-pdf/torch/CUDA must not appear in the resolved tree.
   The footprint markitdown *does* add (magika→onnxruntime, pandas via the xlsx extra) is
   measured and recorded against a documented ceiling. OQ1 holds the format-specific alternative
   (mammoth/python-pptx/openpyxl — ML-free, no onnxruntime, no pandas) as the fallback if the
   ceiling is breached.

2. **Extraction layer** — `co_cli/skills/office/scripts/extract_office.py`, a stdlib-`argparse`
   CLI invoked as `uv run python -m co_cli.skills.office.scripts.extract_office <path>`. It
   **dispatches by extension** (`.docx`/`.pptx`/`.xlsx`), calls the backend, and emits markdown
   with **format-appropriate structure markers**: `## Slide N` for pptx, `## Sheet <name>` for
   xlsx (sheets as markdown tables), and the document's own markdown headings for docx (no
   synthetic markers — docx has no page model). On any failure it prints a one-line message to
   stderr and exits non-zero (no traceback). Ships inside the skill folder, never imported into
   the agent process — reached only as a `shell_exec` subprocess.

3. **Skill layer** — `co_cli/skills/office/SKILL.md`, a Locate → Extract → Answer overlay.
   Locate uses `file_search` or a user path (Drive route optional + config-gated, as in the PDF
   skill). Extract branches by extension (`.docx`/`.pptx`/`.xlsx` → the script; `.pdf` → defer to
   the `documents` skill; URL → `web_fetch`; other local → `file_read`). Answer synthesizes with
   **slide / sheet / heading** citations from the structure markers. Registered bundled by adding
   `office` to `_BUNDLED_NAMES`.

The two skills are reciprocal: `documents` routes Office formats here; `office` routes `.pdf`
back. Each owns one backend and one citation contract.
