# repl-input-queue

## Context

Co's REPL chat loop is single-threaded on the input side: while `run_turn()` is executing, `prompt_toolkit`'s `prompt_async()` is not running, so users cannot type the next prompt until the current turn completes (`co_cli/main.py:509–554`). Peer agent CLIs (codex, gemini-cli, hermes-agent) all expose a typed-ahead input queue owned by the chat-loop layer (see `docs/reference/RESEARCH-input-queue.md`).

The research doc converged on a **codex-style explicit FIFO queue, owned by the REPL session, draining one item per turn boundary**. A separate decisions pass (D1–D9 plus D10–D14 verified from source on 2026-05-23) settled scope and concurrency model. This exec-plan is the implementation companion.

Architectural framing: the queue upgrades co's main agent into a **REPL session agent** — producer/consumer split inside a session whose lifecycle stays owned by `_chat_loop`'s outer scope. Distinct from the existing **dream daemon** (separate process, file-backed disk queue).

## Problem & Outcome

**Problem.** While a turn runs, the REPL is mute: keystrokes are not buffered (prompt_toolkit isn't reading), so type-ahead is lost. Long turns force the user to wait visibly idle.

**Outcome.** All submissions during an active turn — plain text and slash commands alike — enqueue; submissions while idle execute immediately. FIFO drain; one queued item per completed turn. Queue depth is shown in the bottom toolbar. Ctrl+C cancels the current turn but preserves the queue; second Ctrl+C within 2s exits. The prompt remains visible only when idle in Phase 1 (today's behavior); the queue still buffers because keystrokes typed before the next prompt rendering are captured by the producer task as soon as the consumer signals turn-end.

**Failure cost.** Without this fix, typed-ahead input is silently dropped on every long-running turn — users repeatedly type into a void, then re-type after the turn lands. Pure UX regression vs. every peer agent CLI. The longer turns get (with reasoning models, deeper tool chains), the worse the gap.

## Scope

**In (Phase 1):**
- Producer / consumer split inside `_chat_loop` via `asyncio.TaskGroup`.
- In-memory `asyncio.Queue[str]` for user-typed input (plain text and slash commands alike).
- `queue_depth: int` field added to `StatusSnapshot`; rendered in bottom toolbar as the segment `"{n} queued"` for `n > 0` (omitted when zero), inserted between `mode` and `ctx`.
- Producer-cancel + restart coordination around synchronous frontend prompts (`prompt_approval`, `prompt_question`, `prompt_confirm`): consumer cancels producer task → awaits completion → runs sync prompt → restarts producer in the same `TaskGroup`.
- Ctrl+C semantics via prompt_toolkit `KeyBindings`: first press while turn running → cancel running turn task, preserve queue; second press within 2s → set `stop_event` and exit (queue dropped, but `_drain_and_cleanup` still fires).

**Out (deferred or removed):**
- Slash-command bypass surface entirely (`bypass_queue` flag, `/queue` inspect, `CommandContext.input_queue`, `transcript_inbox`, producer-side dispatch). Deferred to Phase 2; resolves Core Dev's data-race hazard against `deps.runtime` / `deps.session` and PO's scope-bundling concern.
- Always-live prompt via `prompt_toolkit.patch_stdout`. Deferred to a separate Phase 2 plan with `rich.live.Live` interop research up front; today's Rich `Console` captures `sys.stdout` at module import, so `patch_stdout` cannot proxy the four `Live` instances `TerminalFrontend` runs.
- `/queue clear` and `/queue pop` (Phase 2).
- Gemini-style "collect" merge mode (Phase 2).
- Hermes-style mid-turn interrupt/steer mode (Phase 3 — requires primitives co doesn't have).
- File-backed queue persistence (Phase 3).
- `InputQueue` wrapper class / non-stdin producers (added when a second producer ships).

## Behavioral Constraints

C1. **`run_turn()` contract is unchanged.** Single-turn, event-emitting, accepts one `user_input: str`. No queue logic inside the agent loop. (Source: `co_cli/context/orchestrate.py:run_turn`.)

C2. **Producer never blocks waiting for the consumer's turn to finish.** Producer reads stdin, enqueues, loops. If consumer is busy, queue grows; no producer-side gating on "is a turn active". The queue is the back-pressure layer. (Note: during a sync `prompt_approval`/`prompt_question`/`prompt_confirm`, the producer **is** suspended by an explicit cancel-and-restart cycle — see C4. That is a stdin-arbitration mechanism, not back-pressure.)

C3. **Consumer drains one item per turn boundary.** No batching, no merging. Each `await queue.get()` is followed by exactly one `_handle_one_input(...)` call. Empty / whitespace strings are dropped at the producer (no enqueue) — matches current `co_cli/main.py:366` no-op.

C4. **Sync frontend prompts require producer cancel + restart.** When `frontend.prompt_approval` (or `prompt_question`, `prompt_confirm`) is about to run from inside `run_turn()`:
   - Consumer calls `producer_task.cancel()` and **awaits its completion** — this lets prompt_toolkit's `prompt_async` finally-block restore termios cooked-mode and release stdin cleanly.
   - Consumer runs the synchronous `Prompt.ask` (existing code path unchanged).
   - Consumer schedules a new producer task in the same `TaskGroup` (`tg.create_task(_producer_task(...))`).
   - Rationale: a shared `asyncio.Event` cannot hand stdin off because prompt_toolkit's `Input` class still holds `/dev/tty` in raw mode while the producer task is suspended in `prompt_async`. Cancel + restart is the only safe pattern. (CD-M-2.)

C5. **All input enqueues — including slash commands.** In Phase 1 there is no bypass surface. Slash commands typed mid-turn enqueue and run when the consumer drains them. Eliminates data races against `deps.runtime` / `deps.session.*` mutated inside `run_turn()` (CD-M-4).

C6. **Ctrl+C semantics, implemented via prompt_toolkit `KeyBindings`:**
   - Press 1, idle: prompt_toolkit's default `c-c` behavior — clear input line (no change vs today).
   - Press 1, turn running: key binding callback cancels consumer's current `_run_foreground_turn` task (CancelledError propagates inside `run_turn`); queue preserved; `last_interrupt_time = monotonic()`.
   - Press 2 within 2s of press 1: key binding sets `stop_event`; `_chat_loop`'s `TaskGroup` unwinds; queue dropped; `_drain_and_cleanup` runs.
   - Window measured from first press, not from cancel completion.
   - Rationale: `loop.add_signal_handler(SIGINT, ...)` is incompatible with `prompt_async`, which installs its own SIGINT handler on every entry and removes it on exit (CD-M-1). Routing through `KeyBindings` on the `PromptSession` is the only stable mechanism.
   - The existing `signal.signal(SIGINT, default_int_handler)` swap inside `TerminalFrontend.prompt_approval` / `prompt_question` (`display/core.py:524-525, 550-551`) stays as-is — it runs synchronously while the producer is cancelled (C4), so there's no asyncio-handler collision.

C7. **Queue depth display is authoritative.** Single source: `asyncio.Queue.qsize()`. Producer triggers a `frontend.update_status(...)` on enqueue; consumer triggers one on dequeue. No depth caching elsewhere. Toolbar segment is `"{n} queued"` for `n > 0`, omitted otherwise; segment order is `session_label · mode · {n} queued · ctx · {bg} bg · {n} approval(s)`.

C8. **Queue lifecycle = session lifecycle.** Queue is constructed inside `_chat_loop` after `create_deps`, destroyed in `finally` before `_drain_and_cleanup`. Not part of session persistence; lost on process exit.

C9. **Consumer-side exceptions preserve the queue.** A non-cancel exception inside `_handle_one_input` / `run_turn` (LLM error, tool error, etc.) is caught by the existing `except Exception as e:` arm at `co_cli/main.py:552-554`, the error is surfaced to the user, and the next iteration `await queue.get()`s the next item. The queue is not drained or reset by error paths. (PO-m-2.)

C10. **`Frontend` protocol stays stable.** No new methods. New state (`queue_depth`) flows via existing `StatusSnapshot`. `HeadlessFrontend` requires no changes beyond the new dataclass field default.

C11. **Failure modes to guard against:**
   - Input lost during cancel/restart of producer → cover with test: typed-ahead during approval round-trip survives.
   - Queue depth drift (display stale after rapid enqueue/dequeue) → cover with test: depth observed via `update_status` matches `qsize` after each transition.
   - Double Ctrl+C race (second press arrives during cancel) → cover with test using injected clock.
   - Producer task fails to restart after sync prompt → integration test asserts producer is alive after a mid-turn approval; if it isn't, the loop is silently broken.

## High-Level Design

```
   ┌─────────────── _chat_loop (session scope) ─────────────────┐
   │                                                            │
   │   asyncio.Queue[str]                                       │
   │   stop_event: asyncio.Event (clear)                        │
   │   ctrl_c key binding on PromptSession                      │
   │                                                            │
   │   async with TaskGroup() as tg:                            │
   │     producer = tg.create_task(_producer_task(...))         │
   │     consumer = tg.create_task(_consumer_task(...))         │
   │                                                            │
   │     _producer_task  (loop):                                │
   │       line = await session.prompt_async(...)               │
   │       if blank: continue                                   │
   │       await queue.put(line)                                │
   │       update_status                                        │
   │                                                            │
   │     _consumer_task  (loop):                                │
   │       if stop_event.is_set(): break                        │
   │       line = await queue.get(); update_status              │
   │       turn_task = asyncio.current_task() side-handle       │
   │       try:                                                 │
   │         await _handle_one_input(line, ...)                 │
   │       except CancelledError: pass  # ctrl+C cancel turn    │
   │       around sync prompts (handled by TerminalFrontend     │
   │       via injected callbacks):                             │
   │         producer.cancel(); await producer                  │
   │         <sync Prompt.ask>                                  │
   │         producer = tg.create_task(_producer_task(...))     │
   │                                                            │
   │     ctrl+c key binding:                                    │
   │       if turn_task is running:                             │
   │         turn_task.cancel(); last_interrupt_time = now      │
   │       elif now - last_interrupt_time < 2:                  │
   │         stop_event.set()                                   │
   │       else:                                                │
   │         <prompt_toolkit default: clear line>               │
   │                                                            │
   └────────────────────────────────────────────────────────────┘
```

**Producer responsibilities.**
- Read one line via `session.prompt_async()`.
- Drop blanks.
- Enqueue every non-blank line (plain text or slash command) into the input queue.
- Update `StatusSnapshot.queue_depth` after each enqueue.
- On `CancelledError`, let it propagate — prompt_toolkit's `prompt_async` `finally` restores termios cooked-mode.

**Consumer responsibilities.**
- `await queue.get()`; update status; run `_handle_one_input(line, ...)`.
- Hold a reference to the in-flight `_run_foreground_turn` task so the Ctrl+C key binding can cancel it.
- Around sync stdin calls (approval/question/confirm), perform the cancel-and-restart sandwich on the producer (C4). This is wired via `input_arbitration_callbacks` set on the `TerminalFrontend` by `_chat_loop` after construction — see "Approval coordination details" below.

**Approval coordination details.**
- Phase 1 uses **coarse cancel/restart at the `_run_foreground_turn` boundary** — see TASK-3. The consumer cancels the producer task at turn start, awaits its completion, runs the entire turn (including any sync `prompt_approval`/`prompt_question`/`prompt_confirm` calls deep inside `run_turn`), then schedules a fresh producer task in the same `TaskGroup` after the turn completes. This matches today's UX (prompt hidden during turns, today via `set_input_active(False)`) while gaining the queue behind it. No `TerminalFrontend` change required.
- Phase 2 refinement (deferred, separate plan): finer-grained per-approval-call cancel via a callback pair (`input_arbitration_callbacks`) on `TerminalFrontend`. This unlocks an always-live prompt during non-approval portions of a turn, but introduces sync/async impedance (the callback must await a cancellation from a sync `Prompt.ask` call path) that requires either a `threading.Event` spike or restructuring `orchestrate.py` to expose an async approval wrapper. Out of scope for Phase 1.

**Ctrl+C wiring (via prompt_toolkit KeyBindings).**
- Construct a `KeyBindings()` instance with a single `@kb.add('c-c')` handler attached to the `PromptSession`.
- Handler reads shared state (`last_interrupt_time`, current `turn_task`, `stop_event`) via closure over `_chat_loop` locals.
- No asyncio signal handler is installed. No competition with prompt_toolkit's internal SIGINT path or with `display/core.py`'s sync signal swap during `Prompt.ask`.

### TASK-4: Ctrl+C handling via prompt_toolkit `KeyBindings`

**Motivation.** Today: a single Ctrl+C raises through `prompt_async` and triggers `_handle_one_input`'s double-press timer to exit. New: when the consumer holds a running turn task, first Ctrl+C must `.cancel()` that task **without setting `stop_event` and without clearing the queue** — queued items must survive a mid-turn cancel. This is the only Phase 1 behavior change to Ctrl+C semantics, and it exists specifically because the queue makes "cancel turn, keep queue" meaningful (today there's no queue to preserve, so all variants collapse to "exit").

- files:
   - `co_cli/main.py` — define an **extractable** module-level callable `_handle_ctrl_c(turn_task_ref, deps, stop_event, now: float) -> Literal["cancelled", "exiting", "default"]` that takes the four inputs explicitly and returns a verdict. Construct `KeyBindings()`; register `@kb.add('c-c')` as a thin wrapper that reads closure-captured `turn_task_ref`/`deps`/`stop_event`, calls `_handle_ctrl_c(...)`, and acts on the verdict (consume vs propagate). Pass `key_bindings=kb` to the `PromptSession(...)` constructor (line 459).
   - Verdict mapping inside `_handle_ctrl_c`:
      - If `turn_task_ref` is non-None and `not turn_task_ref.done()`: `turn_task_ref.cancel()`; `deps.session.last_interrupt_time = now`; return `"cancelled"`.
      - Elif `now - deps.session.last_interrupt_time < 2.0`: `stop_event.set()`; return `"exiting"`.
      - Else: return `"default"` (caller lets prompt_toolkit handle).
   - Consumer holds and exposes `turn_task_ref` via a small mutable cell (e.g. `[None]`-style list or a one-attribute dataclass) shared by reference with the key binding.
   - Update consumer's `_run_foreground_turn` call site to wrap in `try / except asyncio.CancelledError` so first-Ctrl+C cancellation surfaces as "turn cancelled" without bringing down the consumer loop.
   - `tests/test_flow_chat_loop.py` — add with injected `now` clock argument:
      - `test_ctrl_c_cancels_turn_preserves_queue` — push 2 items into queue; build a fake `turn_task` (not done); call `_handle_ctrl_c(turn_task, deps, stop_event, now=10.0)` directly (no prompt_toolkit involved); assert verdict is `"cancelled"`, `turn_task.cancel()` was invoked, queue still has 2 items, `stop_event` is clear.
      - `test_double_ctrl_c_sets_stop_event` — call `_handle_ctrl_c(None, deps, stop_event, now=10.0)` then `_handle_ctrl_c(None, deps, stop_event, now=11.5)`; assert second call returns `"exiting"` and `stop_event.is_set()`. A call at `now=13.0` (>2s gap) returns `"default"`.
