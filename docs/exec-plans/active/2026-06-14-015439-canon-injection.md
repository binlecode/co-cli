# Canon Relevance Injection — measure first whether canon injection is viable and helps, then build (system-side, no LLM surface)

Task type: measurement gate (lexical-viability probe + grounding ablation) → conditional core-loop feature (per-turn provider + lexical canon retrieval), eval-gated

## Context

Canon scenes (`souls/{role}/canon/*.md`) are the source-material grounding for the active
character. They are **indexed** at bootstrap into the shared `IndexStore` under `source='canon'`,
`kind='canon'`, one whole chunk per file (`co_cli/bootstrap/core.py:_sync_canon_store` →
`_sync_canon_dir`, ~183-247). But **nothing reads them**: no code queries `source='canon'`, and
`IndexStore.get_chunk_content` (`co_cli/index/store.py:497`) has zero callers. The
retrieval/injection half of the design is absent — recorded in `docs/specs/personality.md` §2.5.

The spec documents the **intended** design (settled doctrine): canon "is discrete by nature: a scene
either matches the moment or doesn't … not statically injected … surfaces only through the personality
system. The model never queries canon" (`personality.md` §1, lines 55-59; §2.5; `memory.md:34`). Agent
recall already excludes canon (`MemoryStore` searches `sources=[MEMORY_SOURCE]`, `store.py:186,199`) and
the memory write tool rejects `kind='canon'` (`tools/memory/manage.py:149`).

**Two hard truths surfaced in review reshape this plan into measure-first:**
1. **The matcher is structurally suspect (PO-M-1).** Canon scenes are *narrative prose* about the
   character's situations (e.g. "Cooper leaned in and started whispering…"). A real user turn ("help me
   debug this stuck worker") shares almost no surface tokens — the relevant match is *situational/
   semantic*, which BM25 lexical retrieval cannot see, and the embedding path that could is excluded for
   per-turn latency (`_retrieval.py` is hybrid: FTS5 + sqlite-vec + TEI rerank). So lexical-on-narrative
   may mostly **no-fire** (canon stays inert) or **mis-fire** on an incidental word (distraction).
2. **Value is unproven (PO-M-2).** seed + mindsets + critique already distill the character into
   always-on priming (per `personality.md` §2.5, mindsets/seed are themselves "distillation of canon").
   Whether injecting raw scene text *additionally* changes behavior is unmeasured. The sibling plan
   `personality-static-prefix-refactor` is rigorously measure-first; this plan must match that discipline
   rather than build-then-measure.

Therefore the **first and decisive deliverable is measurement**, not the mechanism: a cheap gate that
answers (a) can any cheap matcher surface a sensible scene, and (b) does injected canon actually improve
grounding — *before* any production retrieval/provider is built. A "no" is a valuable, shippable finding.

**Sibling-plan dependency (PO-m-2).** If `personality-static-prefix-refactor` TASK-3 concludes the
distilled layer already carries character grounding well (its FLAT_OK branch), this plan's value premise
weakens further — read that verdict before committing build effort here.

## Problem & Outcome

**Problem.** The canon injection the spec mandates does not exist, AND it is unproven that the obvious
cheap implementation (per-turn lexical match of the user turn) can either surface the right scene or
improve behavior over the already-distilled character layer.

**Failure cost:** two ways to lose. (1) Do nothing → canon is permanent dead weight: disk + FTS index
churned every bootstrap for zero effect, and future canon authoring has no path to behavior. (2) Build
blindly → sink effort into a per-turn mechanism that lexically mis-/no-fires on narrative prose and is
then correctly rejected by the eval — the build-then-discard cost PO-M-2 warns of.

**Outcome (measure-first).**
1. **A measured go/no-go (the decisive deliverable).** A cheap gate answers: can a cheap matcher surface
   a sensible canon scene for representative turns, and does hand-injecting that scene improve character
   grounding without distraction? This ships value even when the answer is "no, don't build it."
2. **Only if the gate passes — a lexical-only canon retrieval primitive** (FTS/BM25, no embedding/rerank).
3. **Only if the gate passes — a per-turn `canon_injection_prompt` provider**, gated to the active role,
   no LLM surface.
4. **A final behavioral eval** confirming the *built* system reproduces the gate's measured lift.

Build-then-discard of the mechanism is an explicitly accepted outcome: the committed deliverable this
cycle is the **measurement (TASK-1)**, not the feature.

## Scope

### In scope
- **TASK-1 viability gate (always, cheap, eval-layer only — no production code):** a lexical-match probe
  and a hand-injected grounding ablation against real `tars` canon. Decides whether — and with what
  matcher — anything downstream is built.
- **Conditional on the gate:** a public **lexical-only** canon retrieval entry on `IndexStore`/retrieval
  (FTS/BM25, never `EmbeddingService`/rerank), a personality-owned `search_canon` helper
  (`co_cli/personality/canon.py`), and the per-turn `canon_injection_prompt` provider added to
  `ORCHESTRATOR_SPEC.per_turn_instructions`.
- **A final behavioral eval** (with/without the built provider).

### Out of scope
- **Any model-callable canon tool / agent read path** — injection is system-side only; agent recall and
  the write tool already exclude canon.
- **Re-indexing / changing `_sync_canon_dir`** — it already emits the rows this plan reads.
- **Embedding/vector/rerank for canon** — excluded for per-turn cost.
- **Mindset selection / router / rules thinning / static-prefix changes** — owned by
  `personality-static-prefix-refactor`.

## Behavioral Constraints
1. **Measure before build.** No production retrieval/provider code is written until TASK-1's gate passes
   on real canon. A failing gate stops the plan (pivot matcher or surface to TL) — it does not proceed.
2. **Per-turn, never static.** The eventual provider is a `per_turn_instructions` entry (reflects the
   moment; never invalidates the cached static prefix). Never added to `static_instruction_builders`.
3. **Lexical-only retrieval.** Canon search uses FTS/BM25 only — no query embedding, no TEI rerank — and
   must work with no embedding backend configured (`embedding_provider="none"`). Per-turn cost = one
   local SQLite FTS query.
4. **No LLM surface.** No new tool; canon stays absent from agent recall and unwritable. The model never
   queries canon — it only receives injected text.
5. **Gated and graceful.** The provider emits `""` (cleanly elided per `agent/__init__.py:1241`) when:
   personality disabled, no `index_store`, no text user turn, or no hit clears the relevance threshold.
6. **Discrete, threshold-gated.** Prefer injecting nothing over a weak match; a `min_score` threshold —
   calibrated against real canon in TASK-1, not hardcoded — governs inclusion. Lean K=1.
7. **Neutral framing (PO-m-3).** The injected block must not announce itself as "canon/precedent" (risks
   the model talking *about* having a precedent rather than absorbing it). Use behaviorally-neutral
   framing; finalize before TASK-3.
8. **System-owned, read-only.** Retrieval logic lives in a personality-owned helper; canon never written.
   Core personality doctrine — core-level review.
9. **Eval-real** (`feedback_eval_real_world_data`): real `make_eval_deps`, real indexed canon, config
   `llm.host`, real model; `ensure_ollama_warm` outside any `asyncio.timeout`; centralized eval settings.

## Failure Modes

AI behavioral feature; annotated from code + design (warm-35B runs are in TASK-1/4, not at plan time):
- **Current state (the gap):** canon contributes **zero** to output — no consumer. Character expression
  rides only on the distilled layer.
- **Lexical-on-narrative mismatch (the top design risk, PO-M-1):** BM25 over a 7-doc narrative corpus
  with raw user-turn queries likely no-fires (inert) or mis-fires on incidental words (distraction). This
  is *why* TASK-1's lexical probe gates the build and tests an alternative query basis (task-type/mindset
  rather than raw user tokens).
- **Value may not exist (PO-M-2):** the distilled layer may already saturate character grounding; injected
  scenes may not move behavior. TASK-1's hand-injected ablation measures this before any build.
- **Per-turn latency:** a hybrid search per turn would tax every turn with embedding + rerank. Mitigated
  by Constraint #3 (lexical-only).
- **Query degeneracy:** tool-continuation turns carry no `UserPromptPart`; an empty/non-text turn yields a
  meaningless query. Mitigated by the extraction skipping non-user turns (CD-m-3).
- **Small-corpus threshold noise (CD-m-5):** 7 short docs make BM25 IDF noisy and `normalize_bm25` ([0,1))
  hard to threshold; calibrated in TASK-1, not guessed.

## High-Level Design

**Gate first (TASK-1, throwaway/eval-layer).** Index the real `tars` canon into a temp FTS5 `IndexStore`
(reuse the `_FTS5_CONFIG`/`_sync_canon_store` pattern from `test_flow_bootstrap_canon.py`). (a) Run the
character-revealing eval prompts through a private FTS-only canon query under two query bases — raw user
turn, and a task-type/mindset-derived phrasing — and record which scene (if any) each surfaces and at what
normalized score; judge whether matching vs non-matching queries separate at a defensible threshold.
(b) For each prompt, hand-inject the best-guess matching scene and pairwise-judge responses with vs without
it (both orders, per the mindset-ablation metric). **Gate:** proceed to build only if (a) a cheap matcher
reliably surfaces sensible scenes AND (b) injection reliably improves grounding without distraction. Else
pivot the matcher or surface "no viable cheap matcher / no measured lift — canon stays static-distilled".

**Retrieval (TASK-2, conditional).** Add a public lexical-only entry — e.g.
`IndexStore.search_lexical(query, *, sources, kinds, limit)` delegating to a new `RetrievalService`
FTS-only method reusing `_run_chunks_fts` + `normalize_bm25`, never constructing/calling `EmbeddingService`
or `_rerank`. The entry must **hard-route FTS-only independent of the configured `search_backend`** — at
runtime the default is hybrid; TASK-1's probe is FTS-only only because the test fixture forces `fts5`, so
bypassing the hybrid default at runtime is the real work of this task (CD-m-2).
`co_cli/personality/canon.py:search_canon(index_store, query, *, limit, min_score)` wraps it
and reads bodies via `get_chunk_content('canon', path, 0)`.

**Provider (TASK-3, conditional).** `canon_injection_prompt(ctx: RunContext[CoDeps]) -> str` in
`agent/_instructions.py`: gate on personality + `index_store`; extract the latest **`UserPromptPart`** text
by walking `ctx.messages` backward (skip tool-result/non-user turns; guard non-`str` multimodal content —
idiom at `context/summarization.py:292`); `search_canon`; format the top scene with neutral framing; `""`
otherwise. Append to `ORCHESTRATOR_SPEC.per_turn_instructions`.

**Post-delivery doc sync (not a task file — `docs/specs/` excluded from task `files:`):** update
`personality.md` §2.5 "Implementation status" via `/sync-doc` to the shipped mechanism (or to "evaluated,
not built — no viable matcher/lift" if the gate stops the plan).

## Tasks

**TASK-1 — Viability gate: can a cheap matcher surface the right scene, and does injection help? (always)**
- files: `evals/eval_canon_injection.py` (new; gate harness — throwaway-grade, eval-layer only),
  `evals/_judge.py` (reuse `judge_pairwise`)
- done_when: `uv run python evals/eval_canon_injection.py` indexes the real `tars` canon into a temp FTS5
  `IndexStore` and, for the character-revealing prompts, records a JSONL run with: (a) per prompt, the
  scene surfaced (if any) and its normalized score under both query bases (raw user turn vs
  task-type/mindset-derived), plus matching-vs-non-matching separation; (b) the pairwise with/without
  verdict (both orders) — the injected scene is preferentially the one (a) actually surfaced, with a
  manual best-guess used only to establish the grounding *ceiling*, so the measured lift stays
  attributable to a matcher-reachable scene (PO-m-4). The record states the **gate decision**: BUILD
  (which query basis + provisional `min_score`) or STOP (no viable matcher / no measured lift).
- success_signal: a measured, reproducible go/no-go that either green-lights the build with a calibrated
  matcher+threshold, or kills it with evidence — no production code risked either way.
- prerequisites: none

**TASK-2 — Lexical-only canon retrieval primitive (conditional: TASK-1 = BUILD)**
- files: `co_cli/index/store.py`, `co_cli/index/_retrieval.py`, `co_cli/personality/canon.py` (new),
  `tests/test_canon_retrieval.py` (new)
- done_when: a test indexes ≥2 real `tars` canon scenes into a temp `IndexStore` configured with
  `embedding_provider="none"` and no reranker, calls `search_canon(store, query)` with the
  TASK-1-validated query/threshold, and asserts (a) the matching scene's body returns top-ranked, (b) a
  non-matching query returns nothing above `min_score`, (c) the call succeeds with no embedding backend
  (proves lexical-only — no `EmbeddingService` constructed/called).
- success_signal: canon retrieval returns the right scene from a plain SQLite FTS query, no embedding/rerank.
- prerequisites: TASK-1 (BUILD verdict)

**TASK-3 — `canon_injection_prompt` per-turn provider (conditional: TASK-1 = BUILD)**
- files: `co_cli/agent/_instructions.py`, `co_cli/agent/orchestrator.py`, `co_cli/personality/canon.py`,
  `tests/test_canon_injection_provider.py` (new)
- done_when: a test builds a real `RunContext[CoDeps]` (deps with a temp FTS5 `IndexStore` holding indexed
  `tars` canon) and asserts, through the real per-turn assembly path: (a) a user turn matching a scene →
  the assembled per-turn instructions contain that scene body (neutral framing, no "canon" heading); (b) a
  below-threshold turn and a tool-result-only turn (no `UserPromptPart`) → no block; (c) `personality=None`
  or `index_store=None` → `""`; (d) the provider is registered in `ORCHESTRATOR_SPEC.per_turn_instructions`;
  (e) the static prefix (`build_base_instructions`) is byte-unchanged (canon is per-turn, not static).
- success_signal: in a live session, a turn matching a canon scene carries it in the prompt; an unrelated
  or tool-continuation turn carries none.
- prerequisites: TASK-1 (BUILD), TASK-2

**TASK-4 — Behavioral confirmation: built system reproduces the measured lift (conditional)**
- files: `evals/eval_canon_injection.py` (extend the TASK-1 harness with a real-provider arm)
- done_when: `uv run python evals/eval_canon_injection.py` adds an arm running the real `tars` agent with vs
  without the registered `canon_injection_prompt` on the character-revealing prompts (pairwise, both orders,
  real warm 35B), and the run record shows the built mechanism reproduces TASK-1(b)'s grounding lift without
  distraction. Ship only if confirmed; else surface to TL.
- success_signal: the shipped provider measurably improves character grounding in a live agent run, matching
  the hand-injected ceiling from TASK-1.
- prerequisites: TASK-1 (BUILD), TASK-2, TASK-3
