# Plan: Remove `co logs`

## Context
`co logs` is currently a thin wrapper around Datasette. It shells out to the `datasette`
CLI against `~/.co-cli/co-cli-logs.db`, optionally loads `co_cli/datasette_metadata.json`,
and opens `http://127.0.0.1:8001` in a browser. The command does not participate in trace
capture, storage, or formatting; it is only a read-side viewer.

The broader observability surface already has two first-party readers over the same SQLite
store:

- `co tail` for live terminal inspection of recent/new spans
- `co traces` for static nested HTML inspection of trace trees

Removing `co logs` is therefore a product-surface reduction and dependency-reduction change,
not an instrumentation redesign. The main engineering value is dropping the Datasette runtime
dependency pair from install/lock state while keeping the trace DB, JSONL logs, `co tail`,
and `co traces` intact.

## Current-State Validation (2026-04-17)
- `co_cli/main.py:399-429` defines the `logs()` command, opens a browser, and runs
  `subprocess.run(["datasette", str(LOGS_DB), "--port", "8001", ...])`.
- `co_cli/datasette_metadata.json` exists solely to configure the Datasette UI for `co logs`.
- `pyproject.toml:12` and `pyproject.toml:22` declare runtime dependencies
  `datasette>=0.65.2` and `datasette-pretty-json>=0.3`.
- `README.md:128-132` documents `co logs` as a Datasette dashboard.
- `docs/specs/observability.md` still describes a three-viewer model centered on
  `co logs`, `co traces`, and `co tail`; this must be updated via `/sync-doc` after delivery,
  not treated as a plan task.
- `docs/reference/RESEARCH-voice.md:36` also mentions `co logs` in the current capability
  surface and will become stale after removal.
- `uv run co --help` currently lists `logs`, `traces`, and `tail`.

## Problem & Outcome
**Problem:** The product carries a dedicated runtime dependency pair and a third observability
entry point for a niche SQL/raw-db workflow that overlaps with the existing first-party trace
viewers for common usage.

**Failure cost:** Leaving `co logs` in place preserves extra package weight, extra user-facing
surface area, and stale “three viewer modes” language across docs/specs. Removing it carelessly
risks leaving broken docs, stale references, or orphaned packaging metadata.

**Outcome:** `co logs` and all Datasette-specific repo surface are removed. `co tail` and
`co traces` remain the supported observability readers. Datasette is no longer a runtime
dependency.

## Scope
- Remove the `co logs` command implementation from the CLI.
- Remove Datasette-specific repo artifacts and runtime dependencies.
- Update non-spec user-facing docs and reference docs that mention `co logs`.
- Verify the command surface, dependency surface, and trace viewers remain coherent.

## Non-Goals
- No redesign of `co tail` or `co traces`.
- No change to OTel instrumentation, SQLite schema, JSONL export, or WAL behavior.
- No replacement SQL UI shipped in the same delivery.
- No manual spec editing inside this plan; spec sync is a delivery output handled afterward.

## Behavioral Constraints
- `co tail` and `co traces` behavior must remain unchanged.
- `~/.co-cli/co-cli-logs.db` remains the canonical span store.
- Removing Datasette must not leave broken imports or dead code in `co_cli/main.py`.
- The final repo state must have zero live runtime references to `datasette` outside exec-plan
  history or intentionally historical docs.

## High-Level Design
This is a straightforward surface-prune:

1. Delete the `logs()` Typer command from `co_cli/main.py`.
2. Delete `co_cli/datasette_metadata.json`.
3. Remove `datasette` and `datasette-pretty-json` from runtime dependencies and regenerate the
   lockfile.
4. Update README and reference docs so the supported observability surface is “`co tail` +
   `co traces`”.
5. Run `/sync-doc` to rewrite specs that still describe the old three-viewer model.

No replacement code path is needed because both supported viewers already read directly from
the SQLite trace DB without Datasette.
