## 2. fork-claude-code

Facts from `query.ts`, `services/compact/*`:

- `query.ts` applies pre-compaction stages in this order: tool-result budget trimming, optional snip compaction, micro-compact, optional context-collapse, then auto-compact.
- `autoCompact.ts` defines the effective compaction window as model context minus reserved output tokens, caps reserved output at `20_000`, and fires auto-compaction when estimated prompt usage reaches `effectiveContextWindow - 13_000`.
- `autoCompact.ts` skips auto-compaction for some forked agent contexts and stops attempting it after 3 consecutive failures.
- `microCompact.ts` clears older outputs for a fixed compactable tool set and uses the placeholder `[Old tool result content cleared]`.
- `compact.ts` strips user images and documents before compaction and can strip attachments that will be re-announced afterward.
- `compact.ts` retries prompt-too-large failures by trimming oldest API-round groups and can insert a synthetic user marker during retry.
- `prompt.ts` requires a plain-text handoff summary with fixed sections covering request, technical concepts, files/code, errors, progress, and pending work.
- `compact.ts` rebuilds post-compact history instead of leaving a bare summary in place. It reintroduces current attachments and current instruction-like deltas after compaction.

Context-attention behavior:

- Older high-volume tool outputs are cleared before the summarizer sees them.
- The compaction path reconstructs current working context by re-attaching current files, plans, tool deltas, agent listing deltas, and MCP instruction deltas after compaction.
- The final compacted history keeps a compact boundary plus summary and current-state attachments rather than only a summary string.

## 3. hermes-agent

Facts from `agent/context_compressor.py`, `run_agent.py`, `gateway/run.py`:

- `context_compressor.py` is the in-agent compressor. Its default threshold is 50% of model context, with a token-budget protected tail derived from `summary_target_ratio=0.20`.
- The compressor protects the first 3 messages by count, then computes the tail boundary by token budget rather than by a fixed message count. It avoids splitting tool groups and keeps the last user message inside the tail.
- Before summarization, `_prune_old_tool_results()` replaces old tool outputs with one-line summaries, deduplicates identical outputs, and may truncate older tool-call arguments.
- The middle region is summarized by `call_llm(task="compression")`. The compressor retains `_previous_summary` so later compressions can update an existing summary rather than start from scratch.
- The summary is prefixed with `SUMMARY_PREFIX`, which explicitly says the compacted portion is background reference and the active task is in the live conversation.
- On summary failure, the compressor inserts a static fallback marker instead of dropping turns silently.
- `run_agent.py` can attempt preflight compression up to three times before the main loop if estimated tokens are already over threshold.
- `run_agent.py` treats runtime output-cap errors differently from input-too-large errors. It can reduce `max_tokens`, reduce the assumed context length, or compress and retry.
- `gateway/run.py` also has a separate transcript hygiene layer at 85% of context length, with an additional hard message-count cap of 400, before starting a new agent run.

Context-attention behavior:

- The design preserves a protected recent tail by token budget, not just by recent-message count.
- Old tool results are compressed into brief textual summaries before the main middle-summary pass.
- The post-compaction prompt state is anchored by a fixed handoff prefix plus live tail messages and an appended todo snapshot.

## 4. codex

Facts from `codex-rs/core/src/codex/turn.rs`, `compact.rs`, `compact_remote.rs`, `tasks/compact.rs`, and `templates/compact/*`:

- `turn.rs` runs pre-turn compaction before a new sampling pass when total usage reaches the model auto-compact limit.
- `turn.rs` also runs mid-turn compaction when total usage reaches that limit and the last model step still requires follow-up.
- `turn.rs` can compact against the previous model before the next turn when the user switched to a smaller-context model and current usage already exceeds the new model's limit.
- `tasks/compact.rs` routes manual compaction to remote compaction only when the provider reports remote-compaction support. Otherwise it uses the local compaction path.
- `compact.rs` local compaction sends recorded history plus a compact prompt that asks for a concise handoff summary covering progress, key decisions, constraints, remaining work, and critical references.
- Local compaction records the summary from the last assistant message produced during the compact task, prefixes it with `SUMMARY_PREFIX`, and then rebuilds replacement history from selected user messages plus that summary.
- `build_compacted_history()` keeps only user messages within a `20_000` token budget, newest-first, then appends the summary as a user message.
- Local compaction preserves `GhostSnapshot` items after rebuilding history.
- Mid-turn local compaction can insert canonical initial context before the last real user message so the summary stays last in the replacement history.
- Local compaction retries `ContextWindowExceeded` by removing the oldest history item and trying the compact prompt again.
- `compact_remote.rs` trims history to fit remote compaction, calls `compact_conversation_history()`, removes developer messages and non-user-content wrappers from the returned compacted transcript, and can re-insert canonical initial context before the last real user message or summary.

Context-attention behavior:

- Local compaction feeds current history directly to the compaction model and only compresses afterward when rebuilding replacement history.
- The compacted history that remains available to the next turn is deliberately smaller than the original conversation: selected user messages, summary, optional canonical context, and preserved ghost snapshots.
- Remote compaction treats the provider as the compaction engine but still normalizes the returned transcript before storing it.

## 5. opencode

Facts from `packages/opencode/src/session/compaction.ts`, `prompt.ts`, `processor.ts`, `message-v2.ts`, `overflow.ts`, and `config.ts`:

- `overflow.ts` triggers auto compaction when token usage reaches the model's usable input budget after subtracting either configured reserved tokens or `min(20_000, maxOutputTokens)`.
- `config.ts` exposes three compaction knobs: `auto`, `prune`, and `reserved`.
- `prompt.ts` checks overflow after each finished assistant message and inserts a compaction task when needed.
- `prompt.ts` also inserts a compaction task when the main processor returns `"compact"`, including the case where a generation overflowed before finishing.
- `session/compaction.ts` pruning is asynchronous and non-blocking. It protects the newest two user turns, stops at the previous summary assistant message, and only commits pruning if more than `20_000` tokens can be freed beyond a `40_000` protected window.
- Pruning is mark-based, not delete-based. It sets `part.state.time.compacted`, and `message-v2.ts` later renders those results as `[Old tool result content cleared]`.
- The compaction pass itself creates a dedicated assistant message with `mode: "compaction"`, `agent: "compaction"`, and `summary: true`.
- `processor.ts` throws if a summary-mode assistant attempts a tool call, so the compaction agent is text-only.
- `session/compaction.ts` strips media before sending history to the model and appends a compaction prompt after the transformed message history.
- Plugins can inject extra context or replace the default compaction prompt through `experimental.session.compacting`.
- On successful auto compaction, the system can either replay the most recent non-compaction user message or inject a synthetic continuation prompt.

Context-attention behavior:

- Old tool outputs can be hidden from normal model context while still remaining in session storage.
- The compaction agent sees the chosen transcript with media removed, plus any plugin-supplied context, and no tool execution path.
- Auto-compaction can preserve forward progress by replaying or synthesizing the next user-facing continuation message.

## 6. co-cli

Facts from `co_cli/agent/_core.py`, `co_cli/context/_history.py`, `co_cli/context/summarization.py`, `co_cli/context/tool_categories.py`, and `co_cli/context/orchestrate.py`:

- `agent/_core.py` registers this fixed history-processor chain: `truncate_tool_results`, `compact_assistant_responses`, `detect_safety_issues`, `inject_opening_context`, `summarize_history_window`.
- `tool_categories.py` defines compactable tools as `read_file`, `run_shell_command`, `grep`, `glob`, `web_search`, `web_fetch`, `read_article`, and `read_note`.
- `_history.py` clears compactable tool results older than the 5 most recent per tool type outside the last turn and uses the placeholder `[tool result cleared — older than 5 most recent calls]`.
- `_history.py` separately caps older assistant `TextPart` and `ThinkingPart` content at 2.5K chars while leaving `ToolCallPart` arguments untouched.
- `_history.py` computes the sliding compaction trigger from the resolved budget and fires when estimated tokens exceed 85% of that budget.
- The dropped middle region is summarized by `summarize_messages()` with a structured prompt from `summarization.py`.
- `_gather_compaction_context()` supplies three context sources to the summarizer: file paths extracted from file-tool call args, pending session todos, and prior summary text found in dropped messages.
- `summarization.py` appends an `## Additional Context` block when compaction context exists and appends a personality-preservation addendum when personality is active.
- `_summarize_dropped_messages()` falls back to a static marker when no model is available or when the compaction failure counter has reached 3.
- `recover_overflow_history()` keeps the first turn group and last turn group, tries to summarize the middle, preserves `search_tools` breadcrumbs from the dropped region, and otherwise inserts a static marker.
- `orchestrate.py` handles provider context overflow by running `recover_overflow_history()`, replacing turn history, and retrying once.

Context-attention behavior:

- High-volume tool outputs and older assistant prose are reduced before the summarizer runs.
- File-working-set context is reconstructed from preserved tool-call args rather than from original tool outputs.
- Prior compaction summaries and pending todos are explicitly carried into later summary passes.

## 7. Common Facts Across the Four Peers

- All four peers separate compaction from the ordinary main-turn prompt path.
- All four peers preserve recent live context more aggressively than older context.
- `fork-claude-code`, `hermes-agent`, and `co-cli` all perform an explicit pre-pass on older tool-heavy history before or during compaction.
- `codex` local compaction is the only scanned implementation here that rebuilds post-compaction history from user-message retention plus a prefixed summary after running a dedicated compact turn.
- `opencode` is the only scanned implementation here where tool-result pruning is mark-based in stored history rather than destructive replacement in the stored transcript.
- `hermes-agent` is the only scanned implementation here with two separate compression layers: gateway transcript hygiene and in-agent compression.

## 8. Converged Practices in the Scanned Systems

This section narrows to repeated patterns across the scanned implementations. A pattern is included here when it appears in multiple systems' live code paths.

### 8.1 Summarize Windowing

- Token-budget triggers are the common gate. `fork-claude-code`, `hermes-agent`, `codex`, `opencode`, and `co-cli` all decide compaction from token or context-window budget rather than from raw transcript length alone.
- Output headroom is reserved before compaction triggers. `fork-claude-code` subtracts reserved output tokens before computing the auto-compact threshold. `opencode` subtracts reserved tokens or provider max output from usable input. `codex` compacts against the model auto-compact token limit. `co-cli` resolves a compaction budget and fires at 85% of that budget. `hermes-agent` derives thresholds from the model context length and summary target ratio.
- The newest live context is protected more strongly than older context. `hermes-agent` preserves a protected tail by token budget. `co-cli` keeps a head segment and a snapped tail segment and drops the middle. `opencode` protects the newest two user turns during prune. `codex` preserves the live continuation path by mid-turn reinjection before the last real user message. `fork-claude-code` runs pre-passes over older history before building post-compact messages.
- Boundary alignment to conversational or tool-group structure recurs. `hermes-agent` avoids splitting tool groups in its tail selection. `co-cli` snaps the tail boundary to turn groups. `fork-claude-code` trims oldest API-round groups on compact retry. `codex` trims oldest history items on local compact retry and then rebuilds replacement history from retained user messages.

### 8.2 Summarizing Logic

- Structured handoff prompts are universal in the scanned set. `fork-claude-code`, `hermes-agent`, `codex`, `opencode`, and `co-cli` all use compaction prompts or templates that ask for task state, decisions, remaining work, or relevant files rather than free-form narration.
- The summarizer path is tool-free in the local compaction implementations. `fork-claude-code` says not to use tools in the compact prompt. `opencode` enforces this in `processor.ts` by throwing on tool calls while `summary: true`. `co-cli` uses a dedicated summarizer agent with no tools. `codex` local compact sends a prompt without tool execution. `hermes-agent` uses `call_llm(task="compression")` for summary generation.
- Older high-volume tool context is usually reduced before or during summarization. `fork-claude-code` uses micro-compact. `hermes-agent` prunes and deduplicates old tool results first. `opencode` mark-prunes old tool outputs before later compaction. `co-cli` clears older compactable tool results before `summarize_history_window()`.
- Summary generation is usually paired with an explicit failure path. `fork-claude-code` has a circuit breaker and prompt-too-large retry logic. `hermes-agent` falls back to a static marker. `opencode` records a context-overflow error and stops if the compaction agent itself overflows. `codex` local compact retries by trimming oldest items. `co-cli` falls back to a static marker after repeated failures or missing model access.
- Incremental summary carry-forward appears in more than one system. `hermes-agent` keeps `_previous_summary` for iterative updates. `co-cli` extracts prior summary text from dropped messages and feeds it back into later summary prompts.

### 8.3 Summary Injection Back Into Context

- The summary is usually inserted as a typed or prefixed context artifact, not as an unmarked free-text blob. `fork-claude-code` stores a compact boundary and summary message. `hermes-agent` uses a fixed summary prefix. `codex` prefixes its compact summary and stores a compaction item. `opencode` uses an assistant message marked `summary: true` plus a compaction task part. `co-cli` inserts a summary marker message into the preserved history.
- Systems usually keep more than the summary itself after compaction. `fork-claude-code` re-attaches current files, plan state, invoked-skill state, deferred tool deltas, agent listing deltas, and MCP instruction deltas. `hermes-agent` keeps head and protected tail messages. `codex` keeps selected user messages, optional canonical initial context, and ghost snapshots. `opencode` keeps pruned history in storage and can replay the triggering user message. `co-cli` keeps preserved head and tail groups and search-tool breadcrumbs across overflow recovery.
- Several systems explicitly re-introduce non-summary state that the model still needs. `fork-claude-code` rebuilds current attachments after compaction. `codex` can re-inject canonical initial context before the last real user message. `hermes-agent` appends a todo snapshot and rebuilds system-prompt state after compression. `opencode` can replay a prior user message or insert a synthetic continuation. `co-cli` injects file working-set paths, pending todos, and prior summary text into the summarizer prompt and preserves `search_tools` breadcrumbs in overflow recovery.
- Placement of the injected summary is used to separate background from live task state. `hermes-agent` explicitly marks the summary as background reference. `codex` inserts canonical context before the last real user message so the summary remains last in mid-turn compacted history. `opencode` follows compaction with replay or synthetic continuation when auto-compacting. `co-cli` keeps the summary between preserved older and newer regions instead of replacing the full prompt with summary-only context.
