# Agentic Context Management — Peer Comparison & Gap Analysis

How peer agents keep the model's **working context** bounded — tool-result lifecycle, history-window selection, compaction/summarization, and long-loop context — and where co's design lags, leads, or has open questions.

**Out of scope:** long-term declarative memory, personality, approvals/sandboxing, model routing, skill lifecycle.

## Core mechanisms

### co — layered budget stack (`co_cli/context/`, spec `docs/specs/compaction.md`)
Runs per LLM-request, not per turn (pressure rises mid-turn from a single tool batch). Five layers, cheap → expensive:
- **L0 admission** — `MAX_TOOL_CALLS_PER_MODEL_REQUEST=3`; rejects excess tool calls per response (bounds fan-out for small-model coherence).
- **L1 spill** — `spill_if_oversized` at emit time: tool output >`SPILL_THRESHOLD_CHARS=4000` is persisted to `tool-results/<sha16>.txt` behind a placeholder + preview. **Recoverable** — content survives off-context, not discarded.
- **L2 force-spill** — `spill_largest_tool_results` processor: when request tokens > `spill_ratio×budget` (0.50), spills largest unspilled tool returns largest-first until under threshold.
- **L3 LLM compaction** — `proactive_window_processor` at `compaction_ratio×budget` (0.50): summarizes the middle, assembles `head | marker | [todo_snapshot] | tail`. Tail is **floor-aware** (v0.8.327): `tail_budget = tail_fraction/compaction_ratio × usable_trigger`, `usable_trigger = max(0, compaction_ratio×budget − static_floor_tokens)` — shrinks the tail when the instruction+schema floor is large so the compacted result lands below the next trigger (`tail_fraction=0.10`). Deferred-tool reveals are **not** carried in the assembly: a loaded tool's name lives in `runtime.revealed_tools` (set by `tool_view`), which compaction never touches, so reveals survive for free with no preservation step. Anti-thrash skips after 2 consecutive passes that each yielded <10% savings (routing into the static-marker path); a **no-progress convergence guard** (v0.8.327) escalates once to overflow recovery when an applied pass fails to shrink, instead of re-firing; circuit breaker after 3 summarizer failures, then periodic probes; static-marker fallback. ~13-section iterative summary template (Active Task copied verbatim; prior summary carried forward in a dedicated `PRIOR SUMMARY` slot — v0.8.326).
- **Recovery + Manual** — on HTTP 400/413, `strip_all_tool_returns` then retry; if still over, plan boundaries + `compact_messages`. `/compact [focus]` runs the same primitive over full history. **Pre-flight fit guard** (v0.8.410): a `SummarizerInputTooLargeError` / `INPUT_TOO_LARGE` fallback reason fires when estimated summarizer input exceeds `window − output_budget − safety_margin`, degrading to the static marker **without eroding the circuit breaker** (it is a sizing miss, not a summarizer failure).
- **Standing processors** — `dedup_tool_results` (collapse identical returns to a back-reference) + `evict_old_tool_results` (keep 5 most recent per tool) run ahead of L2/L3.

### hermes-agent — single-shot 50% compaction (`agent/context_compressor.py`)
- No live tool-output cap; full outputs preserved until compaction fires. (A recent char-based output-cap commit added *defensive* parsing of provider `max_tokens`-overflow errors in `agent/model_metadata.py` — it shrinks the requested output budget on retry; it is not a tool-result truncation tier.)
- Fires post-response at 50% of window; **anti-thrash** skip if last 2 passes saved <10%.
- Window: protected head (first 3), tail = 20% of threshold, last-user-message safeguard, forward/backward tool-pair boundary alignment, orphan-repair stub for summarized tool_calls.
- Three-pass prepass before summarizer: dedup (md5 of content), content-stub (>200 chars → 1-line), arg-truncation (>500 chars).
- Dedicated aux summary model (fallback to main on error); 13-section template, secrets redacted via preamble; 600s cooldown on failure.
- **Long-loop:** `trajectory_compressor.py` distills finished-session JSONL; cron jobs run as isolated agentic loops with their own JSONL.
- **New since last survey:** compression now forks a session — `get_compression_tip()` follows the compression-continuation chain so resume routes to the correct child after an auto-compression split; a `session:compress` event callback lets external systems (memory sync) react to compaction; summaries carry a `_compressed_summary` metadata flag so frontends distinguish them without a content-prefix heuristic; and `_derive_auto_focus_topic()` infers the compaction focus from recent turns instead of requiring a manual `/compress <topic>`.

### openclaw — adaptive multi-phase compaction (`src/agents/`, loop forked in-tree)
- Live tool-output cap `TOOL_RESULT_MAX_CHARS=8000` head-only (`embedded-agent-subscribe.tools.ts:30`), UTF-16-safe truncation; a single message that exceeds **50% of the context window** is flagged oversized at planning time (`isOversizedForSummary`, `compaction-planning.ts:273`). *(The earlier "≤30% of window" per-result ceiling is gone — the over-size gate is now the 50%-of-window planning check.)*
- No internal threshold — invoked by the in-tree runner on context-overflow detection (`isLikelyContextOverflowError`); optional mid-turn precheck (lane-queued via `embedded-agent-runner/compact.queued.ts` + `lanes.ts`).
- Window: token-share chunking (adaptive ratio 0.4→0.15 via `computeAdaptiveChunkRatio`, safety margin 1.2; constants at `compaction-planning.ts:13`), pending-tool-call set prevents mid-pair boundaries, oldest chunks dropped (`pruneHistoryForContextShare:342`) with tool-pair repair (`repairToolUseResultPairing:382`).
- Staged chunk→summarize→merge (`summarizeInStages`, `compaction.ts:318`); `toolResult.details` stripped pre-summary (`stripToolResultDetails`, comment "must never enter LLM-facing compaction"); explicit identifier-preservation policy (strict/off/custom); main session model; 3-retry backoff (attempts 3, 500–5000ms) + size-fallback chain.
- **Long-loop:** post-compaction transcript rotation keeps active file bounded (`compaction-successor-transcript.ts`); lane-queued compaction; **handoff mode runs on a tighter 20%-of-window budget** (`defaultShare = isHandoff ? 0.2 : 0.5`, `compaction-planning.ts:359`) with leader/subordinate framing. *(No explicit grace-period deadline-extension found at HEAD — only a `compaction-safety-timeout.ts` guard; the prior "grace-period extension" line was not re-confirmed.)*
- **New since last survey:** a context-engine projection layer (`CodexProjectedContextRange`, `fitCodexProjectedContextForTurnStart`) reserves next-step context across thread rotation (cap `CODEX_TURN_START_TEXT_INPUT_MAX_CHARS = 1MB`); compaction model calls now emit diagnostic spans; active-compaction retries are preserved across attempt handling; a `session-registry-maintenance.ts` module does background session cleanup.

### opencode — truncate-in-place (`packages/core/src/session/`)
- Tool output truncated in place only: `TOOL_OUTPUT_MAX_CHARS=2000` on the core compaction path (`packages/core/src/session/compaction.ts:14`); the UI-layer `truncateToolOutput` is a separate configurable cap (`packages/opencode/src/session/message-v2.ts:51`). No spill, no content-addressed recovery; oversized output dropped inline (media → `FilePart` attachments).
- Compaction emits `Compaction.Started`/`Ended` (`reason: auto|manual`), runs sequentially in the prompt loop, not concurrent with streaming.

### codex — schema-trim + checkpoint + provenance (`codex-rs/{tools,core,rollout-trace}/`)
- Dual-layer: tool-schema trimming to ≤4KB (`MAX_COMPACT_TOOL_SCHEMA_BYTES=4_000`, `tools/src/json_schema.rs:222`, applied at tool-registration time) + history-replacement checkpoint (`CompactionInstalled` event, `rollout-trace/src/raw_event.rs:203`). The two layers are decoupled — schema trim at registration, history replacement at session compaction.
- **Multi-agent provenance:** `AgentThread.origin = AgentOrigin::Spawned` lineage (`rollout-trace/src/model/session.rs:55`) + `RolloutTrace.interaction_edges` directed graph (`mod.rs:87`; kinds SpawnAgent / AssignAgentTask / SendMessage / AgentResult / CloseAgent at `runtime.rs:319`) — keeps auditable context lineage across spawned agents.
- **New since last survey (multi-agent context pooling):** a rollout-level token budget is now **shared across all agent threads** in a rollout (configurable reminder intervals + token weights for sampling vs. prefill); `CompactedItem` carries a stable `window_id` (`protocol/src/protocol.rs:2980`) tracking context-window boundaries across compactions; and a `TurnContextContribution` extension trait injects turn-local, `model_context_window`-aware context fragments. Codex is deepening *cross-agent* context budgeting, not just within-thread compaction.

## Marker construction & no-LLM-drop logic

How each agent builds the single replacement message that stands in for a dropped/summarized region, and — the axis that actually differs — what that message preserves when **no LLM summary** is produced (skip, failure, or cooldown). Verified at peer HEADs 2026-06-18.

- **co** — one dispatcher `build_compaction_marker(dropped_count, summary_text, *, has_tail)` (`co_cli/context/_compaction_markers.py:127`) branches on `summary_text is None`: `summary_marker` wraps the LLM recap, `static_marker` (`:46`) is a **content-free** fixed placeholder (`STATIC_MARKER_PREFIX` + "{n} earlier messages were removed — treat as background, do not redo/re-answer" + `has_tail` trailer). Both prefixes are recognised by `is_compaction_marker` (`:136`) so a later pass treats a marker as a boundary, not fresh content. `compact_messages` assembles `head | marker | [todo_snapshot] | tail` identically for both branches — only marker *content* differs. The static (no-LLM) path is reached today by the **circuit breaker** (3 summarizer failures), **model-absent**, the **anti-thrash gate** (2 consecutive <10%-yield passes set `summarize=False` into this same path — the `antithrash-static-marker-fallback` work is shipped), and (v0.8.410) the **`INPUT_TOO_LARGE` pre-flight fit guard** (estimated summarizer input won't fit `window − output_budget − safety_margin`; degrades here without eroding the breaker); it preserves nothing from the dropped middle beyond the verbatim tail + side-channel `todo_snapshot` (deferred-tool reveals survive independently in `runtime.revealed_tools`, untouched by compaction).
- **hermes-agent** — one dispatcher `_with_summary_prefix(summary)` (`agent/context_compressor.py:1659`); both the LLM path and the no-LLM fallback flow through it and get the same `SUMMARY_PREFIX` sentinel (constant at `:43`; literal prefix match excludes an existing marker from the next summarization window). The no-LLM fallback is **not** content-free: `_build_static_fallback_summary` (`:1089`) deterministically extracts recent user asks / tool outcomes / errors / touched files from the dropped region into the 13-section template (capped 8K chars via `_FALLBACK_SUMMARY_MAX_CHARS=8_000`, `:169`). Summarizer failure / 600s cooldown → drop region + insert this deterministic recap (not left intact, not empty). Orphan tool-call repair stub: `"[Result from earlier conversation — see context summary above]"` (`:1793`).
- **openclaw** — one `compactionSummary` session message (`src/agents/cli-runner/session-history.ts:413`); no distinct static path. Retry-exhaustion fallback is a terse string `"Context contained {n} messages ({m} oversized). Summary unavailable due to size limits."` (`src/agents/compaction.ts:312`) — still the same message type, no recap of content. Recognition is **structural** (message `role`), not a text prefix; prior summaries re-fed for re-distillation are wrapped in `<previous-compaction-summary>` XML (`compaction-safeguard.ts:74`). Split tool pairs: orphaned result dropped, synthetic error `toolResult` `"[openclaw] missing tool result …"` inserted (`session-transcript-repair.ts:150`). A recent commit (`98d5c46530`, "keep compaction notices additive") makes the post-compaction notice path additive — plugin hook messages and the built-in user notice now both fire instead of one suppressing the other.
- **opencode** — single-path, **always summarizes** (no static placeholder). Compaction boundary is tracked by the retained `recent` + `summary` fields on the v2 Compaction message, not a text marker. (`tail_start_id` originated on the v1 `CompactionPart` schema but is **still actively used as a message-id boundary in the v2 layer** for tail filtering — `message-v2.ts:545`, not purely legacy.) Summary stored as an assistant message via a structured template ("keep every section even when empty; do not mention that context was compacted", `packages/core/src/session/compaction.ts:48`). In-place tool-output truncation marker: `"[Tool output truncated for compaction: omitted {n} chars]"` (`packages/opencode/src/session/message-v2.ts:54`).
- **codex** — two-phase: a `CompactionMarker` conversation item with an **intentionally empty body** as a pure structural boundary (`rollout-trace/src/reducer/conversation.rs:317` — comment: keep empty so renderers can't mistake the boundary for prompt content), plus the LLM summary as a separate `Summary` part. `SUMMARY_PREFIX` text sentinel via `is_summary_message → starts_with` (`core/src/compact.rs:451`). On a no-/empty-summary drop it still **retains recent user messages verbatim** (up to `COMPACT_USER_MESSAGE_MAX_TOKENS=20_000`, const at `compact.rs:52`); summary text falls back to `"(no summary available)"` (`compact.rs:596`). `is_summary_message` text-sentinel check at `compact.rs:480`.

**Best-practice synthesis.** (1) **Single dispatch-on-summary builder is universal** — every peer funnels both the LLM and no-LLM cases through one builder (co/hermes/openclaw a single message builder; codex one marker item + a separate summary part). None maintains two parallel marker-construction paths; co's `summarize=False`-into-the-same-builder approach is the surveyed-standard shape. (2) **Sentinel to prevent re-summarization** — co/hermes/codex use a text prefix; openclaw uses the structural role; opencode uses a boundary id. All have *some* idempotency marker. (3) **The real divergence is what a no-LLM drop preserves.** hermes (deterministic breadcrumb extraction) and codex (recent-user-message retention) preserve continuity from the dropped region without an LLM; co's `static_marker` and openclaw's fallback string are **content-free** (co keeps only tail + todos; deferred-tool reveals survive in `runtime.revealed_tools`, not the marker). This is the single axis where co sits below the richest peers on the no-summary path — a candidate enrichment (deterministic breadcrumb in `static_marker` for *all* no-LLM paths: breaker, model-absent, anti-thrash), not a correctness gap.

## Gap analysis

**Where co leads.** Co is the only surveyed agent with a **recoverable spill** tier — oversized tool output is content-addressed to disk behind a placeholder, not truncated away (opencode 2K and openclaw 8K both lose the tail). Combined with L0 admission + L2 force-spill, co caps per-request growth at multiple cheap tiers *before* paying for LLM summarization — a finer-grained ladder than any peer's single compaction shot. This is the deliberate small-local-model trade: keep each request small and coherent rather than relying on one big summarize.

**No active gaps.** On every surveyed *within-session* dimension co is at or beyond peer parity. The peer features co lacks are not deficiencies — each is either rejected by design or conditional on a feature co has not built.

**Deliberate divergences (peer has it; co rejects the premise):**

- **Per-session distillation.** Hermes' `trajectory_compressor.py` distills a finished session's JSONL into a per-session trajectory artifact. Co treats the session boundary as *arbitrary* (Ctrl+C, lunch, task-switch, context limit), not a semantic unit, so precomputing a recap keyed to that boundary produces an artifact whose value is bounded by the arbitrariness of its scope (canon: a `session_summary` daemon task was cut at Gate 1 on exactly this principle). Co retains cross-session knowledge **content-driven, not boundary-driven**: the dream reviewer fires `memory_review` (generalization → durable facts) and `skill_review` (procedural extraction) over recent transcript *content* on a counter cadence + session-end always-fire, and finished transcripts stay BM25-searchable on demand (`session_search`/`session_view`). Co answers over content on whatever range the user cares about; hermes precomputes an answer keyed to an arbitrary cut.

- **Background-loop context isolation.** Hermes runs cron jobs as independent agents, each with its own isolated JSONL; openclaw hands a bounded snapshot between runs. Co's dream daemon is **not** an external background flow — it is a *split-brain of the main agent*: one agent's curation function extracted into a separate process for non-blocking concurrency, sharing all long-term state (memory, skills, sessions, index). Its reviewers are the same agent (`build_task_agent`), run stateless single passes over REPL-produced transcripts, and write back into the shared stores. Isolating its context would *defeat the purpose* — it shares stores precisely so its generalizations feed the same memory the REPL reads. Hermes' isolated-per-job loop is a different architecture (an independent agent on its own task), not a capability co is missing. (Bounding is a non-issue too: within-session compaction L0–L3 applies to any agent loop, daemon included.)

**Conditional (only bites if co builds the triggering feature):**

- **Multi-agent provenance + cross-agent context budgeting.** Codex tracks `AgentThread`/`AgentOrigin` lineage + an interaction-edge graph, and at this HEAD has gone further: a rollout-level token budget **shared across all agent threads**, stable `window_id`s on compacted items, and a turn-scoped context-contribution trait — i.e. compaction is becoming a *multi-agent* budgeting problem, not a per-thread one. Openclaw uses leader/subordinate handoff framing + a context-engine projection that reserves next-step context across thread rotation. Co spawns no subagents, so there is no lineage to track and no cross-agent budget to pool — relevant prior art only *if* co ever adds subagent spawning and needs to reconstruct/share context across agents.

**Likely non-gaps (in-place rewrite is sufficient):**

- **Transcript rotation.** Openclaw rotates the active transcript file after compaction; co rewrites `sessions.md` in place. For interactive sessions this is fine — only worth revisiting if a single background run produces an unbounded JSONL.

- **Mid-turn precheck.** Openclaw can compact before a turn completes. Co's L0 (cap 3 tool calls/response) + L2 (force-spill) already bound per-turn growth tightly, so a precheck likely buys little for the small-model target — but unconfirmed.
