# PROPOSAL: co-agent-context — Engineering Specification
_Date: 2026-03-16_

## Purpose

This document is the engineering specification for `co`'s context architecture. The primary objective is to establish a robust system founded entirely on **Pydantic-AI's idiomatic state management patterns**—utilizing strongly-typed dependency injection, pure functional tools, and explicit state transitions.

The core design rules are strict:
- **Explicit Dependency Injection:** Context is assembled at runtime using `@agent.system_prompt` pulling exclusively from a strongly-typed `RunContext[CoDeps]`.
- **Functional Tool Purity:** Tools must not have hidden side effects (e.g., no write-on-read memory gravity). Durable mutations must be explicit LLM decisions verified by Pydantic schemas.
- **Typed Boundaries:** Subagents do not inherit ambient state; they receive strictly typed Pydantic payloads (e.g., `SpecialistDeps`) defining their exact context boundary.
- **Explicit Workflows:** Transcript trimming, doom-loop guards, and compaction must use native Pydantic-AI features (`UsageLimits`, `ModelRetry`) or explicit agent tasks (e.g., a `CompactionAgent`).

---

## 1. Context Model

The system uses four explicit context layers, mapped natively to Pydantic-AI constructs.

### 1.1 Memory (Learned Agent State)
- **Role:** Preferences, corrections, decisions, and workflows.
- **Implementation:** Stored durably. Injected dynamically via `@agent.system_prompt(ctx: RunContext[CoDeps])` fetching from `ctx.deps.memory_svc`.
- **Always-on Subset:** Certain records marked `always_on=true` are injected by default; others must be retrieved via explicit pure-function search tools.

### 1.2 Knowledge (Reference Material)
- **Role:** Library items, workspace docs, fetched articles.
- **Implementation:** Kept out of the system prompt. Retrieved exclusively on-demand via pure `search_knowledge` tools.

### 1.3 Conversation History (Raw Transcript)
- **Role:** Append-only thread of messages and tool calls.
- **Implementation:** Managed natively by Pydantic-AI's `message_history`.

### 1.4 Session Summaries (Derived Artifacts)
- **Role:** Compaction and handoff artifacts.
- **Implementation:** Generated by explicitly invoking a `CompactionAgent` when token thresholds are reached. Yields a strict `SessionSummary` Pydantic model.

---

## 2. Engineering Principles

1. **Context Loading is Type-Safe:** All contextual data must flow through the `CoDeps` generic parameter of the Agent.
2. **Retrieval is Side-Effect Free:** Reads do not mutate memory, knowledge, history, or summaries.
3. **Recovery is a Shared Native Pattern:** Doom loops and poor tool results trigger native `ModelRetry("reasoning")` exceptions, hooking directly into Pydantic-AI's built-in reflection cycle.
4. **Bounded Reflection for Mutation:** Memory updates require an explicit tool invocation with a schema that enforces a `reasoning` field to justify the mutation.

---

## 3. Event-Driven Context Management

### 3.1 Session Start & Bootstrap
- **Action:** Initialize `CoDeps` (Services, Config, Runtime).
- **Execution:** When `agent.run(user_input, deps=CoDeps(...))` is called, Pydantic-AI evaluates all `@agent.system_prompt` functions. These functions strictly pull `always_on` memory, project rules, and active skill instructions from `ctx.deps`.

### 3.2 Tool Selection & Execution
- **Action:** Model selects tools based on the active `@agent.system_prompt`.
- **Execution:** All tools receive `ctx: RunContext[CoDeps]`. Tool execution is side-effect free. If a tool fails or detects a doom-loop, the tool function raises a `ModelRetry` exception to force the model to reformulate its approach.

### 3.3 Context-Expanding Evidence (Subagents)
- **Action:** A task requires delegation to a specialist.
- **Execution:** The orchestration layer constructs a typed boundary model:
  ```python
  class SpecialistDeps(BaseModel):
      task_objective: str
      standing_instructions: str
      relevant_memories: list[str]
  ```
  The parent explicitly invokes the subagent: `specialist.run(task, deps=SpecialistDeps(...))`.

### 3.4 History Compaction (Token Pressure)
- **Action:** The `UsageLimits` threshold approaches.
- **Execution:** The runtime pauses the main run and invokes an explicit `CompactionAgent(deps=RawHistory)` that outputs a `SessionSummary` model. The main run is then restarted or continued with the new summary replacing the raw history.

---

## 4. Tool & Delegation Contracts

### 4.1 Retrieval Tools (Pure Functions)
- `search_memories(ctx, query)`: Returns memories without altering ranking or timestamps.
- `search_knowledge(ctx, query)`: Returns external facts based strictly on the query.

### 4.2 Mutation Tools (Explicit Reflection)
- `update_memory_priority(ctx, memory_id, reasoning, new_priority)`: To ensure quality control, the LLM must explicitly call this tool, providing `reasoning` for *why* a memory should be promoted to `always_on` or altered.

### 4.3 Recovery & Guard Tools
- Use Pydantic-AI's `UsageLimits(request_limit=X)`. If the request limit is breached, catch `UsageLimitExceeded`, formulate a recovery prompt, or yield to the user.

---

## 5. Curation Rules

### 5.1 Memory Curation
- **No Write-on-Read:** Recency gravity is never updated automatically when a memory is fetched.
- **Explicit Promotion:** Promoting a memory to `always_on` requires explicit LLM reasoning (via the mutation tools) or explicit user directive.

### 5.2 History & Session Curation
- History is kept raw. When Pydantic-AI's message history grows too large, a structured `CompactionAgent` summarizes it into a `SessionSummary` artifact. Summaries are passed explicitly during session resumption.

---

## 6. Implementation Phases

This specification should be implemented in strict phases to ensure stability and alignment with the Pydantic-AI framework.

### Phase 0: Pure Functional Tooling
- Ensure all retrieval paths (`search`, `recall`) are strictly side-effect free.
- Guarantee that tools do not modify timestamps, access counts, or ranking states during read operations, enabling deterministic Pydantic-AI retries and Logfire tracing.

### Phase 1: Explicit Dependency Injection & Recovery
- Establish the central `CoDeps` dataclass (combining config, services, and session state).
- Implement `@agent.system_prompt` decorators that dynamically pull context directly from `RunContext[CoDeps]`.
- Implement native shared recovery by using `ModelRetry("reasoning")` inside tool execution flows to handle loops or invalid outputs.

### Phase 2: Type-Safe Subagents & Reflection
- Define strict Pydantic models (e.g., `SpecialistDeps`) for all subagent handoffs.
- Update delegation tools to instantiate and pass these explicit boundary models.
- Introduce the `UpdateMemoryPriority` tool with a required `reasoning` schema to enforce reflection before durable memory writes.

### Phase 3: Explicit Compaction Tasks
- Implement `UsageLimits` tracking on main agent runs.
- Build the explicit `CompactionAgent` to handle token threshold breaches and output strongly-typed `SessionSummary` artifacts.
